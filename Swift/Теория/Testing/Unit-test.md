### Что такое unit-тесты (в 2026 году)

**Unit-тест** — это **автоматизированный тест**, который проверяет **одну маленькую часть кода** (функцию, метод, класс, ViewModel, UseCase, валидатор и т.д.) в **изоляции** от остальной системы.

Ключевые характеристики unit-теста в 2026 году:

- **Изолированность** — не ходит в сеть, не пишет в БД, не рисует UI  
- **Быстрый** — 0.01–2 секунды на тест  
- **Детерминированный** — всегда даёт один и тот же результат при тех же входных данных  
- **Повторяемый** — можно запускать тысячи раз без изменения состояния  
- **Покрывает поведение** — проверяет «что должно произойти», а не «как именно это реализовано»

### Основные преимущества unit-тестов (реальные плюсы 2026)

1. **Раннее обнаружение багов**  
   → Баг ловится на этапе написания, а не на проде или в QA

2. **Безопасный рефакторинг**  
   → Можно переписывать ViewModel, менять алгоритмы, переходить на [[async]]/[[await]] — тесты дают уверенность

3. **Живая документация**  
   → Тесты показывают **как именно** должен работать код (лучше, чем комментарии)

4. **Защита от регрессий**  
   → Новый код не ломает старый функционал

5. **Лучшая архитектура**  
   → Чтобы код было легко тестировать, приходится писать его **чище** (маленькие функции, протоколы, DI)

6. **Быстрый CI/CD**  
   → Тесты за 5–30 секунд → мгновенная обратная связь в PR

### Основные недостатки unit-тестов (честные минусы 2026)

1. **Дополнительное время на написание**  
   → 1 час кода → 0.5–1.5 часа тестов (в среднем)

2. **Тесты тоже нужно поддерживать**  
   → Рефакторинг → рефакторинг тестов  
   → Переход на новый [[API]] → массовое обновление тестов

3. **Хрупкие тесты** (если пишут «по реализации»)  
   → Тест падает при любом рефакторинге, даже если поведение не изменилось

4. **Не ловят интеграционные и UI-баги**  
   → 100% покрытие unit-тестами ≠ работающее приложение

5. **Перетестирование** (over-testing)  
   → 90% времени на поддержку тестов, которые проверяют очевидное

### Структура unit-теста в 2026 году (AAA + современные фичи)

```swift
import XCTest
@testable import MyApp

final class CalculatorTests: XCTestCase {
    
    func testAdditionOfTwoPositiveNumbers() {
        // Arrange (Подготовка)
        let calculator = Calculator()
        let a = 5
        let b = 7
        
        // Act (Действие) — только один вызов!
        let result = calculator.add(a, b)
        
        // Assert (Проверка)
        XCTAssertEqual(result, 12, "5 + 7 должно быть 12")
    }
    
    @MainActor
    func testAsyncDivisionByZeroThrowsError() async throws {
        // Arrange
        let calculator = Calculator()
        
        // Act + Assert вместе (современный стиль)
        await XCTAssertThrowsError(try await calculator.divide(10, by: 0)) { error in
            XCTAssertEqual(error as? CalculatorError, .divisionByZero)
        }
    }
}
```

**Современные улучшения** ([[Swift]] 6+ и [[Xcode]] 16+):

- `@MainActor` — обязательно для ViewModel / UI-тестов  
- `async throws` + `await XCTAssertThrowsError` — для асинхронных тестов  
- `Swift Testing` (новый фреймворк) — `#expect`, `#require`, `@Test`, `@Suite`

### Лучшие практики unit-тестирования в Swift 2026

1. **Тестируй поведение, а не реализацию**  
   → Плохо: `XCTAssertEqual(vm.privateVar, 42)`  
   → Хорошо: `XCTAssertEqual(vm.displayedText, "42")`

2. **Один тест — одна идея**  
   → Не больше 1–3 Assert на тест  
   → Если Act больше одного — разбей тест

3. **Моки / стабы / фейки через протоколы**  
   → `any Protocol` + конкретные реализации для тестов  
   → Избегай библиотек типа Cuckoo / Mockingbird в простых случаях (ручной мок читаемее)

4. **Async/await тесты** — всегда `async throws` + `await`  
   → `await XCTAssertEqual(await vm.users.count, 5)`

5. **Coverage 70–85%** — золотая середина  
   → 100% обычно означает перетестирование

6. **Не тестируй очевидное** (YAGNI в тестах)  
   → Не пиши тест на `user.name = "Alice"` → `user.name == "Alice"`

7. **Swift Testing** (Xcode 16+) — переходи на него  
   → `@Test`, `#expect`, `#require` — проще и выразительнее

### Когда unit-тесты действительно окупаются

| Тип кода / фичи                              | Стоит ли писать unit-тесты? | Рекомендуемое покрытие |
| -------------------------------------------- | --------------------------- | ---------------------- |
| Бизнес-логика, UseCase, ViewModel            | Да, обязательно             | 80–95%                 |
| Валидаторы, форматтеры, мапперы              | Да                          | 90–100%                |
| Сетевые сервисы, репозитории                 | Да                          | 70–90%                 |
| UI-компоненты ([[UIView]], [[SwiftUI]] View) | Частично (snapshot + UI)    | 20–50%                 |
| Простые модели без логики                    | Нет / минимально            | 0–30%                  |
| Временные прототипы / эксперименты           | Нет                         | 0%                     |

**Короткий вердикт 2026**:
> «Unit-тесты — это страховка от регрессий и гарантия, что завтрашний рефакторинг не сломает сегодняшний функционал.  
> Но если тесты занимают больше времени, чем сам код — ты переусердствовал.»
