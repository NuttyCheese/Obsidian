### 2. Визуальная схема: merge vs rebase

```mermaid
gitGraph
   commit id: "C1 (main)"
   commit id: "C2 (main)"
   branch feature
   checkout feature
   commit id: "F1"
   commit id: "F2"
   checkout main
   commit id: "C3 (main)"
   
   %% merge
   merge feature id: "Merge commit M" type: REVERSE
   commit id: "После merge" tag: "merge результат"
   
   %% rebase (альтернативная история)
   checkout feature
   rebase main id: "Rebased F1' и F2'"
   commit id: "После rebase" tag: "rebase результат"
```

**После merge** история выглядит так (нелинейно):

```
C1 ← C2 ← C3 ← M (merge commit)
       ↖ F1 ← F2
```

**После rebase** история становится линейной:

```
C1 ← C2 ← C3 ← F1' ← F2' 
```

### 3. Пошаговое сравнение на реальном примере

Исходная ситуация:

```
main:    A ← B ← C
feature:      ↖ D ← E
```

#### Вариант 1 — git merge (рекомендуется для общей ветки)

```bash
git checkout main
git merge feature
```

Результат:

```
A ← B ← C ← M (merge commit)
       ↖ D ← E
```

- История сохранена полностью  
- Видно, что feature жила отдельно  
- Безопасно: никто не теряет коммиты  
- Merge commit содержит информацию о слиянии

#### Вариант 2 — git rebase (рекомендуется только для личных веток)

```bash
git checkout feature
git rebase main
```

Результат:

```
A ← B ← C ← D' ← E'
```

- История линейная и чистая  
- Нет merge commit  
- Коммиты D и E получили **новые хэши** (D' и E')  
- Опасно, если кто-то уже pull'нул feature

### 4. Таблица: когда использовать merge, а когда rebase (2026)

| Ситуация                                      | Рекомендация       | Почему                                                                 |
|-----------------------------------------------|--------------------|-------------------------------------------------------------------------|
| Feature-ветка личная, никто её не pull'нул    | **rebase**         | Чистая линейная история, нет лишних merge-коммитов                      |
| Feature-ветка уже в общем доступе (push'нута) | **merge**          | Rebase перепишет историю → проблемы у коллег (diverged branches)       |
| Слияние develop в main (релиз)                | **merge**          | Сохраняется вся история разработки                                      |
| Hotfix в production                           | **merge**          | Безопаснее, история не ломается                                         |
| Поддержание чистой линейной истории в main    | **rebase + squash**| Самый чистый результат, но только если никто не работает с веткой       |
| Работа в команде с частыми релизами           | **merge**          | Безопаснее и понятнее для всех                                          |

### 5. Как правильно использовать rebase (безопасный способ)

Самый безопасный паттерн 2026 года:

```bash
# 1. Обновляем свою ветку
git checkout feature/add-login
git fetch origin
git rebase origin/main

# 2. Если конфликты — разрешаем их
# после разрешения:
git add .
git rebase --continue

# 3. Force push (только если ветка не была в общем доступе!)
git push --force-with-lease  # безопаснее, чем --force
```

**--force-with-lease** — защита от потери чужих коммитов.

### 6. Как разрешать конфликты (общее для merge и rebase)

```bash
# Конфликт возник
git status  # покажет файлы с конфликтами

# Открываем файл → ищем <<<<<<< HEAD ... ======= ... >>>>>>> branch
# Выбираем нужные изменения → удаляем маркеры

git add conflicted-file.swift
git rebase --continue   # или git merge --continue
```

### 7. Лучшие практики 2026 года ([[iOS]]/[[Swift]] команды)

- **Для личных feature-веток** — rebase на main перед PR  
- **Для merge в main/develop** — всегда **merge --no-ff** (сохраняет историю ветки)  
- **В команде > 5 человек** — избегайте rebase на общих ветках (main, develop)  
- Используйте **Squash & Merge** в GitHub/GitLab — чище история  
- В PR описывайте:  
  - Что сделано  
  - Как тестировали  
  - Скриншоты/UI-тесты  
- Внедряйте **Conventional Commits** + **semantic-release** → автоматические версии и changelog  
- Защищайте main/develop:  
  - Require PR  
  - Require 1–2 approval  
  - Require passing [[CI]] ([[Xcode]] build + tests)

**Короткий девиз 2026**:
> «Merge — для сохранения правды истории и безопасности.  
> Rebase — для красоты линейной истории, но только на личных ветках.  
> В команде > 3 человек — merge почти всегда побеждает.»
