**Flux** — это классическая архитектурная концепция однонаправленного потока данных, предложенная Facebook в 2014 году для React-приложений.  
Она никогда не была полноценной библиотекой, а скорее **паттерном проектирования**, который решал проблему хаотичного управления состоянием в больших SPA.

В 2026 году Flux как самостоятельная реализация почти не используется, но его идеи лежат в основе практически всех современных state management решений.

### Краткая история и актуальность на 2026 год

Год создания: 2014  
Оригинальный доклад: Jordan Walke и команда React на F8 2014  
Официальная реализация Facebook: Flux Utils (очень лёгкая, почти не развивается с 2016–2017)  
Судьба:  
- 2015–2018 — пик популярности Flux  
- 2016 — появляется **Redux** → моментально вытесняет оригинальный Flux  
- 2019–2021 — Zustand, Jotai, Recoil, MobX, Valtio и т.д.  
- 2022–2026 — React Server Components + React Query / TanStack Query + Zustand / Jotai → Flux-идеи остались, но оригинальный Flux практически мёртв  

Сегодня Flux изучают в основном в двух контекстах:
- чтобы понять, **почему появился Redux** и как он решает проблемы Flux  
- чтобы понять принципы **однонаправленного потока данных**, которые до сих пор лежат в основе большинства современных решений

### Сравнение оригинального Flux и его прямых наследников (2026)

| Характеристика                  | Оригинальный Flux (Facebook 2014–2016) | Redux (2015–наши дни) | Zustand / Jotai (2022–2026) | React Query + Zustand (современный стандарт) |
|---------------------------------|----------------------------------------|-----------------------|-----------------------------|-----------------------------------------------|
| Однонаправленный поток          | Да                                     | Да                    | Да                          | Да                                            |
| Единый Dispatcher               | Да (обязательный)                      | Нет (combineReducers) | Нет (каждый store независим) | Нет                                           |
| Один большой Store              | Нет — много Stores                     | Да — один глобальный  | Нет — много маленьких       | Нет — много маленьких + кэш запросов          |
| Boilerplate                     | Средний                                | Высокий (особенно с TypeScript) | Очень низкий                | Низкий                                        |
| Middleware / side-effects       | Нет встроенных                         | Redux Thunk / Saga    | Встроенные actions / middleware | React Query handles side-effects              |
| DevTools                        | Нет официальных                        | Отличные (Redux DevTools) | Хорошие (Zustand DevTools)  | Отличные (React Query DevTools)               |
| Актуальность в 2026             | Почти мёртв (только для изучения)      | Умеренная (legacy-проекты) | Высокая                     | Очень высокая                                 |

### Классическая структура оригинального Flux (как её показывали в 2014–2016)

```
User interaction / Server response
          ↓
       Action Creator → Action (plain object { type, payload })
          ↓
       Dispatcher (singleton) — единственный, кто рассылает Actions
          ↓
   Stores (много штук) — каждый подписан на Dispatcher
          ↓
Stores emit change events
          ↓
Views (React-компоненты) — подписаны на change events Stores
          ↓
Views перерисовываются → пользователь видит изменения
          ↓
Новый interaction → новый Action → цикл повторяется
```

### Почему Flux быстро уступил место Redux и дальше

1. **Один Dispatcher + много Stores** → очень сложно отлаживать последовательность обновлений  
2. **Нет единой точки правды** (состояние размазано по многим Stores)  
3. **Нет встроенных инструментов для асинхронных действий** ([[API]]-запросы, таймеры)  
4. **Нет официальных DevTools**  
5. **Сложно типизировать** (особенно в TypeScript)

Redux решил большинство этих проблем одним большим Store + middleware + отличными DevTools.

А современные Zustand / Jotai / Jotai + React Query решили проблемы Redux:  
- меньше boilerplate  
- проще типизация  
- лучше производительность  
- лучше интеграция с React Server Components

### Когда в 2026 году всё ещё говорят «Flux»?

- В старых курсах и книгах по React  
- Когда объясняют, почему Redux / Zustand / Recoil устроены именно так  
- В разговорах о «однонаправленном потоке данных»  
- В legacy-проектах Facebook (очень редко)

### Короткий итог 2026

> Flux — это **паттерн однонаправленного потока данных** 2014 года,  
> который показал, как **управлять состоянием** в больших React-приложениях.

Он практически полностью вытеснен более удобными и мощными решениями:

- 2015–2020 → Redux  
- 2020–2023 → Redux Toolkit + RTK Query  
- 2023–2026 → Zustand / Jotai / Valtio + TanStack Query / React Query

Но понимание Flux всё ещё очень полезно, потому что именно он заложил фундамент современной философии state management:

**однонаправленный поток + действия → состояние → UI → новые действия**

Если вы встречаете Flux в 2026 году — это либо учебный материал, либо очень старый проект.

Удачи с современным и приятным управлением состоянием в твоих приложениях! 🚀
