## Что такое Debug Navigator

**Debug Navigator** — это панель в Xcode, которая показывает состояние запущенного приложения в реальном времени:

- активные потоки (threads)
    
- стек вызовов (call stack)
    
- загрузку CPU
    
- использование памяти
    
- дисковую активность
    
- сетевую активность
    

Проще говоря:

> Это центр управления отладкой, где видно, что происходит внутри процесса.

---

# 📍 Где находится

Левая панель [[Xcode]] → иконка 🐞 (Debug Navigator)  
Или `⌘ + 7`

Работает только во время запущенного приложения.

---

# 🧱 Основные разделы Debug Navigator

---

## 1️⃣ Threads & Call Stack (Потоки и стек вызовов)

Самый используемый раздел.

Показывает:

- список всех потоков
    
- текущий активный поток
    
- стек вызовов выбранного потока
    

### Что можно делать:

- переключаться между потоками
    
- переключаться между stack frames
    
- анализировать, где возник crash
    

---

### 🧠 Практика

Если приложение зависло:

1. Открой Debug Navigator
    
2. Выбери main thread
    
3. Посмотри, где он остановлен
    

Если main thread заблокирован — UI будет "заморожен".

---

## 2️⃣ CPU Usage

Показывает:

- текущую загрузку процессора
    
- график активности
    

Используется для:

- поиска бесконечных циклов
    
- диагностики heavy вычислений
    
- проверки фоновых задач
    

Если CPU стабильно 100% — где-то ошибка.

---

## 3️⃣ Memory Usage

Показывает:

- текущее потребление памяти
    
- рост памяти со временем
    

Используется для:

- поиска утечек
    
- диагностики [[retain cycle]]
    
- проверки больших моделей
    

Если память растёт и не освобождается — подозрение на утечку.

---

## 4️⃣ Disk Activity

Показывает:

- операции чтения/записи
    

Полезно при:

- работе с кэшем
    
- [[Core Data]]
    
- загрузке файлов
    

---

## 5️⃣ Network Activity

Отображает:

- сетевые операции
    
- объём трафика
    

Помогает понять:

- идут ли запросы
    
- не завис ли networking
    

---

# 🔬 Работа со стеком вызовов (самое важное)

Когда выполнение остановлено:

В Debug Navigator видно:

```
Thread 1
 0  ViewController.viewDidLoad
 1  UINavigationController.pushViewController
 2  ...
```

### Как анализировать:

- Сверху — текущая точка остановки
    
- Ниже — кто вызвал этот метод
    
- Идём вниз, пока не найдём свой код
    

---

# 💥 Анализ краша через Debug Navigator

1. Приложение упало
    
2. Debug Navigator автоматически откроет поток с crash
    
3. Ищи первый фрейм из своего проекта
    
4. Кликни на него
    
5. Проверь Variables View
    

Это быстрее, чем читать лог вручную.

---

# 🧵 Работа с потоками

Можно увидеть:

- [[Main Thread]]
    
- Background Queue
    
- [[URLSession]] threads
    
- Dispatch queues
    

Если видишь:

```
UI API called on background thread
```

→ Переключись на поток и посмотри стек.

---

# 🚀 Продвинутые сценарии

---

## 🔥 1. [[Deadlock]]

Если приложение зависло:

- Main thread стоит
    
- Background thread ждёт main
    
- Main ждёт background
    

В Debug Navigator видно оба стека.

---

## 🔥 2. Поиск [[race condition]]

Можно увидеть:

- параллельное выполнение
    
- одновременный доступ к данным
    

Особенно полезно вместе с Thread Sanitizer.

---

## 🔥 3. Анализ [[async]]/[[await]]

При async:

- появятся дополнительные стеки
    
- можно переключаться между continuation
    

Важно смотреть не только main thread.

---

# 📊 Когда использовать Debug Navigator

|Сценарий|Использовать|
|---|---|
|Приложение зависло|Threads|
|Высокая загрузка CPU|CPU gauge|
|Память растёт|Memory|
|Не работает UI|Main thread|
|Не приходит ответ|Network activity|

---

# ⚠️ Частые ошибки

### ❌ Смотреть только верхний фрейм

Иногда причина ниже в стеке.

---

### ❌ Игнорировать фоновые потоки

Многие баги происходят вне main thread.

---

### ❌ Путать системные фреймы с собственным кодом

Ищи первый фрейм своего модуля.

---

# 📈 Правильный алгоритм работы

1. Запустить приложение
    
2. Воспроизвести баг
    
3. Открыть Debug Navigator
    
4. Найти нужный поток
    
5. Проанализировать стек
    
6. Проверить переменные
    
7. Сделать вывод
    

---

# 🎯 Почему Debug Navigator важен

- Он показывает архитектуру выполнения
    
- Видно, как взаимодействуют потоки
    
- Можно понять реальные причины зависаний
    
- Помогает анализировать runtime поведение
    

Это фундаментальный инструмент для понимания живого приложения.
