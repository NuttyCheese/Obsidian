## Что такое Time Profiler

**Time Profiler** — это инструмент из набора **Instruments**, который позволяет анализировать:

- какие методы занимают больше всего времени CPU
    
- где происходят задержки
    
- какие функции вызываются чаще всего
    
- что тормозит приложение
    

Проще говоря:

> Это инструмент, который показывает, куда уходит процессорное время.

Он не ищет утечки памяти — он ищет узкие места производительности.

---

# 📍 Где находится

1. [[Xcode]]
    
2. Product → Profile (`⌘ + I`)
    
3. Выбрать шаблон **Time Profiler**
    
4. Запустить профилирование
    

Откроется отдельное приложение — Instruments.

---

# 🧠 Как работает Time Profiler

Он делает **sampling CPU**:

- каждые несколько миллисекунд фиксирует, какой код выполняется
    
- строит статистику
    
- показывает, где приложение проводит больше всего времени
    

Это не замедляет приложение критично и даёт реальную картину.

---

# 🧱 Основные области интерфейса

---

## 1️⃣ Timeline (верхняя часть)

Показывает:

- график CPU
    
- пики нагрузки
    
- время выполнения
    

Если видишь резкий пик — значит в этот момент выполнялась тяжёлая операция.

---

## 2️⃣ Call Tree (дерево вызовов)

Самая важная часть.

Показывает:

- методы
    
- процент времени
    
- self time (время в самом методе)
    
- total time (с учётом вложенных вызовов)
    

---

## 3️⃣ Heaviest Stack Trace

Показывает цепочку вызовов, которая потребляет больше всего времени.

---

# 📊 Основные показатели

|Показатель|Значение|
|---|---|
|Self Time|Время только внутри метода|
|Total Time|Метод + все вложенные вызовы|
|%|Доля CPU|
|Samples|Количество зафиксированных попаданий|

---

# 🧪 Практические сценарии

---

## 📱 1. Приложение лагает при открытии экрана

Алгоритм:

1. Запустить Time Profiler
    
2. Открыть проблемный экран
    
3. Остановить запись
    
4. В Call Tree отсортировать по Self Time
    

Если видишь:

```
JSONDecoder.decode → 35%
```

Значит декодирование тяжёлое.

---

## 🧵 2. UI фризится

Проверяем:

- [[main|Main Thread]]
    
- Heavy synchronous операции
    

Если на main:

```
Data(contentsOf:)
image resizing
large loop
```

→ Это причина лагов.

---

## 📦 3. Медленная загрузка списка

Может оказаться:

- сложные вычисления в cellForRowAt
    
- heavy AutoLayout
    
- expensive image processing
    

---

## 🔄 4. Частые вызовы метода

Если метод вызывается слишком часто:

```
layoutSubviews
updateUI
didSet
```

Это видно по количеству samples.

---

# ⚙️ Важные настройки Call Tree

Включить:

- ✅ Separate by Thread
    
- ✅ Hide System Libraries
    
- ✅ Invert Call Tree
    

### Invert Call Tree

Позволяет видеть, какие методы занимают больше всего времени сверху.

Очень полезная настройка.

---

# 🚀 Продвинутый анализ

---

## 🔥 1. Анализ main thread

Если main thread загружен более 16ms на кадр → будет просадка FPS.

60 FPS = 16.6 ms на кадр.

---

## 🔥 2. Поиск неоптимального алгоритма

Если видишь:

```
Array.filter → 20%
Array.map → 25%
```

Возможно, лишние проходы по массиву.

---

## 🔥 3. Проверка [[lazy]] / computed property

Иногда computed property вызывается слишком часто.

Time Profiler покажет это сразу.

---

## 🔥 4. Сетевые проблемы

Если network parsing занимает много CPU — это будет видно.

---

# 📈 Реальный пример

Допустим:

Экран тормозит при скролле.

Time Profiler показывает:

```
UIImageJPEGRepresentation → 40%
```

Значит ты:

- пересжимаешь изображение в cell
    
- делаешь это на main thread
    

Решение:

- вынести в background queue
    
- кэшировать результат
    

---

# 🧠 Правильный алгоритм работы

1. Чётко определить проблему (лаг, задержка)
    
2. Запустить Time Profiler
    
3. Воспроизвести сценарий
    
4. Остановить запись
    
5. Отсортировать по Self Time
    
6. Найти самый тяжёлый метод
    
7. Понять, почему он тяжёлый
    
8. Оптимизировать
    
9. Повторно проверить
    

---

# ⚠️ Частые ошибки

### ❌ Смотреть только Total Time

Иногда метод сам быстрый, но вызывает тяжёлые вложенные операции.

---

### ❌ Не скрывать системные библиотеки

Иначе сложно найти свой код.

---

### ❌ Оптимизировать без измерений

Сначала профилирование — потом изменения.

---

# 🧬 Time Profiler vs другие инструменты

|Инструмент|Для чего|
|---|---|
|Time Profiler|CPU / производительность|
|Allocations|выделение памяти|
|Leaks|утечки|
|Memory Graph|retain cycle|
|Thread Sanitizer|гонки потоков|

---

# 🎯 Когда использовать Time Profiler

- лаги при скролле
    
- фризы UI
    
- долгий запуск
    
- медленная обработка данных
    
- подозрение на тяжёлый алгоритм
    
- высокая загрузка CPU
    

---

# Почему это важно

Производительность — это не случайность.

Она достигается:

- пониманием архитектуры
    
- измерением
    
- дисциплиной анализа
    

Time Profiler — инструмент для зрелой оптимизации, а не для догадок.
