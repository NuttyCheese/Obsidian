## Что такое LLDB

**LLDB** — это низкоуровневый отладчик, встроенный в [[Xcode]], который позволяет:

- останавливать выполнение программы,
    
- исследовать состояние памяти,
    
- смотреть значения переменных,
    
- выполнять код во время паузы,
    
- изменять поведение выполнения,
    
- диагностировать краши и сложные runtime-ошибки.
    

Он является стандартным отладчиком в Xcode и работает поверх Clang/[[Swift]] компилятора.

Проще говоря:

> LLDB — это инструмент, который позволяет "залезть внутрь" работающего приложения и посмотреть, что реально происходит в памяти.

---

# 🔎 Как LLDB работает в [[iOS]]

Когда приложение запускается в Debug режиме:

1. Xcode подключает LLDB к процессу.
    
2. LLDB получает доступ к:
    
    - стеку вызовов
        
    - регистрам процессора
        
    - памяти
        
    - переменным
        
    - [[Runtime]] Swift/ObjC
        
3. При достижении breakpoint выполнение останавливается.
    
4. Ты начинаешь управлять состоянием программы через консоль.

# 🧱 Основные сущности в LLDB

### 1️⃣ Breakpoint — точка остановки

Останавливает выполнение в конкретной строке.

Можно:

- поставить вручную
    
- создать symbolic breakpoint
    
- добавить условие
    
- выполнить действие (например, лог без остановки)
    

---

### 2️⃣ Stack Frame — текущий фрейм стека

Позволяет:

- видеть цепочку вызовов
    
- переключаться между уровнями
    
- анализировать, кто вызвал текущий метод
    

Команда:

```
bt
```

---

### 3️⃣ Expression Engine

Позволяет выполнять Swift-код прямо во время паузы.

Это мощнейшая часть LLDB.

---

# 🛠 Основные команды LLDB (которые реально используются)

## 📌 1. po (print object)

Выводит описание объекта.

```
po user
po view.frame
po navigationController?.viewControllers
```

Используется 90% времени.

---

## 📌 2. p (print)

Выводит значение без красивого описания.

```
p count
```

---

## 📌 3. v (frame variable)

Показывает переменные текущего фрейма.

```
v
```

---

## 📌 4. bt (backtrace)

Показывает стек вызовов.

```
bt
```

Очень полезно при креше.

---

## 📌 5. expr

Выполняет выражение.

```
expr user.name = "Test"
expr view.backgroundColor = .red
```

Можно изменить состояние приложения прямо во время отладки.

---

## 📌 6. c (continue)

Продолжить выполнение.

```
c
```

---

## 📌 7. n (next)

Шаг вперёд (без захода в функцию).

```
n
```

---

## 📌 8. s (step)

Зайти внутрь функции.

```
s
```

---

## 📌 9. finish

Выйти из текущего метода.

```
finish
```

---

# 💥 Реальные сценарии использования в iOS

---

## 🧠 1. Проверка состояния ViewController

```swift
po self
po view.subviews
po tableView.visibleCells
```

Можно понять:

- добавлена ли view в иерархию
    
- сколько ячеек реально отображается
    
- корректны ли constraint’ы
    

---

## 🧠 2. Проверка AutoLayout

```
po view.constraints
po view._autolayoutTrace()
```

Очень полезно при:

- ambiguous layout
    
- unsatisfiable constraints
    

---

## 🧠 3. Проверка памяти (retain cycle)

```
po self
```

Если объект не деаллоцируется — можно проверить, кто его удерживает через [[Memory Graph Debugger (Отладчик графа памяти)|Debug Memory Graph]].

---

## 🧠 4. Изменение UI на лету

```
expr view.alpha = 0.2
expr button.isHidden = true
```

Позволяет тестировать поведение без перезапуска приложения.

---

## 🧠 5. Работа с Optional

```
po optionalValue
po optionalValue!
```

Если [[nil]] — LLDB покажет.

# 🎯 Advanced возможности

---

## 🔥 Conditional Breakpoints

Можно поставить условие:

```
index == 10
```

Остановка произойдет только при выполнении условия.

---

## 🔥 Breakpoint Action (лог без остановки)

Можно добавить:

```
po model
```

И не останавливать выполнение.

Используется как мощный логгер.

---

## 🔥 Symbolic Breakpoints

Можно поставить breakpoint на:

- `UIViewAlertForUnsatisfiableConstraints`
    
- `objc_exception_throw`
    
- `Swift runtime failure`
    

Это помогает ловить:

- AutoLayout ошибки
    
- runtime exception
    
- force unwrap crash
    

---

# 🧬 LLDB и Swift особенности

Swift сложнее для отладчика из-за:

- оптимизаций
    
- [[value type]]
    
- [[generic]]
    
- [async]/[[await]]
    

Важно:

- использовать Debug сборку
    
- отключать оптимизации при сложной отладке

# ⚠️ Типичные проблемы

### 1. "variable not available"

Происходит из-за оптимизаций компилятора.

### 2. expr не работает с async

Для async-кода:

- ставим breakpoint внутри await
    
- проверяем [[Task]]

# 🧪 LLDB при краше

Когда приложение падает:

1. Смотри строку падения
    
2. Выполни:
    

```
bt
```

3. Найди первый свой фрейм (не системный)
    
4. Проверь переменные:
    

```
po self
po model
```

---

# 💡 Как пользоваться эффективно (практический алгоритм)

1. Ставишь breakpoint в подозрительном месте
    
2. Проверяешь входные параметры (`po`)
    
3. Проверяешь состояние объекта (`po self`)
    
4. Проверяешь стек (`bt`)
    
5. При необходимости меняешь состояние через `expr`
    
6. Продолжаешь выполнение

# 🚀 Почему LLDB важен для iOS разработчика

- Логи показывают прошлое
    
- LLDB показывает настоящее
    

Это единственный способ:

- точно понять причину бага
    
- увидеть реальное состояние [[Runtime]]
    
- диагностировать [[retain cycle]]
    
- разобраться с [[Auto Layout]]
