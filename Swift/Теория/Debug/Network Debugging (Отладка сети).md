## Что такое Network Debugging

**Network Debugging** — это процесс анализа сетевых запросов и ответов приложения:

- какие [[URL]] вызываются
    
- какие заголовки отправляются
    
- какое тело запроса
    
- какой статус-код приходит
    
- как выглядит [[JSON]]-ответ
    
- где происходит ошибка
    

Проще говоря:

> Это контроль всего [[HTTP]]-трафика приложения.

Для iOS-разработчика это критично — большинство багов сегодня связаны с [[API]].

---

# 🧱 Основные инструменты для отладки сети

---

## 1️⃣ Xcode Debug Navigator (Network)

Позволяет видеть:

- активные запросы
    
- объём переданных данных
    
- длительность
    

Но не показывает тело запроса подробно.

Подходит для базовой диагностики.

---

## 2️⃣ Логирование через `URLSession`

Можно логировать вручную:

```swift
print(request)
print(response)
print(data)
```

Или использовать `Logger`:

```swift
logger.debug("Request: \(request)")
```

Минус — не всегда удобно анализировать большие JSON.

---

## 3️⃣ Proxy-инструменты

### 🔹 Charles Proxy

### 🔹 Proxyman

Это самые мощные инструменты.

Позволяют:

- перехватывать [[HTTPS]]
    
- смотреть тело запроса
    
- редактировать ответ
    
- повторять запрос
    
- замедлять сеть
    
- подменять JSON
    

Это уже профессиональный уровень анализа.

---

# 📍 Как работает Proxy

1. Приложение отправляет запрос
    
2. Proxy перехватывает его
    
3. Показывает request/response
    
4. Передаёт на сервер
    

Для HTTPS нужно установить SSL сертификат в систему и доверить его.

---

# 🧪 Практические сценарии

---

## 📦 1. Проверка корректности запроса

Проверяем:

- [[URL]]
    
- метод ([[GET-HTTP|GET]]/[[POST-HTTP|POST]])
    
- заголовки (Authorization)
    
- body
    
- query параметры
    

Очень часто ошибка — в неправильном header или параметре.

---

## 🔑 2. Проблемы с авторизацией

Смотрим:

```
401 Unauthorized
403 Forbidden
```

Проверяем:

- access token
    
- refresh token
    
- expiresAt
    
- корректность Bearer
    

---

## 📄 3. Ошибка декодирования

Если JSON не совпадает с моделью:

- смотрим raw response
    
- сравниваем с [[Codable]] моделью
    
- ищем отсутствующие поля
    

---

## 🐢 4. Медленный ответ сервера

Проверяем:

- время запроса
    
- размер ответа
    
- latency
    

Можно увидеть, что проблема не в клиенте.

---

## 🔄 5. Повторный запрос (Retry)

Через Proxy можно:

- повторить запрос
    
- изменить тело
    
- проверить реакцию сервера
    

Очень полезно при разработке сложного API.

---

# 🚀 Продвинутые возможности

---

## 🔥 1. Mock ответа

Можно:

- подменить JSON
    
- вернуть ошибку
    
- протестировать edge-case
    

Например:

- 500 server error
    
- пустой массив
    
- некорректный формат
    

Без изменения сервера.

---

## 🔥 2. Throttling (замедление сети)

Можно эмулировать:

- 3G
    
- плохой WiFi
    
- высокий ping
    

Это важно для UX.

---

## 🔥 3. Работа с SSL Pinning

Если используется SSL Pinning:

- обычный Proxy не перехватит трафик
    
- нужно отключать pinning в debug
    

---

# 🧠 Правильный подход к Network Debugging

1. Проверить URL
    
2. Проверить метод
    
3. Проверить заголовки
    
4. Проверить body
    
5. Проверить статус-код
    
6. Проверить JSON
    
7. Проверить декодирование
    

Никогда не начинать с "сервер не работает" — сначала проверка запроса.

---

# 📊 Где что использовать

|Инструмент|Когда использовать|
|---|---|
|print / Logger|быстрая локальная проверка|
|Debug Navigator|проверить активность|
|Charles / Proxyman|глубокий анализ|
|Time Profiler|если сеть тормозит UI|

---

# ⚠️ Частые ошибки

### ❌ Логировать токены в production

Это небезопасно.

---

### ❌ Доверять только клиентской модели

Всегда сверять raw JSON.

---

### ❌ Не проверять статус-код

Иногда сервер возвращает 200 с ошибкой в теле.

---

# 🎯 Что должен уметь iOS-разработчик

- читать HTTP-запрос
    
- понимать коды 200 / 201 / 204 / 400 / 401 / 403 / 500
    
- анализировать JSON
    
- находить mismatch с Codable
    
- проверять headers
    

Без этого невозможно писать стабильный networking слой.

---

# 💡 В контексте твоего опыта

Если ты строишь собственный network layer:

- добавь логирование request/response
    
- логируй status-code
    
- логируй время выполнения
    
- добавь возможность включать verbose-режим
    

Это сильно упрощает диагностику.
