**`NSLock`** — это самый простой и классический **рекурсивный мьютекс** из [[Foundation]].

Он обеспечивает **эксклюзивный доступ** к критической секции:

- один поток захватил lock → все остальные ждут  
- освободил → следующий в очереди заходит  

**Ключевые характеристики в 2026 году**:

- **рекурсивный** (можно несколько раз lock в одном потоке без [[deadlock]])  
- **очень медленный** по современным меркам (в 5–15 раз медленнее `os_unfair_lock`)  
- **тяжёлый** (объект [[Objective-C]], [[ARC]], [[KVO]], исключения и т.д.)  
- **всё ещё поддерживается** Apple и используется в огромном количестве legacy-кода  
- **официально не deprecated**, но Apple уже много лет рекомендует [[os_unfair_lock]]

### 2. Основные методы и свойства NSLock

| Метод / Свойство      | Описание                                          | Возвращает          | Частота использования 2026 |
|-----------------------|---------------------------------------------------|----------------------|-----------------------------|
| `lock()`              | Захватывает замок (блокирует, если занят)         | Void                 | ★★★★★                       |
| `unlock()`            | Освобождает замок                                 | Void                 | ★★★★★                       |
| `try()`               | Пытается захватить без блокировки                 | `Bool`               | ★★★☆☆                       |
| `name`                | Имя замка для отладки (Instruments, logs)         | `String?`            | ★★☆☆☆                       |
| `lock(before:)`       | Пытается захватить до дедлайна                    | `Bool`               | ★☆☆☆☆ (редко)               |

### 3. Самые популярные (и часто ошибочные) шаблоны 2026

#### Паттерн 1 — Классический (и до сих пор встречающийся) — защита счётчика

```swift
let lock = NSLock()
var counter = 0

for _ in 0..<10_000 {
    DispatchQueue.global().async {
        lock.lock()
        counter += 1
        lock.unlock()
    }
}

// Итог всегда 10_000 (без lock — почти никогда)
print(counter)
```

#### Паттерн 2 — Самый безопасный и рекомендуемый способ в 2026 — с defer

```swift
let lock = NSLock()
var sharedCache: [String: Data] = [:]

func saveToCache(_ data: Data, key: String) {
    lock.lock()
    defer { lock.unlock() }  // гарантированно освободит даже при throw / return
    
    sharedCache[key] = data
}
```

#### Паттерн 3 — try() — неблокирующий захват (редко, но полезно)

```swift
if lock.try() {
    // критическая секция
    lock.unlock()
} else {
    // кто-то другой уже внутри — делаем что-то альтернативное
    print("Кэш занят, пропускаем")
}
```

### 4. Сравнение производительности и выбора инструмента в 2026

| Инструмент              | Скорость (1 млн захватов/освобождений) | Накладные расходы | Рекурсивный? | Рекомендация 2026 | Когда использовать |
|-------------------------|----------------------------------------|-------------------|--------------|-------------------|---------------------|
| `os_unfair_lock`        | ~15–25 мкс                             | ★★★★★ (минимальные) | Нет          | ★★★★★             | Горячие секции, максимальная производительность |
| `NSLock`                | ~80–150 мкс                            | ★★☆☆☆ (тяжёлый)     | Да           | ★★☆☆☆             | Только legacy-код   |
| `pthread_mutex_t`       | ~40–80 мкс                             | ★★★☆☆               | Да (если настроить) | ★★☆☆☆             | Интеграция с C      |
| `actor`                 | ~100–200 мкс (включая переключение)    | ★★★☆☆               | Да (по сути) | ★★★★★             | Почти всё остальное |
| `Atomic` + CAS          | ~8–15 мкс                              | ★★★★★               | —            | ★★★★★             | Примитивы, счётчики |

**Вывод по скорости** (Apple Silicon M4, 2026):
- `os_unfair_lock` ≈ 5–10× быстрее `NSLock`  
- `Atomic` (relaxed) ≈ 5–15× быстрее `os_unfair_lock` (но только для примитивов)  
- `actor` ≈ в 5–10 раз медленнее `os_unfair_lock`, но **гораздо безопаснее** и **читабельнее**

### 5. Когда NSLock всё ещё оправдан в 2026 году

| Сценарий                                      | Почему NSLock всё ещё встречается | Рекомендация |
|-----------------------------------------------|------------------------------------|--------------|
| Поддержка очень старого кода (iOS 12–14)      | Совместимость                      | Оставлять как есть |
| Рекурсивная блокировка (lock внутри lock)     | NSLock рекурсивный                 | Переписать на actor или отдельные lock’и |
| Интеграция с очень старыми Objective-C библиотеками | Уже используют NSLock              | Обернуть в actor |
| Простейшая защита в legacy-тестах             | Быстро и понятно                   | Переходить на actor |

**В новом коде 2026 года** NSLock считается **антипаттерном** — его использование в pull request обычно вызывает вопросы ревьюера.

### 6. Лучшие практики 2026 года

- **actor** — **основной** выбор для защиты состояния  
- **`os_unfair_lock`** — **единственный** оправданный мьютекс в новом коде  
- **`NSLock`** — **только** в legacy-коде и при рекурсивной блокировке  
- **Никогда** не используйте `NSLock` внутри `@MainActor` — это бессмысленно и дорого  
- **defer { unlock() }** — золотое правило при использовании любого lock  
- **Swift 6 strict concurrency** — включает предупреждения о legacy-синхронизации — не игнорируйте  
- **Тестирование** — **Thread Sanitizer** + **Instruments** + stress-тесты на 16+ ядер  
- **Документируйте** — пишите комментарий «NSLock — legacy, планируется миграция на actor»

**Короткий девиз 2026**:

> Основной инструмент — actor.  
> Для максимальной скорости — os_unfair_lock.
