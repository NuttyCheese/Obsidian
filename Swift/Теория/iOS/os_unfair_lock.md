**os_unfair_lock** — это самый низкоуровневый, быстрый и рекомендуемый Apple примитив блокировки (mutex) в современных [[iOS]]/macOS-приложениях (2026 год).

Это **не** классический мьютекс в стиле pthread_mutex или [[NSLock]] — это **unfair lock**, оптимизированный под современные ARM-процессоры (Apple Silicon M-серии и A-серии).

### Ключевые характеристики os_unfair_lock (2026)

| Характеристика                     | Значение / Особенность                                              | Сравнение с NSLock |
| ---------------------------------- | ------------------------------------------------------------------- | ------------------ |
| Скорость                           | ★★★★★ (самый быстрый мьютекс в экосистеме Apple)                    | 5–10× быстрее      |
| Накладные расходы                  | Минимальные (~10–20 нс на захват/освобождение)                      | Высокие            |
| Рекурсивность                      | **Нет** (non-recursive)                                             | Есть (recursive)   |
| Fairness (очерёдность)             | **Unfair** — первый пришёл — первый обслужен (без гарантии порядка) | Fair (FIFO)        |
| Потокобезопасность                 | Полная                                                              | Полная             |
| Размер в памяти                    | 8 байт ([[struct]])                                                 | ~64+ байт (объект) |
| Поддержка Swift strict concurrency | Полная ([[Sendable]]-safe)                                          | Полная             |
| Официальная рекомендация Apple     | **Основной** примитив блокировки с 2019–2020 годов                  | Legacy             |

### Когда использовать os_unfair_lock в 2026 году

**Используй**, если:

- Критическая секция **очень короткая** (< 100–200 нс)  
- Высокая конкуренция (много потоков/задач дерутся за ресурс)  
- Нужно **максимально выжать производительность** (горячие петли, кэш, счётчики)  
- Заменяешь старый `NSLock` в legacy-коде и хочешь ускорить в 3–10 раз

**Не используй**, если:

- Критическая секция долгая (> 1 мкс) → лучше `actor`  
- Нужна рекурсивная блокировка → используй `NSRecursiveLock` (редко) или перепроектируй  
- Код должен быть максимально читаемым и безопасным → [[actor]] почти всегда лучше  
- Работаешь только с UI или ViewModel → [[@MainActor]]  

### Пример использования (современный стиль 2026)

```swift
import os.lock

final class FastCache<Key: Hashable, Value> {
    private var storage: [Key: Value] = [:]
    private var lock = os_unfair_lock()
    
    func get(_ key: Key) -> Value? {
        os_unfair_lock_lock(&lock)
        defer { os_unfair_lock_unlock(&lock) }
        return storage[key]
    }
    
    func set(_ value: Value, for key: Key) {
        os_unfair_lock_lock(&lock)
        defer { os_unfair_lock_unlock(&lock) }
        storage[key] = value
    }
    
    // Более безопасный вариант с guard
    func updateValue(for key: Key, _ transform: (inout Value?) -> Void) {
        os_unfair_lock_lock(&lock)
        defer { os_unfair_lock_unlock(&lock) }
        transform(&storage[key])
    }
}

// Использование в многопоточном коде
let cache = FastCache<String, Data>()

Task.detached {
    let data = await downloadImage()
    cache.set(data, for: url.absoluteString)
}

Task {
    if let cached = cache.get(url.absoluteString) {
        // используем кэш
    }
}
```

### Сравнение производительности (примерные цифры на M4 / A18 Pro, 2026)

| Операция (1 млн захватов + освобождений) | os_unfair_lock | NSLock | pthread_mutex_t | actor (простой) | Победитель |
|------------------------------------------|----------------|--------|------------------|------------------|------------|
| Пустая критическая секция                | ~12–18 нс      | ~80–150 нс | ~40–80 нс        | ~100–200 нс      | os_unfair_lock |
| Increment счётчика                       | ~20–30 нс      | ~100–180 нс | ~60–100 нс       | ~150–250 нс      | os_unfair_lock |
| Сложная логика (10 операций)             | ~50–80 нс      | ~150–250 нс | ~100–180 нс      | ~200–350 нс      | actor / os_unfair_lock |

### Лучшие практики os_unfair_lock в 2026 году

- **[[defer]] { unlock() }** — золотое правило (гарантирует освобождение даже при [[throw]] / [[return]])  
- **Минимальная критическая секция** — держи внутри lock только необходимое  
- **Не держи lock долго** — >1 мкс → переходи на actor  
- **Не используй в UI-потоке** — бессмысленно и дорого  
- **Swift 6 strict concurrency** — lock должен быть в `nonisolated(unsafe)` контексте или внутри actor  
- **Тестирование** — Thread Sanitizer + Instruments + stress-тесты на 16+ ядер  
- **Документируйте** — пиши комментарий «os_unfair_lock — критическая секция <50 нс»

**Короткий девиз 2026**:
> «os_unfair_lock — это когда тебе нужна максимальная скорость и минимальные накладные расходы на блокировку.  
> В 2026 году это **единственный оправданный** мьютекс в новом коде.  
> Всё остальное — actor.»
