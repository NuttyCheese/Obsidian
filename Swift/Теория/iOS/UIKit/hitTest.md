**hitTest(_:with:)** — это метод класса [[UIView]], который рекурсивно ищет самую подходящую вью под заданной точкой на экране. Он возвращает `UIView?` — вью, которая должна стать первой в цепочке обработки события, или `nil`, если ничего не нашёл.

**Аналогия**: Представьте экран как многослойный пирог. Палец — это игла, которая протыкает пирог. [[hitTest]] — это процесс, который проверяет каждый слой снизу вверх: "Это мой слой? Нет? Тогда следующий". В итоге игла останавливается на самом верхнем подходящем слое.

**Почему важен в 2026 году**:
- Без правильного hitTest касания могут "проваливаться" сквозь вью, игнорироваться или попадать не туда.
- Это основа для кастомных UI: увеличение зоны касания кнопок, пропуск касаний сквозь оверлеи, перехват событий в сложных иерархиях.
- В SwiftUI это скрыто под капотом, но в [[UIKit]] — must-know для любого среднего+ разработчика.

Метод объявлен так:
```swift
@MainActor
func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView?
```
- `point`: Координаты касания **относительно этой вью** (в её системе координат).
- `event`: Опциональный UIEvent (содержит все touches, но редко используется в переопределении).
- Возвращает: UIView? — вью, которая должна получить событие, или nil.

## 2. Как работает `hitTest` под капотом (шаг за шагом)

UIKit не вызывает `hitTest` вручную — это делает система автоматически при каждом касании. Но давайте разберём логику по умолчанию (из Apple docs и исходников UIKit).

Упрощённый код (реальный чуть сложнее, но суть та же):
```swift
func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
    // Шаг 1: Проверяем, может ли эта вью вообще получать касания
    // Если скрыта, прозрачна или отключена — сразу nil
    if isHidden || alpha < 0.01 || !isUserInteractionEnabled {
        return nil
    }
   
    // Шаг 2: Проверяем, находится ли точка внутри границ этой вью
    // point(inside:) — отдельный метод, который можно переопределить
    if !point(inside: point, with: event) {
        return nil
    }
   
    // Шаг 3: Рекурсивно проверяем все subviews (от самого верхнего к нижнему)
    // subviews.reversed() — чтобы начинать с самого переднего (z-index выше)
    for subview in subviews.reversed() {
        // Конвертируем точку в координаты subview
        let convertedPoint = convert(point, to: subview)
       
        // Рекурсивно вызываем hitTest на subview
        if let hit = subview.hitTest(convertedPoint, with: event) {
            return hit // Нашли! Возвращаем самую вложенную подходящую вью
        }
    }
   
    // Шаг 4: Никто из детей не подошёл — значит, это мы сами
    return self
}
```
**Разжёвываем шаг за шагом с аналогией "пирог"**:
1. **Шаг 1: Проверка "доступности" вью**
   Аналогия: Если слой пирога "невидимый" (isHidden = true), "прозрачный" (alpha < 0.01) или "отключённый" (isUserInteractionEnabled = false) — игла просто проходит сквозь него, как будто его нет.
2. **Шаг 2: point(inside:)**
   Аналогия: Игла проверяет, попала ли она в границы слоя (bounds). По умолчанию — просто `bounds.contains(point)`. Но ты можешь переопределить, чтобы сделать слой "больше" или "меньше" для касаний (расширить зону кнопки).
3. **Шаг 3: Рекурсия по subviews**
   Аналогия: Если игла в слое — проверяем все "начинки" (subviews) сверху вниз (reversed() — самый верхний сначала). Для каждой конвертируем координаты (convert(point, to: subview)) и спрашиваем: "Ты под иглой?" Рекурсия — чтобы копать глубже.
4. **Шаг 4: Возврат [[self]]**
   Аналогия: Если ни одна начинка не подошла — значит, игла в самом слое.
**Важно**: Поиск всегда возвращает **самую вложенную** вью (глубже всего в иерархии), чтобы события шли к самой "конкретной" цели (кнопка внутри ячейки внутри таблицы).

## 3. Связь с Responder Chain и Gesture Recognizers

hitTest — это **первый этап**. Он находит стартовую вью.  
Дальше:
- Если вью имеет **gesture recognizers** (тап, свайп, лонг-пресс) — они проверяются первыми. Если жест распознан — событие "съедается" и responder chain не запускается.
- Если нет жестов или они не распознали — событие (touchesBegan, touchesMoved и т.д.) начинает **Responder Chain** от найденной вью вверх по иерархии (nextResponder).

Схема:
```
Событие касания
  ↓
UIWindow.hitTest(point, event)
  ↓
UIView (найденная)
  ↓
Gesture Recognizers (если есть)
  ↓ (если не распознали)
Responder Chain: touchesBegan → nextResponder → ...
```

## 4. Когда и как переопределять hitTest (самые частые кейсы 2026)

Переопределяй hitTest, когда стандартная логика не подходит. Вот топ-5 сценариев:

1. **Расширение зоны касания** (маленькая кнопка должна "ловить" касания вокруг)
```swift
final class BigTouchButton: UIButton {
   
    // Увеличиваем зону на 20 pt со всех сторон
    private let touchInset: CGFloat = -20
   
    override func point(inside point: CGPoint, with event: UIEvent?) -> Bool {
        // Шаг 2: расширяем bounds
        let expandedBounds = bounds.insetBy(dx: touchInset, dy: touchInset)
        return expandedBounds.contains(point)
    }
}
```
   - Почему point(inside:)? Это проще, чем полный hitTest. Переопределяй hitTest только если нужно менять логику subviews.

2. **Пропуск касаний сквозь вью** (полупрозрачный оверлей не должен блокировать кнопки под собой)
```swift
final class OverlayView: UIView {
   
    override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
        // Стандартный поиск
        let hit = super.hitTest(point, with: event)
       
        // Если нашли себя — пропускаем (возвращаем nil)
        if hit == self {
            return nil
        }
       
        // Иначе возвращаем найденную subview
        return hit
    }
}
```

3. **Передача касаний в другую вью** (кастомный контейнер делегирует касания дочернему элементу)
```swift
final class ContainerView: UIView {
   
    private let childView = UIButton(type: .system)
   
    override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
        // Всегда возвращаем childView, даже если точка не в ней
        let converted = convert(point, to: childView)
        if childView.point(inside: converted, with: event) {
            return childView
        }
        return nil
    }
}
```

4. **Игнорирование касаний в определённой области** (вью с "дыркой" для касаний)
```swift
override func point(inside point: CGPoint, with event: UIEvent?) -> Bool {
    // Игнорируем касания в прямоугольнике (0,0 - 50,50)
    let ignoreRect = CGRect(x: 0, y: 0, width: 50, height: 50)
    if ignoreRect.contains(point) {
        return false
    }
    return super.point(inside: point, with: event)
}
```

5. **Перехват всех касаний для всей иерархии** (floating view, которая "ловит" касания даже за пределами)
```swift
override func hitTest(_ point: CGPoint, with event: UIEvent?) -> UIView? {
    // Возвращаем себя, даже если точка не внутри
    return self
}
```

## 5. Связь с Responder Chain (почему это разные вещи)

- **hitTest** — "Где?": Геометрический поиск. Находит **стартовую вью** для события. Это про **пространство** (frame, bounds, subviews).
- **Responder Chain** — "Кто?": Цепочка ответственности. Передаёт событие от найденной вью вверх по иерархии (nextResponder), пока кто-то не обработает. Это про **архитектуру** (UIResponder, touchesBegan, target-action).

Аналогия:  
hitTest — GPS, который находит адрес доставки.  
Responder Chain — курьер, который несёт посылку по цепочке: квартира → подъезд → дом → улица.  
Если hitTest вернул nil — Responder Chain даже не запустится.

## 6. Ключевые свойства, которые влияют на hitTest

| Свойство / Метод | Влияние на hitTest | Когда менять |
| -------------------------- | ----------------------------------------------------- | ----------------------------------------- |
| `isUserInteractionEnabled` | = false → hitTest возвращает nil | Отключить касания для всей вью и subviews |
| `isHidden` | = true → nil | Скрыть визуально + касательно |
| `alpha` | < 0.01 → nil | Полупрозрачные вью не ловят касания |
| `point(inside:with:)` | Переопределение → кастомная проверка, внутри ли точка | Расширить/уменьшить зону касания |
| `clipsToBounds` | Не влияет на hitTest (только на отрисовку) | — |
| `bounds` / `frame` | Основной критерий для point(inside:) | Изменение размера влияет на зону |

## 7. Самые частые ошибки и ловушки 2026

| Ошибка / Ловушка | Симптомы / Последствия | Как исправить |
| ---------------------------------------------- | ---------------------------------------------------- | ------------------------------------ |
| Забыл `isUserInteractionEnabled = true` | Вью не ловит касания | Всегда true для interactive вью |
| Переопределил hitTest, но забыл super.hitTest | subviews не ловят касания | Всегда вызывай super первым |
| Переопределил point(inside:), но не учёл event | Не работает с multitouch или gesture | Редко, но проверь event |
| Вызов hitTest вручную без контекста | Не работает вне [[UIWindow]] иерархии | Вызывай только для тестов / custom |
| Игнор safeAreaInsets в custom hitTest | Касания в статус-баре / home indicator проваливаются | Учитывай safeAreaInsets в расчётах |
| Кастомный hitTest в SwiftUI Hosting (редко) | Конфликты с SwiftUI gesture | Лучше используй [[SwiftUI]] .gesture |

## 8. Итог и ментальная модель

`hitTest` — это **первый фильтр ввода** в UIKit. Он решает, **существует ли касание** для твоей вью. Без правильного hitTest твои кнопки/ячейки/жесты просто не сработают.

Ментальная модель:  
Экран — это стопка прозрачных листов (вью). Палец — лазерный луч. hitTest — сканер, который проверяет каждый лист сверху вниз: "Это твой лазер? Да? Тогда ты и получи событие".  

Responder Chain — следующий шаг: "Ты обработал? Нет? Тогда передай соседу сверху".  

Вместе они делают UIKit **гибким и мощным**: геометрия определяет "где", цепочка — "кто".  

Если ты пишешь кастомный UI — переопределяй hitTest/point(inside:). Если нет — просто знай, что это работает под капотом.
