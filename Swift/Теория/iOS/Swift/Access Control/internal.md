**Internal** в [[Swift]] — это ключевое слово уровня доступа (access level modifier), которое ограничивает видимость сущности **только внутри текущего модуля** (модуль = таргет в Xcode, т.е. приложение, фреймворк, пакет и т.д.).

По состоянию на февраль 2026 года (Swift 6) `internal` остаётся **самым часто используемым** уровнем доступа по умолчанию и самым безопасным для большинства кода в приложении.

### Все уровни доступа в Swift 6 (краткая таблица)

| Уровень доступа           | Видимость                                          | Где можно использовать     | Когда использовать в 2026                                    |
| ------------------------- | -------------------------------------------------- | -------------------------- | ------------------------------------------------------------ |
| [[private]]               | Только внутри текущего файла                       | Самый строгий              | Вспомогательные свойства/методы внутри файла                 |
| [[fileprivate]]           | Весь текущий файл                                  | Редко                      | Когда нужно делить приватные вещи между типами в одном файле |
| `internal` (по умолчанию) | Весь текущий модуль (таргет)                       | Самый частый               | Всё, что не нужно экспортировать наружу                      |
| [[public]]                | Весь модуль + любой импортирующий модуль           | Для публичного [[API]]     | API библиотек, фреймворков, пакетов                          |
| [[open]]                  | То же + подклассы/переопределения в других модулях | Только для классов/методов | Наследуемые классы/методы в публичных библиотеках            |

### Когда `internal` — лучший выбор (реальные сценарии 2026)

1. **Большинство кода в основном приложении**  
   → 80–95% типов, свойств и методов в приложении должны быть `internal`  
   → Нет смысла делать их `public` — никто снаружи не должен их видеть

2. **Внутренние реализации ViewModel / Repository / Service**  
   → ViewModel, UseCase, Interactor, Mapper — почти всегда `internal`

```swift
internal class ProfileViewModel: ObservableObject {
    @Published var user: User?
    
    internal func load() async { ... }
    
    private func parse(data: Data) -> User? { ... }  // ещё строже
}
```

3. **Внутренние расширения и утилиты**  
   → extension на типы внутри модуля

```swift
internal extension String {
    func trimmed() -> String { trimmingCharacters(in: .whitespacesAndNewlines) }
}
```

4. **Модульные фреймворки / пакеты**  
   → Внутренние детали реализации (implementation details) всегда `internal`

```swift
// Внутренний хелпер (не виден снаружи пакета)
internal struct NetworkRequestBuilder {
    func build(url: URL, token: String) -> URLRequest { ... }
}
```

### Лучшие практики использования `internal` в Swift 2026

- **Делай всё internal по умолчанию** — это самый безопасный уровень для кода приложения  
- **Только public / open** — для [[API]], который ты хочешь экспортировать (библиотеки, фреймворки, [[SPM]]-пакеты)  
- **private / fileprivate** — только когда нужно строго ограничить видимость внутри типа или файла  
- **Swift 6 strict concurrency** — `internal` не влияет на конкурентность, но следи за `Sendable` и `@MainActor`  
- **Не делай internal публичным** — если тип используется только внутри модуля — оставь `internal`  
- **Документируйте** — если `internal` содержит важную логику — пиши `/// internal` или комментарий  
- **Тестирование** — internal-типы видны в тест-таргете (если `@testable import`), так что тестируй спокойно

**Короткий девиз 2026**:
> «internal — это когда ты говоришь: «это моё, никому снаружи не показывать».  
> В приложении почти всё должно быть internal — это самый безопасный и правильный уровень доступа.  
> public / open — только для того, что реально является публичным API.»
