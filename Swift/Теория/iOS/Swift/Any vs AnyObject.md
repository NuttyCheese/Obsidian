**`Any`** и **`AnyObject`** — два специальных типа в [[Swift]], которые используются для работы с «неизвестными» или «разнородными» значениями. Они часто путают новичков, потому что выглядят похоже, но решают совершенно разные задачи и имеют принципиально разные ограничения.

Вот подробное, разжёванное сравнение (актуально на 2026 год, Swift 6+).

### 1. Коротко и чётко — главная разница

| Характеристика                        | `Any`                                                                                        | `AnyObject`                        | Когда выбрать в 2026 году                                     |
| ------------------------------------- | -------------------------------------------------------------------------------------------- | ---------------------------------- | ------------------------------------------------------------- |
| Может содержать                       | **Любой** тип: [[struct]], [[class]], [[enum]], [[tuple]], [[func]]tion, [[metatype]] и т.д. | **Только классы** (ссылочные типы) | `Any` — универсальный контейнер, `AnyObject` — только объекты |
| Поддерживает [[value type]]s?         | Да ([[Int]], [[String]], [[Array]], struct, enum…)                                           | Нет                                | `Any` — для смешанных коллекций                               |
| Поддерживает [[reference type]]s?     | Да (классы, AnyObject, протоколы)                                                            | Да                                 | Оба подходят для классов                                      |
| [[Dynamic dispatch]]?                 | Да (если протокол)                                                                           | Да (всегда, потому что классы)     | `AnyObject` быстрее при вызове методов                        |
| Можно привести к конкретному типу?    | Да (`as?`, `as!`)                                                                            | Да (`as?`, `as!`)                  | Оба поддерживают type casting                                 |
| Можно хранить в массиве разных типов? | Да                                                                                           | Да, но только классы               | `Any` — самый гибкий                                          |
| Потеря информации о типе              | Полная (кроме приведения)                                                                    | Частичная (всегда класс)           | `Any` — «чёрный ящик»                                         |
| Использование в Swift 6+              | Явно: `any Protocol`                                                                         | Неявно (как раньше)                | `AnyObject` почти не требует `any`                            |

### 2. Когда использовать `Any` (самые частые реальные сценарии 2026)

| Сценарий                                                    | Почему именно `Any`                                       | Пример кода                                               |
| ----------------------------------------------------------- | --------------------------------------------------------- | --------------------------------------------------------- |
| Универсальный контейнер ([[JSON]], plist, [[UserDefaults]]) | Храним всё подряд: числа, строки, массивы, словари        | `UserDefaults.standard.set(anyValue, forKey: "settings")` |
| Массив/словарь со смешанными типами                         | Разные элементы: Int, String, [[Double]], [[Date]] и т.д. | `let mixed: [Any] = [1, "hello", Date(), [2.5, 3.7]]`     |
| Декодирование неоднородных данных                           | JSON с полями неизвестного типа                           | `let value: Any = try JSONSerialization.jsonObject(...)`  |
| Передача «чего угодно» в функцию                            | Функция-универсал, callback с неизвестным типом           | `func log(_ value: Any) { print(value) }`                 |
| Работа с метатипами и функциями                             | Редко, но возможно (типы как значения)                    | `let type: Any.Type = String.self`                        |

### 3. Когда использовать `AnyObject` (реальные сценарии 2026)

| Сценарий                                           | Почему именно `AnyObject`                         | Пример кода                                                            |
| -------------------------------------------------- | ------------------------------------------------- | ---------------------------------------------------------------------- |
| Работа со старыми Objective-C API                  | Многие Cocoa-фреймворки возвращают `AnyObject?`   | `let obj: AnyObject? = notification.object`                            |
| Динамический вызов методов без знания типа         | Нужно вызвать метод, который есть у всех объектов | `anyObject.perform(#selector(doSomething))`                            |
| Коллекции только классов (legacy-код)              | Когда точно знаешь, что все элементы — классы     | `let objects: [AnyObject] = [vc1, vc2, vc3]`                           |
| [[KVO]], notifications, [[delegate]]s (старый код) | Многие старые API используют `AnyObject`          | `observe(\.value, options: .new) { (obj: AnyObject?, change) in ... }` |
| Оптимизация (редко)                                | `AnyObject` быстрее при dynamic dispatch          | В очень горячих местах (профилирование)                                |

### 4. Самые важные примеры и ловушки 2026 года

#### Пример 1. Массив с `Any` vs `AnyObject`

```swift
// Можно хранить всё
let mixed: [Any] = [1, "hello", Date(), UIButton(), [2.3, 4.5]]
print(mixed)  // работает

// Только классы
let objects: [AnyObject] = [UIButton(), UIViewController(), NSString("test")]
// let objects: [AnyObject] = [1, "hello"]  // Ошибка компиляции
```

#### Пример 2. Приведение типов (самое частое)

```swift
let value: Any = "Hello, world!"

if let str = value as? String {
    print(str.uppercased())  // HELLO, WORLD!
}

let obj: AnyObject = NSString("Test")
if let str = obj as? String {
    print(str)  // Test
}
```

#### Пример 3. Ловушка: `Any` + протокол с `associatedtype`

```swift
protocol Container {
    associatedtype Item
    var count: Int { get }
}

// let c: any Container = [1,2,3]  // Ошибка! Нельзя использовать any с associatedtype
```

Решение: использовать generic или type erasure (AnyContainer).

### 5. Лучшие практики Any / AnyObject в Swift 2026

- **Избегайте `Any` везде, где можно** — это снижает безопасность типов и производительность  
- **Предпочитайте generic** (`<T>`) или конкретные типы  
- **`AnyObject` почти не нужен в чистом Swift** — используйте только при работе с Objective-C API  
- **Не храните `Any` в свойствах классов** — это часто приводит к ошибкам в будущем  
- **Используйте type erasure** (AnyView, AnyPublisher и т.д.) вместо `any` для публичных API  
- **Swift 6 strict concurrency** — `Any` и `AnyObject` могут быть проблемными в concurrent коде → используйте `Sendable` протоколы  
- **Документируйте** — пиши комментарий «[Any] — временный массив смешанных данных из JSON»

**Короткий девиз 2026**:
> `Any` — это **коробка без типа** («тут может быть что угодно»).  
> `AnyObject` — это **коробка только для классов** («тут лежит объект»).  
> В 2026 году:  
> - `Any` — используй редко и осознанно (JSON, UserDefaults, отладка)  
> - `AnyObject` — только для Objective-C совместимости  
> - Всё остальное — конкретные типы или generic.
