**Short-circuit evaluation** (короткое замыкание, короткое вычисление) — это особенность поведения логических операторов `&&` и `||` в Swift (и почти во всех современных языках), при которой **второй операнд вычисляется только в том случае, если результат уже не определён первым операндом**.

Это не просто оптимизация — это **гарантированное поведение языка**, на которое можно и нужно полагаться.

### Как работает короткое замыкание (правила 2026)

| Оператор | Условие, при котором второй операнд НЕ вычисляется | Результат операции в этом случае | Самый частый полезный эффект |
|----------|-------------------------------------------------------|-----------------------------------|-------------------------------|
| `&&`     | Левый операнд == `false`                              | `false`                           | Защита от nil, out-of-bounds, дорогих вызовов |
| `||`     | Левый операнд == `true`                               | `true`                            | Быстрый успех, пропуск ненужных проверок |

**Важно**:  
Оператор `!` (логическое НЕ) **не** использует short-circuit — он всегда вычисляет свой единственный операнд.

### Самые важные и часто используемые паттерны (2026)

#### 1. Защита от nil (самый популярный и критически важный случай)

```swift
if user != nil && user!.isActive {
    // здесь user! безопасен
    grantAccess(user!)
}
```

Без short-circuit второе условие вызвало бы **краш** при `user == nil`.

#### 2. Защита от выхода за границы массива/строки

```swift
if index < array.count && array[index] == target {
    // array[index] безопасно
}
```

#### 3. Пропуск дорогой операции при быстром успехе / провале

```swift
if isPremiumUser || expensiveServerCheck() {
    showPremiumFeatures()
}
```

`expensiveServerCheck()` не вызовется, если пользователь уже премиум.

#### 4. Цепочка проверок с ранним выходом

```swift
if let token = apiToken,
   !token.isEmpty,
   token.count >= 32,
   isTokenValid(token) {          // вызывается только если все предыдущие true
    makeAuthenticatedRequest()
}
```

#### 5. Комбинирование с guard (очень популярный стиль)

```swift
guard let user = currentUser,
      user.isActive,
      user.subscription.isValid else {
    showLoginScreen()
    return
}

// здесь user точно не nil и активен
showDashboard(user)
```

### Опасности и ловушки (чего избегать)

| Ловушка                                      | Почему опасно                                           | Как правильно |
|----------------------------------------------|---------------------------------------------------------|---------------|
| Побочные эффекты в правом операнде           | Может не выполниться → неожиданное поведение            | Не полагайтесь на побочные эффекты в правой части `&&` / `||` |
| Слишком длинные цепочки условий              | Код становится нечитаемым                               | Разбивать на guard или отдельные переменные |
| Использование `!` после `&&` без проверки    | Если short-circuit не сработал — краш                   | Всегда `if let` / `guard let` вместо `&& !` |
| Ожидание, что `||` всегда коротко замкнётся   | Если левая часть false → правая всё равно выполнится    | — |

### Рекомендации 2026 года

- **Пишите** короткие цепочки `&&` / `||` — не больше 2–3 условий в одном выражении  
- **Предпочитайте** `guard let` / `if let` вместо `&&` с `!` — код становится линейным и понятным  
- **Не полагайтесь** на порядок вычисления для побочных эффектов — это антипаттерн  
- **Используйте** `&&` для **защиты** (guard-like поведение), `||` для **быстрого успеха**  
- **Документируйте** сложные цепочки — пишите комментарий «short-circuit — проверка на nil перед обращением к user!.profile»

**Короткий девиз 2026**:
> Short-circuit — это когда Swift говорит: «я уже знаю ответ — зачем мне вычислять дальше?»  
> Самое важное применение:  
> - защита от nil (`obj != nil && obj!.property`)  
> - защита от выхода за границы  
> - пропуск дорогих операций  
> Используйте его **осознанно** — это не просто оптимизация, это **гарантия безопасности**.
