**open** в [[Swift]] — это ключевое слово, которое используется для **уровня доступа** (access level modifier) и позволяет **наследовать** и **переопределять** классы, методы и свойства **из других модулей**.

Это один из самых важных модификаторов доступа в Swift, особенно при создании **библиотек**, **фреймворков**, **[[SPM]]-пакетов** и **публичных [[API]]**, где нужно разрешить расширение функциональности в чужом коде.

### Все уровни доступа в Swift (краткая таблица 2026)

| Уровень доступа             | Видимость                                                                         | Наследование / переопределение из других модулей | Когда использовать в 2026                                |
| --------------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------ | -------------------------------------------------------- |
| [[private]]                 | Только внутри текущего файла                                                      | Нет                                              | Вспомогательные свойства/методы внутри файла             |
| [[fileprivate]]             | Весь текущий файл                                                                 | Нет                                              | Редко (когда нужно делить между типами в файле)          |
| [[internal]] (по умолчанию) | Весь текущий модуль (таргет)                                                      | Нет                                              | Всё, что не нужно экспортировать                         |
| [[public]]                  | Весь модуль + любой импортирующий модуль                                          | Нет (нельзя переопределять)                      | Публичный [[API]] библиотек (методы, свойства, [[init]]) |
| `open`                      | То же, что public + **разрешено наследование и переопределение** в других модулях | **Да**                                           | Базовые классы и методы, которые должны расширяться      |

### Когда использовать `open` в 2026 году (реальные сценарии)

| Сценарий                                                                         | Почему нужен именно `open` (а не public)                                  | Пример                                                    |
| -------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------- |
| Создание **базового класса** для наследования в других модулях                   | Пользователи библиотеки должны иметь возможность создавать свои подклассы | `open class BaseViewController: UIViewController { ... }` |
| **Переопределяемые методы** в публичном API                                      | Пользователи должны иметь возможность менять поведение метода             | `open func viewDidLoad() { ... }`                         |
| **Публичные свойства** с возможностью переопределения                            | Редко, но иногда нужно разрешить override get/set                         | `open var title: String { get set }`                      |
| **Фреймворки / библиотеки** ([[Alamofire]], [[Kingfisher]], SnapKit, TCA и т.д.) | Разработчики хотят, чтобы пользователи могли расширять функциональность   | `open class ImageDownloader { ... }`                      |
| **[[SwiftUI]]** (некоторые базовые View)                                         | `open class View { ... }` — позволяет создавать кастомные View            | —                                                         |

### Примеры использования `open` vs `public` в 2026

#### Правильно (open для наследования)

```swift
// В библиотеке MyUIFramework
open class BaseButton: UIButton {
    open func customizeAppearance() {
        backgroundColor = .systemBlue
        setTitleColor(.white, for: .normal)
    }
    
    open override func layoutSubviews() {
        super.layoutSubviews()
        layer.cornerRadius = bounds.height / 2
    }
}

// В приложении пользователя
class CustomSubmitButton: BaseButton {
    override func customizeAppearance() {
        super.customizeAppearance()
        setTitle("Submit Now", for: .normal)
    }
}
```

#### Ошибка (public вместо open)

```swift
public class BaseButton: UIButton {  // Ошибка: нельзя наследоваться из другого модуля
    public func customizeAppearance() { ... }
}
```

### Лучшие практики `open` в Swift 2026

- **open** — используй **только** для классов, методов и свойств, которые **реально должны переопределяться** в других модулях  
- **public** — для всего остального публичного API (структуры, [[enum]], функции, init, свойства без override)  
- **Минимизируй open** — каждый `open` — это точка расширения, но и потенциальная дыра в безопасности/стабильности API  
- **Swift 6 strict concurrency** — `open` классы и методы должны быть `@MainActor` или Sendable, если используются в UI/Concurrency  
- **Документируйте** — пиши `/// open` или комментарий «переопределяемый метод — можно кастомизировать в подклассах»  
- **Не делай open init** — если нужно, используй `required` + `public`  
- **Тестирование** — проверяй, что подклассы не ломают базовое поведение (override + тесты)

**Короткий девиз 2026**:
> «open — это когда ты говоришь: «мой класс/метод можно наследовать и переопределять из любого модуля».  
> В 2026 году это **единственный** способ разрешить расширение в публичных библиотеках и фреймворках.  
> public — для API, который можно использовать, но не менять.  
> Всё остальное — internal или private.»
