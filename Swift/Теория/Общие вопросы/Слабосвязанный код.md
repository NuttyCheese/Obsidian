# Слабосвязанный код (Loosely Coupled Code) в Swift / iOS

**Слабосвязанность** — это степень, при которой компоненты системы **минимально зависят** друг от друга.  
Чем слабее связь, тем проще менять, тестировать, переиспользовать и масштабировать код.

### Преимущества слабосвязанного кода

- Легко заменять / модифицировать один модуль без влияния на другие
- Упрощённое **юнит-тестирование** (моки, стабы)
- Высокая **повторная используемость** компонентов
- Быстрее **рефакторинг** и добавление фич
- Меньше **спагетти-кода** и **божественных классов**
- Лучшая **поддерживаемость** в команде и при долгосрочной разработке

### Основные способы достижения слабой связанности в [[Swift]] (2026)

| Метод                                                            | Как реализуется в Swift                                                                                                    | Уровень слабости | Пример использования                            |
| ---------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- | ---------------- | ----------------------------------------------- |
| **Протоколы (Interfaces)**                                       | [[protocol]] вместо конкретных классов                                                                                     | ★★★★★            | ViewModel, Service, Repository                  |
| **Dependency Injection**                                         | Через конструктор / property / Environment                                                                                 | ★★★★★            | Swinject, Resolver, Cleanse, ручной DI          |
| **Событийно-ориентированный подход**                             | [[NotificationCenter]], [[Combine]], AsyncStream                                                                           | ★★★★☆            | Реакция на события без прямой зависимости       |
| **[[SOLID]] принципы**                                           | Особенно [[Single Responsibility Principle\|SRP]], [[Open-Closed Principle\|OCP]], [[Dependency Inversion Principle\|DIP]] | ★★★★★            | Разделение ответственности, открытые расширения |
| **Composition over Inheritance**                                 | Предпочтение композиции над наследованием                                                                                  | ★★★★☆            | Вместо subclassing — делегирование              |
| **[[Value Type]]**                                               | [[struct]] вместо [[class]]                                                                                                | ★★★★☆            | Модели данных, ViewState                        |
| **Environment / @Environment**                                   | SwiftUI — передача зависимостей через окружение                                                                            | ★★★★☆            | SwiftUI-приложения                              |
| **[[Coordinator]] / [[Clean Swift (VIP) Architecture\|Router]]** | Отдельный объект для навигации                                                                                             | ★★★★☆            | Координаторный паттерн в [[UIKit]]/SwiftUI      |

### Примеры слабой связанности в коде

#### 1. Через протокол (самый чистый способ)

```swift
protocol NetworkService {
    func fetchUser(id: String) async throws -> User
}

class APIService: NetworkService { … }

class UserViewModel {
    private let service: any NetworkService   // ← слабая связь
    
    init(service: any NetworkService) {
        self.service = service
    }
}
```

#### 2. Dependency Injection через конструктор

```swift
class ProfileViewController {
    private let authService: AuthServiceProtocol
    private let userRepository: UserRepositoryProtocol
    
    init(authService: AuthServiceProtocol,
         userRepository: UserRepositoryProtocol) {
        self.authService = authService
        self.userRepository = userRepository
    }
}
```

#### 3. События через Combine / AsyncStream

```swift
class EventBus {
    static let shared = EventBus()
    let userDidLogin = PassthroughSubject<User, Never>()
}

class LoginViewModel {
    func login() {
        // ...
        EventBus.shared.userDidLogin.send(currentUser)
    }
}

class ProfileViewModel {
    init() {
        EventBus.shared.userDidLogin
            .sink { [weak self] user in
                self?.update(with: user)
            }
            .store(in: &cancellables)
    }
}
```

### Сравнение сильной и слабой связанности

| Аспект                     | Сильная связанность (Tight Coupling)       | Слабая связанность (Loose Coupling)        |
|----------------------------|--------------------------------------------|--------------------------------------------|
| Зависимость от реализации  | Высокая (конкретный класс)                 | Низкая (протокол / интерфейс)              |
| Тестирование               | Сложно (нужны реальные зависимости)        | Легко (моки, стабы)                        |
| Замена компонента          | Требует изменения множества мест           | Изменяется в одном месте                   |
| Повторное использование    | Низкое                                     | Высокое                                    |
| Масштабируемость           | Плохая                                     | Отличная                                   |

### Короткие рекомендации (2026)

- **По умолчанию** — используй `protocol` вместо конкретного класса
- **Dependency Injection** — через конструктор или `@Environment` / `@Observable`
- **События** — Combine, AsyncStream, NotificationCenter (с осторожностью)
- **Избегай** сильных ссылок на конкретные классы в свойствах
- **Проверяй** слабость связи через тесты и рефакторинг: легко ли заменить один сервис на другой?

**Главное правило**:
> «Если ты пишешь имя конкретного класса в типе свойства или параметра — это почти всегда сильная связь.  
> Замени его на протокол — и код станет слабее связанным.»
