В [[Swift]], когда мы говорим о **коллизиях**, чаще всего речь идёт о **хеш-таблицах** — таких структурах данных, как:

- `Dictionary<Key, Value>`
    
- `Set<Element>`
    

Коллизия — это **ситуация, когда два разных ключа имеют одинаковый хеш**, и поэтому потенциально могут «наступить друг на друга» в хеш-таблице.

Важно понимать, что:

> коллизия — это нормальное и ожидаемое явление, которое **должно быть корректно обработано** внутри структуры данных.

---

## 2. Зачем нужны хеши

Хеш-таблица работает так:

```text
ключ → hash → индекс → значение
```

Пример на Swift:

```swift
let dict: [String: Int] = [
    "apple": 1,
    "banana": 2
]
```

- `"apple"` → `hashValue` → индекс в массиве бакетов
    
- `"banana"` → другой `hashValue` → другой индекс
    

Swift использует **хеш-функцию SipHash** для ключей [[Hashable]].

---

## 3. Что такое коллизия

**Коллизия** возникает, когда **разные ключи дают одинаковый хеш / индекс**.

Пример:

```swift
struct BadHash: Hashable {
    let value: Int
    
    func hash(into hasher: inout Hasher) {
        hasher.combine(0) // ужасная хеш-функция
    }
}

let a = BadHash(value: 1)
let b = BadHash(value: 2)

a.hashValue == b.hashValue // true — коллизия
```

Здесь `a` и `b` имеют одинаковый хеш, хотя это разные объекты.

> Коллизии не ошибка — это естественная особенность хеш-таблиц.

---

## 4. Почему коллизии неизбежны

Если у нас N возможных ключей и M ячеек в таблице, и N > M:

- по принципу Дирихле, коллизии **гарантированы**
    
- даже если хеш-функция идеальна, иногда несколько разных ключей попадают в одну ячейку
    

**Следовательно:** любая хеш-таблица должна уметь справляться с коллизиями.

---

## 5. Как Swift справляется с коллизиями

Swift использует **[[open addressing]]** (открытая адресация), а не **связанные списки (chaining)**:

- все элементы хранятся **непосредственно в массиве бакетов**
    
- если ячейка занята → ищем следующую свободную по правилу probing
    
- при удалении используется **[[tombstone]]** (маркер «удалено»)
    

Пример (упрощённо):

```
bucket array: [nil, nil, nil, nil]

insert "apple" → index 1 → [nil, apple, nil, nil]
insert "banana" → index 1 занят → пробуем 2 → [nil, apple, banana, nil]
```

> Таким образом, даже при коллизии элементы успешно хранятся.

---

## 6. Пример на Swift: коллизии на практике

Создадим плохую хеш-функцию для демонстрации коллизий:

```swift
struct CollidingKey: Hashable {
    let id: Int
    
    func hash(into hasher: inout Hasher) {
        hasher.combine(0) // все ключи дают одинаковый hash
    }
}

var dict = [CollidingKey: String]()

dict[CollidingKey(id: 1)] = "One"
dict[CollidingKey(id: 2)] = "Two"
dict[CollidingKey(id: 3)] = "Three"

print(dict)
```

Вывод:

```text
[CollidingKey(id: 2): "Two", CollidingKey(id: 1): "One", CollidingKey(id: 3): "Three"]
```

- Все ключи **коллизируют**, но `Dictionary` корректно хранит все значения
    
- Внутри Swift использует **open addressing** + `==` для различения ключей
    

---

## 7. Какие есть последствия коллизий

1. **Снижение производительности**
    
    - Lookup перестает быть «идеально O(1)»
        
    - При сильной коллизии скорость поиска падает
        
2. **Ресайз таблицы**
    
    - При высоком load factor происходит **перехеширование**
        
    - Все элементы перераспределяются по новым бакетам
        
3. **Не ломает данные**
    
    - Даже при 100% коллизий все ключи различимы через `==`
        
    - Swift гарантирует корректное поведение
        

---

## 8. Как избежать коллизий

- Использовать **качественные `hash(into:)`**
    
- Не злоупотреблять слишком простыми хеш-функциями
    
- Для кастомных типов всегда комбинировать **несколько свойств**:
    

```swift
struct User: Hashable {
    let id: Int
    let name: String
    
    func hash(into hasher: inout Hasher) {
        hasher.combine(id)
        hasher.combine(name)
    }
}
```

- Swift `Dictionary` уже использует хороший hash для стандартных типов ([[String]], [[Int]], [[UUID]])
    

---

## 9. Коллизии и ARC

Интересный нюанс: **внутри Swift `Dictionary` хранит элементы как [[Reference Type]]** для больших структур:

- При коллизиях Swift использует **внутренние структуры с [[ARC]]**, чтобы избежать лишнего копирования
    
- Поэтому большое количество коллизий может увеличить **кол-во [[retain]]/[[release]]** операций

## 10. Проверка коллизий

Для исследования можно сделать:

```swift
var seenHashes = Set<Int>()

for i in 0..<1000 {
    let key = CollidingKey(id: i)
    let hash = key.hashValue
    if !seenHashes.insert(hash).inserted {
        print("Collision detected for id \(i)")
    }
}
```

---

## 11. Итоги

- Коллизия — это нормальное явление в хеш-таблицах
    
- Swift использует **open addressing** для их разрешения
    
- Все ключи различаются через `==`
    
- Плохой `hash(into:)` → коллизии → снижение производительности
    
- ARC и внутреннее хранение могут давать небольшие накладные расходы при коллизиях
    
- Swift гарантирует **корректное поведение [[Dictionary]] и [[Set]] даже при коллизиях**
