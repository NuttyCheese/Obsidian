# 📌 Зачем вообще нужна пагинация

Пагинация — это способ загрузки данных частями.

Если не использовать пагинацию:

- растёт время ответа
    
- увеличивается потребление памяти
    
- возрастает нагрузка на БД
    
- ухудшается UX
    

Особенно критично для:

- мобильных приложений
    
- лент
    
- чатов
    
- каталогов
    
- истории операций
    

---

# 1️⃣ Offset-based pagination

## 📖 Суть

Данные выбираются с использованием `OFFSET` и `LIMIT`.

```http
GET /products?offset=40&limit=20
```

или

```http
GET /products?page=3&limit=20
```

---

## 🧠 Как это работает на уровне БД

```sql
SELECT *
FROM products
ORDER BY id
LIMIT 20 OFFSET 40;
```

База:

1. Сортирует
    
2. Пропускает 40 строк
    
3. Возвращает 20
    

---

## 📊 Схема

```
[1..20]   -> page 1
[21..40]  -> page 2
[41..60]  -> page 3
```

---

## 📦 Пример ответа

```json
{
  "items": [...],
  "total": 1000,
  "page": 3,
  "limit": 20
}
```

---

## ✅ Плюсы

- Простая реализация
    
- Можно получить `total`
    
- Можно перейти на любую страницу
    

## ❌ Минусы

- Медленно на больших таблицах
    
- При вставке/удалении возникают дубликаты
    
- Плохая масштабируемость
    

---

## 📌 Когда использовать

- Админки
    
- Статичные каталоги
    
- Когда нужен `total`
    

---

# 2️⃣ Page-number pagination

Это частный случай offset.

```http
GET /products?page=3
```

Отличается только интерфейсом.

---

## 📊 Схема UI

```
[1] [2] [3] [4] [5]
```

Чаще используется в web.

---

# 3️⃣ Cursor-based pagination

## 📖 Суть

Каждая страница возвращает специальный "курсор", который указывает, с какого места продолжать.

```http
GET /products?cursor=eyJpZCI6MTIzfQ==&limit=20
```

Ответ:

```json
{
  "items": [...],
  "nextCursor": "eyJpZCI6MTQzfQ=="
}
```

---

## 📊 Схема

```
Request #1 → null cursor
Response → items + nextCursor=ABC

Request #2 → cursor=ABC
Response → items + nextCursor=DEF
```

---

## 🧠 Что такое cursor?

Обычно содержит:

- id последнего элемента
    
- timestamp
    
- composite key
    
- иногда шифруется
    

---

## SQL пример

```sql
SELECT *
FROM products
WHERE id > 123
ORDER BY id
LIMIT 20;
```

---

## ✅ Плюсы

- Нет дубликатов
    
- Устойчив к изменениям
    
- Работает быстро
    
- Отлично масштабируется
    

## ❌ Минусы

- Нельзя прыгнуть на страницу 10
    
- Нет total
    

---

## 📌 Лучший вариант для мобильных приложений

Подходит для:

- ленты
    
- каталога
    
- истории
    
- бесконечного скролла
    

---

# 4️⃣ Keyset pagination

По сути упрощённая cursor-пагинация.

```http
GET /products?lastId=123&limit=20
```

---

## SQL

```sql
WHERE id > 123
ORDER BY id
LIMIT 20
```

---

## 📊 Схема

```
lastId = 0
→ 1..20

lastId = 20
→ 21..40
```

---

## Особенность

Работает только при стабильной сортировке.

---

## ✅ Плюсы

- Максимально быстрая
    
- Использует индекс
    
- Минимальная нагрузка
    

## ❌ Минусы

- Нет total
    
- Нет произвольного перехода
    

---

# 5️⃣ Time-based pagination

Используется для данных, привязанных ко времени.

```http
GET /messages?before=2026-02-20T10:00:00Z&limit=50
```

---

## SQL

```sql
WHERE created_at < '2026-02-20'
ORDER BY created_at DESC
LIMIT 50;
```

---

## 📊 Схема (чат)

```
[10:00]
[09:59]
[09:58]
```

---

## 📌 Используется в:

- чатах
    
- уведомлениях
    
- логах
    
- событиях
    

---

# 6️⃣ Infinite Scroll (UI-паттерн)

Это не backend-пагинация, а способ отображения.

На iOS:

```
Пользователь скроллит
→ достиг конца
→ запрос next page
→ append к массиву
→ reload
```

Обычно используется вместе с:

- cursor
    
- keyset

# 📉 Проблема дубликатов (важно!)

Представим:

```
Page 1: 1 2 3 4 5
```

Пока пользователь листает, добавился новый элемент в начало.

```
0 1 2 3 4 5
```

Запрос Page 2:

```
4 5 6 7 8
```

→ 4 и 5 продублировались

Это главная проблема offset-пагинации.

---

# 🏗 Архитектурный взгляд ([[iOS]])

Лучше не привязываться к конкретному типу пагинации.

Пример абстракции:

```swift
protocol PaginationRequest {
    associatedtype Cursor

    var cursor: Cursor? { get }
    var limit: Int { get }
}
```

И отдельно:

```swift
struct PageResponse<T, Cursor> {
    let items: [T]
    let nextCursor: Cursor?
}
```

---

# 🚀 Какой тип выбирать

## Если проект современный

👉 Cursor-based

## Если большая таблица

👉 Keyset

## Если админка и нужен total

👉 Offset

## Если чат

👉 Time-based или Keyset

# 🎯 Итог

Пагинация — это не просто деление на страницы.

Это:

- вопрос производительности
    
- вопрос архитектуры
    
- вопрос масштабируемости
    
- вопрос корректности данных
    

Современные мобильные приложения почти всегда используют cursor / keyset.

Offset — это классика, но плохо масштабируется.

# 📌 Короткое правило

Если нужен:

- стабильный UX
    
- отсутствие дублей
    
- хорошая производительность
    

→ используйте cursor-based pagination.
