**`@Sendable`** — это **атрибут замыкания** ([[closure]]), который говорит компилятору:

> «Это замыкание **потокобезопасно** и его можно безопасно передать между разными задачами ([[Task]]), актёрами ([[actor]]) и потоками без риска гонок данных».

Атрибут `@Sendable` заставляет компилятор **проверить**:

- все **захваченные значения** (captured values) должны быть **Sendable** (либо value-типы, либо типы, явно помеченные как `@Sendable` / `@unchecked Sendable`)  
- внутри замыкания **нельзя мутировать** ничего, что не является изолированным ([[actor]]-[[isolated]]) или Sendable  
- замыкание **не может** содержать небезопасных операций (например, прямой доступ к non-Sendable классу)

**Коротко**:
- `@Sendable` = «это замыкание можно бросить в другой поток / задачу / актёр — и ничего не сломается».

Без `@Sendable` в строгом режиме [[Swift]] 6 компилятор будет ругаться на почти любое замыкание, которое захватывает что-то mutable или non-Sendable.

### 2. Когда @Sendable обязателен (2026)

| Ситуация                                      | Требуется @Sendable? | Почему |
|-----------------------------------------------|-----------------------|--------|
| Замыкание в `Task { ... }` / `Task.detached`  | Да                    | Task может выполняться на любом потоке |
| Замыкание в `async let`                       | Да                    | Параллельное выполнение |
| Замыкание, переданное в API с `@Sendable`     | Да                    | Требование сигнатуры |
| Замыкание внутри `@MainActor` / actor         | Часто да              | Swift 6 требует в строгом режиме |
| Замыкание, которое только читает immutable данные | Нет (часто)           | Компилятор может вывести сам |
| Замыкание, которое мутирует локальные let/var | Нет                   | Локальные значения не захватываются |

**Правило 2026**:
> В Swift 6 с полной проверкой конкурентности (`strict concurrency checking = complete`) почти **все** замыкания, которые передаются в `Task`, `async let`, или используются в concurrency-контексте, **должны** быть помечены `@Sendable` — иначе ошибка компиляции.

### 3. Самые популярные и правильные шаблоны 2026 года

#### Шаблон 1 — Простое безопасное замыкание в [[Task]]

```swift
Task {
    // ✅ @Sendable автоматически выводится
    print("Простая задача")
}

Task { @Sendable in
    // явно указываем — полезно для читаемости
    print("Явно помеченное замыкание")
}
```

#### Шаблон 2 — Захват Sendable-структуры

```swift
struct User: Sendable {
    let id: UUID
    let name: String
}

let user = User(id: UUID(), name: "Alex")

Task { @Sendable in
    print("Пользователь:", user.name)  // ✅ безопасно
}
```

#### Шаблон 3 — Ошибка без Sendable (классика)

```swift
class Logger {
    var logs: [String] = []
}

let logger = Logger()

Task { @Sendable in
    // ❌ Ошибка: Logger не Sendable
    logger.logs.append("Ошибка")
}
```

**Решение**:

```swift
final class SafeLogger: @unchecked Sendable {
    private let queue = DispatchQueue(label: "logger")
    private var logs: [String] = []
    
    func append(_ message: String) {
        queue.async { self.logs.append(message) }
    }
}
```

#### Шаблон 4 — Универсальная функция с @Sendable

```swift
func doDelayedWork(_ action: @escaping @Sendable () async -> Void) async {
    try? await Task.sleep(nanoseconds: 1_000_000_000)
    await action()
}

doDelayedWork { @Sendable in
    print("Отложенная работа")
}
```

### 4. Типичные ошибки и ловушки 2026 года

| Ошибка                                               | Последствия                            | Как избежать                                               |
| ---------------------------------------------------- | -------------------------------------- | ---------------------------------------------------------- |
| Забыть `@Sendable` в замыкании для Task              | Ошибка компиляции в Swift 6            | Всегда добавлять `@Sendable`                               |
| Захват non-Sendable класса                           | Ошибка: «Capture of non-Sendable type» | Сделать класс `@unchecked Sendable` или использовать actor |
| Захват mutable var без синхронизации                 | Гонка данных в [[Runtime]]             | Использовать actor или сериальную очередь                  |
| Думать, что `@Sendable` автоматически синхронизирует | Нет — только проверяет типы            | Самостоятельно обеспечивать безопасность                   |
| Использовать @unchecked Sendable без необходимости   | Скрытые гонки данных                   | Применять только когда точно уверен в безопасности         |

### 5. @Sendable vs другие механизмы конкурентности (2026 сравнение)

| Механизм                 | Защита от гонок данных                         | Требует ручной синхронизации | Уровень безопасности  | Рекомендация 2026  | Когда использовать   |
| ------------------------ | ---------------------------------------------- | ---------------------------- | --------------------- | ------------------ | -------------------- |
| `@Sendable` на замыкании | Проверка на компиляции                         | Да (если захват [[mutable]]) | Высокий (с проверкой) | Основной выбор     | Все замыкания в Task |
| `actor`                  | Полная встроенная                              | Нет                          | Максимальный          | Для состояния      | Mutable данные       |
| `@unchecked Sendable`    | Нет проверки                                   | Полностью на вас             | Низкий (unsafe)       | Только когда нужно | Legacy-классы        |
| `Sendable` протокол      | Проверка типов                                 | Частично                     | Высокий               | Для value-типов    | Структуры, enum      |
| [[DispatchQueue]]        | Ручная ([[DispatchBarrier\|barrier]]/[[sync]]) | Да                           | Средний               | Legacy             | Старый код           |

**Вывод 2026**:
- `@Sendable` — **обязательный** атрибут для почти всех замыканий в конкурентном коде  
- Это **не магия** — это **проверка компилятором**, что вы не передаёте небезопасные данные  
- Лучшая защита — комбинация: `@Sendable` + `actor` + `Sendable`-типы

### 6. Лучшие практики 2026 года

- **Всегда пиши `@Sendable`** в замыканиях для `Task`, `async let`, callback-ах  
- **Делай типы Sendable** — структуры, enum, final-классы с неизменяемым состоянием  
- **Для изменяемых классов** — используй `@unchecked Sendable` + внутреннюю синхронизацию (очень осторожно)  
- **Не захватывай mutable var** в `@Sendable`-замыканиях — это почти всегда ошибка  
- **Для UI** — используй `@MainActor` + `@Sendable`  
- **Swift 6 strict concurrency** — включай полную проверку — она ловит почти все проблемы с `@Sendable`  
- **Тестируйте** — проверяйте Thread Sanitizer и Instruments на гонки данных

**Короткий девиз 2026**:
> «@Sendable — это не просто атрибут, это обещание компилятору: «это замыкание безопасно для любого потока».  
> В 2026 году без него почти ни одно замыкание в Task не скомпилируется в строгом режиме.»
