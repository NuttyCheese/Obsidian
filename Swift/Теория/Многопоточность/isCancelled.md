**isCancelled** — это **свойство типа [[Bool]]** в классе [[Operation]], которое показывает:

- была ли операция **отменена** через вызов `cancel()`  
- или нет

**Ключевые факты 2026 года**:

- `isCancelled` становится `true` **немедленно** после вызова `cancel()`  
- Значение **только для чтения** — вы **не можете** установить его вручную  
- `cancel()` **не останавливает** уже выполняющуюся операцию автоматически — он лишь **меняет флаг**  
- Чтобы операция действительно остановилась, вы **обязаны** проверять `isCancelled` внутри кода операции  
- Свойство **[[KVO]]-совместимо** — можно наблюдать за изменениями через KVO  
- После отмены операция **может** продолжать выполняться, но должна **как можно быстрее** завершиться и установить `isFinished = true`

### 2. Когда и как правильно проверять isCancelled

**Правило №1 2026 года**:
> Если операция может длиться дольше 0.1–0.2 секунды — **обязательно** проверяйте `isCancelled`:
> - в начале `main()` или блока  
> - в каждом тяжёлом цикле  
> - перед каждой длительной операцией

**Когда проверять обязательно**:

- Загрузка данных из сети / диска  
- Обработка изображений / видео  
- Циклы по большим массивам  
- Вычисления (ML, фильтры, сжатие)  
- Любая операция > 100–200 мс

**Когда можно не проверять** (почти никогда в production):

- Очень короткие операции (< 50 мс)  
- Операции без циклов и I/O

### 3. Самые популярные и правильные шаблоны использования в 2026

#### Шаблон 1 — Проверка в [[BlockOperation]] (самый частый)

```swift
let queue = OperationQueue()
queue.maxConcurrentOperationCount = 4

let operation = BlockOperation { [weak self] in
    guard let self else { return }
    
    // Проверка в начале
    if operation.isCancelled {
        print("Операция отменена до начала")
        return
    }
    
    for i in 0..<10000 {
        // Периодическая проверка в цикле
        if operation.isCancelled {
            print("Операция отменена на итерации \(i)")
            return
        }
        
        // тяжёлая работа
        processItem(i)
    }
    
    DispatchQueue.main.async {
        self.updateUI()
    }
}

operation.completionBlock = {
    print("Операция либо завершена, либо отменена")
}

queue.addOperation(operation)

// Отмена при уходе с экрана
override func viewWillDisappear(_ animated: Bool) {
    super.viewWillDisappear(animated)
    operation.cancel()
}
```

#### Шаблон 2 — Кастомная асинхронная операция ([[isAsynchronous]] = true)

```swift
class AsyncDownloadOperation: Operation {
    override var isAsynchronous: Bool { true }
    
    // ... (isExecuting / isFinished с KVO как обычно)
    
    private let url: URL
    private var task: URLSessionDataTask?
    
    init(url: URL) {
        self.url = url
        super.init()
    }
    
    override func start() {
        if isCancelled {
            finish()
            return
        }
        
        _isExecuting = true
        
        task = URLSession.shared.dataTask(with: url) { [weak self] data, _, error in
            guard let self else { return }
            
            if self.isCancelled {
                self.finish()
                return
            }
            
            // обработка результата...
            
            self.finish()
        }
        
        task?.resume()
    }
    
    override func cancel() {
        super.cancel()
        task?.cancel()  // ← важно отменить вложенную задачу!
    }
    
    private func finish() {
        _isExecuting = false
        _isFinished = true
    }
}
```

### 4. Типичные ошибки с isCancelled в 2026

| Ошибка                                                  | Последствия                                                        | Как избежать                                                       |
| ------------------------------------------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------------ |
| Не проверять `isCancelled` вообще                       | Операция продолжает работу после отмены (зря тратит CPU / батарею) | Проверять в начале и в циклах                                      |
| Проверять только в начале                               | Долгая операция не реагирует на отмену                             | Проверять в каждом тяжёлом цикле / перед I/O                       |
| Забыть отменить вложенные задачи ([[URLSession]], etc.) | Задача продолжает работать после отмены операции                   | Отменять task / stream в `cancel()`                                |
| Обновление UI без проверки `isCancelled`                | UI обновляется даже после отмены                                   | Проверять перед обновлением UI                                     |
| Думать, что `cancel()` мгновенно останавливает операцию | Операция продолжает работать                                       | `cancel()` только меняет флаг — остановка на вашей ответственности |

### 5. isCancelled vs современные альтернативы (2026 сравнение)

| Механизм                  | Отмена задач | Проверка отмены внутри задачи | Автоматическая остановка вложенных задач | Рекомендация 2026 | Когда использовать        |
| ------------------------- | ------------ | ----------------------------- | ---------------------------------------- | ----------------- | ------------------------- |
| [[Operation]].isCancelled | Да           | Ручная (проверять флаг)       | Нет (нужно вручную)                      | Legacy            | Зависимости, миграция     |
| Task.cancel()             | Да           | Да (`Task.isCancelled`)       | Да (при правильном использовании)        | Основной выбор    | Асинхронные задачи        |
| [[actor]] + [[Task]]      | Да           | Да (`Task.isCancelled`)       | Да                                       | Основной выбор    | Изменяемое состояние      |
| [[DispatchWorkItem]]      | Да           | Да (`isCancelled`)            | Нет                                      | Legacy            | Простые отменяемые задачи |

**Вывод 2026**:
- **Новый код** → **Task.isCancelled** + `cancel()`  
- **Legacy-код** → `Operation.isCancelled`  
- Полная миграция → всё на `Task` / `actor`

### 6. Лучшие практики 2026 года

- **Переходите на [[Swift Concurrency]]** — Operation уже legacy  
- **Проверяйте `isCancelled`**:
  - в начале операции  
  - в каждом тяжёлом цикле  
  - перед каждой длительной операцией (I/O, сеть, вычисления)  
- **cancel()** — вызывайте при уходе с экрана, смене состояния, ошибке  
- **Для UI** — всегда проверяйте `isCancelled` перед обновлением  
- **Для тестов** — используйте `XCTestExpectation` + `cancel()`  
- **Мониторинг** — добавляйте `os_signpost` для отслеживания времени отмены  
- **Swift 6 strict concurrency** — часто подсвечивает небезопасную работу с отменяемыми операциями

**Короткий девиз 2026**:
> «isCancelled — это флаг, который говорит: «Пользователь / система сказал остановиться».  
> В 2026 году его нужно проверять везде, где операция может длиться > 100 мс.  
> Но лучший способ отмены — это Task.cancel() в Swift Concurrency.»
