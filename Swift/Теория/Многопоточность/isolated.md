**`isolated`** — это **атрибут параметра функции** (или [[self]]), который говорит компилятору:

> «Этот параметр (или `self`) **уже находится в изолированном контексте** нужного актёра.  
> Поэтому внутри функции можно обращаться к его изолированным свойствам и методам **без [[await]] и без ошибок изоляции**».

Без `isolated` компилятор в строгом режиме требует `await` для доступа к любому изолированному члену актёра — даже если ты уже «внутри» правильного executor-а.

С `isolated` ты явно говоришь:  
«Мы уже на правильном executor-е этого актёра, можно работать напрямую».

**Ключевые моменты** (2026):

- `isolated` работает **только** с параметрами типа `Actor` (или протоколом, унаследованным от `Actor`)  
- Чаще всего используется с `isolated self` или `isolated any ActorType`  
- Позволяет **обойти необходимость `await`** внутри функции  
- Очень полезен для **универсальных функций**, **замыканий**, **делегатов** и **передачи актёров** как параметров

### 2. Где и когда `isolated` обязателен / полезен

| Ситуация                                              | Без `isolated`                    | С `isolated`                             | Когда применять       |
| ----------------------------------------------------- | --------------------------------- | ---------------------------------------- | --------------------- |
| Функция принимает `actor` и работает с его состоянием | Ошибка: нужен `await actor.value` | Прямой доступ: `actor.value += 1`        | Почти всегда          |
| Замыкание внутри `actor`, которое трогает `self`      | Ошибка или потеря контекста       | `isolated self` — безопасный захват      | Часто                 |
| Универсальная функция для любого `actor`              | Ошибка изоляции при передаче      | `isolated(any) ActorType` — универсально | Для [[generic]]-кода  |
| Делегат / callback, вызываемый из актёра              | Ошибка при доступе к состоянию    | `isolated self` в замыкании              | Делегаты, уведомления |
| Передача `actor` как параметр в `Task`                | Ошибка Sendable или изоляции      | `isolated` + `@Sendable` замыкание       | Редко, но мощно       |

### 3. Самые популярные шаблоны 2026 года

#### Шаблон 1 — Простой параметр `isolated` (самый частый)

```swift
actor Counter {
    var value = 0
}

func add(delta: isolated Counter, amount: Int) {
    delta.value += amount  // без await — потому что isolated
}

let counter = Counter()

Task {
    await add(delta: counter, amount: 42)
    print(await counter.value)  // 42
}
```

#### Шаблон 2 — `isolated self` в замыкании (очень частый)

```swift
actor Logger {
    private var messages: [String] = []
    
    func log(_ message: String) {
        Task { [isolated self] in
            // self уже изолирован → можно работать напрямую
            messages.append(message)
            print("Logged:", message)
        }
    }
}
```

#### Шаблон 3 — Универсальная функция с `isolated(any)`

```swift
func clearStorage<S: Actor>(_ storage: isolated(any) S) where S: Storage {
    storage.items.removeAll()
    storage.lastCleared = Date()
}

actor UserStorage: Storage {
    var items: [String] = []
    var lastCleared: Date?
}

let users = UserStorage()
await clearStorage(users)  // работает без await внутри функции
```

#### Шаблон 4 — `isolated` + `@Sendable` замыкание

```swift
actor Counter {
    var value = 0
    
    func runWithClosure(_ closure: @Sendable (isolated Counter) -> Void) {
        closure(self)  // self уже isolated
    }
}

let counter = Counter()

Task {
    await counter.runWithClosure { isolatedCounter in
        isolatedCounter.value += 100
    }
}
```

### 4. Типичные ошибки и ловушки 2026 года

| Ошибка                                      | Последствия                              | Как избежать |
|---------------------------------------------|------------------------------------------|--------------|
| Поставить `isolated` на не-actor тип        | Ошибка компиляции                        | Только `Actor`-типы |
| Вызвать функцию с `isolated` параметром без `await` | Ошибка компиляции                        | Всегда `await function(actor: actor)` |
| Забыть `[isolated self]` в замыкании        | Ошибка изоляции                          | Явно захватывать `[isolated self]` |
| Комбинировать `isolated` и `@MainActor`     | Конфликт изоляции                        | Нельзя одновременно |
| Думать, что `isolated` отключает проверку   | Нет — только разрешает прямой доступ     | Компилятор всё равно проверяет безопасность |

### 5. `isolated` vs другие механизмы изоляции (2026 сравнение)

| Механизм                  | Требуется await при доступе | Изоляция наследуется | Универсальность | Рекомендация 2026 | Когда использовать |
|---------------------------|------------------------------|-----------------------|-----------------|-------------------|---------------------|
| `isolated(any) Actor`     | Нет (внутри функции)         | Да (от параметра)     | Высокая         | Современный       | Универсальные функции |
| `isolated self`           | Нет (внутри замыкания)       | Да (от self)          | Средняя         | Часто             | Замыкания в actor |
| `@MainActor`              | Нет (если уже на main)       | Да (глобально)        | Низкая          | Для UI            | ViewModel, UI       |
| обычный `actor` метод     | Да                           | Да                    | Низкая          | Основной          | Прямые вызовы       |
| `@_inheritActorContext`   | Нет                          | Да                    | Средняя         | Legacy            | Редко, обходной     |

**Вывод 2026**:
- `isolated`([[any]]) — **самый современный** и **мощный** способ писать универсальные функции, работающие внутри любого актёра  
- `isolated self` — **стандарт** для безопасного захвата в замыканиях внутри актёра  
- Заменяет старые костыли ([[@_inheritActorContext]]) и делает код чище и безопаснее

### 6. Лучшие практики 2026 года

- **Используй `isolated(any)`** для всех функций, которые работают с состоянием переданного актёра  
- **Всегда явно захватывай `[isolated self]`** в замыканиях внутри актёра  
- **Не комбинируй** `isolated` и `@MainActor` на одной функции — конфликт  
- **Для UI** — предпочитай `@MainActor`  
- **Тестирование** — легко мокать через протоколы и `isolated(any)`  
- **Swift 6 strict concurrency** — полностью поддерживается и очень полезен

**Короткий девиз 2026**:
> «isolated — это когда ты говоришь функции: «мы уже внутри правильного актёра, работай напрямую, без лишних await».  
> Это один из самых красивых и мощных инструментов современной конкурентности в Swift.»
