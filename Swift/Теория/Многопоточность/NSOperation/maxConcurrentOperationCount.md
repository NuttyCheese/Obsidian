**maxConcurrentOperationCount** — это свойство объекта [[OperationQueue]], которое определяет:

- **максимальное количество операций**, которые могут выполняться **одновременно** (параллельно) в данной очереди

**Ключевые факты 2026 года**:

- Значение по умолчанию: **-1** → означает «нет ограничения» (очередь может запускать столько операций одновременно, сколько позволит система)  
- Значение **1** → очередь становится **сериальной** (все операции выполняются строго последовательно)  
- Положительные значения (2, 4, 8 и т.д.) → очередь становится **конкурентной** с заданным лимитом параллелизма  
- Значение **0** или отрицательное (кроме -1) → запрещено (выбросит исключение или проигнорируется)

**Самое важное правило 2026 года**:
> Почти никогда **не оставляйте значение -1** в production-приложении.  
> Это одна из самых частых причин **[[Thread Explosion]]** и перегрузки [[CPU]] / батареи.

### 2. Как правильно устанавливать maxConcurrentOperationCount

| Рекомендуемое значение | Когда использовать (2026)                                          | Почему именно это значение                                | Пример сценария                                         |
| ---------------------- | ------------------------------------------------------------------ | --------------------------------------------------------- | ------------------------------------------------------- |
| **1**                  | Нужна строгая последовательность (аналог [[serial]] queue)         | Гарантированный порядок, простая защита от race condition | Работа с базой данных, последовательные сетевые запросы |
| **3–5**                | Большинство UI-приложений (загрузка изображений, обработка данных) | Баланс между скоростью и стабильностью                    | Загрузка 10–20 изображений в галерее                    |
| **6–10**               | Высокопроизводительные задачи на мощных устройствах                | Максимум для большинства современных iPhone/iPad          | Обработка фото, ML-инференс, сжатие видео               |
| **-1** (по умолчанию)  | Только в очень редких случаях                                      | Риск [[Thread Explosion]] и перегрузки                    | Почти никогда в production                              |
| **0**                  | Запрещено                                                          | Выбрасывает исключение или игнорируется                   | —                                                       |

**Рекомендация Apple 2026**:
> «Для большинства приложений оптимально 3–6 параллельных операций.  
> На старых устройствах (iPhone 8 и ниже) лучше 2–4.  
> На мощных (iPhone 16 Pro, M-чипы) можно поднять до 8–10.»

### 3. Самые популярные шаблоны использования в 2026 году

#### Шаблон 1 — Ограничение загрузки изображений (самый частый)

```swift
let imageQueue = OperationQueue()
imageQueue.maxConcurrentOperationCount = 4          // не больше 4 одновременно
imageQueue.qualityOfService = .utility

var images: [UIImage?] = []

for url in imageURLs {
    let operation = BlockOperation {
        guard let data = try? Data(contentsOf: url),
              let image = UIImage(data: data) else { return }
        
        DispatchQueue.main.async {
            images.append(image)
            collectionView.reloadData()
        }
    }
    
    imageQueue.addOperation(operation)
}
```

#### Шаблон 2 — Защита от перегрузки сети

```swift
let networkQueue = OperationQueue()
networkQueue.maxConcurrentOperationCount = 5  // не больше 5 одновременных запросов

for request in apiRequests {
    let operation = BlockOperation {
        let result = performAPIRequestSync(request)
        // обработка результата
    }
    
    networkQueue.addOperation(operation)
}
```

#### Шаблон 3 — Сериализация операций (maxConcurrent = 1)

```swift
let databaseQueue = OperationQueue()
databaseQueue.maxConcurrentOperationCount = 1  // строго последовательное выполнение

func saveUser(_ user: User) {
    let op = BlockOperation {
        saveUserToCoreData(user)
    }
    
    databaseQueue.addOperation(op)
}
```

### 4. Типичные ошибки с maxConcurrentOperationCount в 2026

| Ошибка                          | Последствия                                    | Как избежать                                  |
| ------------------------------- | ---------------------------------------------- | --------------------------------------------- |
| Оставить -1 (по умолчанию)      | Thread Explosion, перегрузка [[CPU]] / батареи | Всегда явно задавать 3–8                      |
| maxConcurrentOperationCount = 0 | Очередь не работает (исключение)               | Никогда не ставить 0                          |
| Слишком большое значение (20+)  | Замедление устройства, лаги UI                 | 4–8 — золотая середина                        |
| Не учитывать слабые устройства  | Плохая производительность на старых iPhone     | Тестировать на iPhone 8 / SE                  |
| Забыть про [[QoS]]              | Низкоприоритетные задачи могут «забиваться»    | Всегда задавать `.utility` / `.userInitiated` |

### 5. OperationQueue.maxConcurrentOperationCount vs современные альтернативы (2026 сравнение)

| Механизм                         | Ограничение concurrency                   | Отмена задач | Зависимости | Потокобезопасность | Рекомендация 2026   | Когда использовать    |
| -------------------------------- | ----------------------------------------- | ------------ | ----------- | ------------------ | ------------------- | --------------------- |
| [[OperationQueue]]               | Да (maxConcurrent)                        | Да           | Да          | Ручная             | Legacy              | Зависимости, миграция |
| [[actor]]                        | Да (1 поток)                              | Да           | Частично    | Встроенная         | Основной выбор      | Изменяемое состояние  |
| [[TaskGroup]]                    | Да (addTask лимит)                        | Да           | Нет         | Встроенная         | Параллельные задачи | Асинхронные загрузки  |
| [[async let]]                    | Да (параллельно)                          | Да           | Нет         | Встроенная         | Простые сценарии    | 2–5 независимых задач |
| [[DispatchQueue]].[[concurrent]] | Нет (только [[DispatchBarrier\|barrier]]) | Ограниченно  | Нет         | Ручная             | Legacy              | Редко                 |

**Вывод 2026**:
- **Новый код** → **TaskGroup** (с `addTask` и лимитом) или **actor**  
- **Сложные зависимости / отмена** → **OperationQueue** (если не хотите переписывать)  
- Полная миграция → всё на Swift Concurrency

### 6. Лучшие практики 2026 года

- **Всегда явно задавайте** `maxConcurrentOperationCount` (3–8 в большинстве случаев)  
- **Не оставляйте -1** — это главная причина Thread Explosion  
- **isCancelled** — проверяйте в начале каждой операции и в тяжёлых циклах  
- **Для UI** — только `@MainActor` / `MainActor.run`  
- **Для тестов** — используйте `XCTestExpectation` + `waitUntilFinished`  
- **Мониторинг** — добавляйте `os_signpost` для отслеживания длительности операций в Instruments  
- **Swift 6 strict concurrency** — часто подсвечивает небезопасную передачу данных в операции

**Короткий девиз 2026**:
> «maxConcurrentOperationCount — это когда ты хочешь контролировать, сколько задач могут работать одновременно.  
> В 2026 году -1 — это почти всегда ошибка.  
> Золотая середина — 4–8.  
> А вообще лучший лимит concurrency — это actor + TaskGroup.»
