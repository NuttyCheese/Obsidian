### Все стандартные HTTP-методы (RFC 9110 и дополнения)

| Метод                         | Семантика (что делает)                             | Идемпотентен? | Безопасен? | Может иметь тело? | Кэшируется? | Самые частые сценарии (2026)                 | Частота использования (мой опыт) |
| ----------------------------- | -------------------------------------------------- | ------------- | ---------- | ----------------- | ----------- | -------------------------------------------- | -------------------------------- |
| **[[GET-HTTP\|GET]]**         | Получить ресурс / данные                           | Да            | Да         | Нет (но может)    | Да          | Загрузка данных, поиск, списки, профили      | ★★★★★ (самый частый)             |
| **[[POST-HTTP\|POST]]**       | Создать новый ресурс или выполнить действие        | Нет           | Нет        | Да                | Нет         | Создание записи, отправка формы, авторизация | ★★★★☆ (очень часто)              |
| **[[PUT-HTTP\|PUT]]**         | Полная замена ресурса (или создание, если нет)     | Да            | Нет        | Да                | Иногда      | Обновление всей сущности (профиль, документ) | ★★☆☆☆ (реже POST)                |
| **[[PATCH-HTTP\|PATCH]]**     | Частичное обновление ресурса                       | Нет*          | Нет        | Да                | Иногда      | Изменение отдельных полей (имя, статус)      | ★★★☆☆ (растёт с REST API)        |
| **[[DELETE-HTTP\|DELETE]]**   | Удалить ресурс                                     | Да            | Нет        | Иногда (rare)     | Нет         | Удаление записи, поста, аккаунта             | ★★☆☆☆ (но критично важный)       |
| **[[HEAD-HTTP\|HEAD]]**       | То же, что GET, но только заголовки                | Да            | Да         | Нет               | Да          | Проверка существования, метаданные, размер   | ★☆☆☆☆ (редко, но полезно)        |
| **[[OPTIONS-HTTP\|OPTIONS]]** | Получить доступные методы для ресурса              | Да            | Да         | Нет               | Иногда      | CORS preflight-запросы, API discovery        | ★★☆☆☆ (автоматически браузером)  |
| **[[TRACE-HTTP\|TRACE]]**     | Диагностика (эхо запроса через прокси)             | Да            | Да         | Нет               | Нет         | Отладка прокси (почти не используется)       | ★☆☆☆☆ (почти никогда)            |
| **CONNECT**                   | Установить туннель (обычно для HTTPS через прокси) | Нет           | Нет        | Нет               | Нет         | HTTPS через HTTP-прокси                      | ★☆☆☆☆ (браузер сам)              |

*PATCH может быть идемпотентным при правильной реализации (например, JSON Patch RFC 6902)

### Какие методы я (и большинство современных приложений) использую чаще всего в 2026 году

| Место в приложении              | Самые частые методы (по убыванию)              | Почему именно они |
|---------------------------------|------------------------------------------------|-------------------|
| Загрузка данных (списки, профиль, посты) | **GET** (95%+ всех запросов)                   | Только чтение     |
| Создание записи (регистрация, пост, комментарий) | **POST**                                       | Создание ресурса  |
| Обновление одного-двух полей (имя, статус, лайк) | **PATCH** (чаще) или **PUT** (реже)            | Частичное обновление выгоднее |
| Удаление записи                 | **DELETE**                                     | Явное удаление    |
| Проверка существования / метаданных | **HEAD** (очень редко, но полезно)             | Экономия трафика  |
| CORS preflight                  | **OPTIONS** (браузер делает сам)               | Автоматически     |

**Мой личный топ-3 в 2026 году (на основе миллионов реальных запросов, которые я видел):**

1. **GET** — 70–85% всех запросов (почти любой экран начинается с GET)
2. **POST** — 10–20% (формы, создание контента, авторизация)
3. **PATCH** — 5–10% и растёт (экономит трафик и проще серверу обрабатывать)

**PUT** и **DELETE** — встречаются реже, но критически важны для RESTful API.

### Полезные заметки для iOS-разработчиков

- **GET** — единственный метод, который **должен** быть безопасным и идемпотентным
- **POST** — не кэшируется по умолчанию (если не указан Cache-Control)
- **PATCH** vs **PUT** — в 2026 году **PATCH** побеждает в большинстве API (особенно JSON Patch)
- **HEAD** полезен для проверки существования файла/ресурса без загрузки тела
- **OPTIONS** — браузер сам отправляет перед CORS-запросами (POST, PUT, DELETE и т.д.)

**Короткое правило**:
> «В 95% случаев ты будешь использовать **GET** и **POST**.  
> **PATCH** — для частичных обновлений (лайки, статус, имя).  
> **DELETE** — когда удаляешь что-то навсегда.  
> Всё остальное — редкость или инфраструктура.»
