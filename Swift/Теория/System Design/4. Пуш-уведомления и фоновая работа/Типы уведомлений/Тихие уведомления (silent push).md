#system_design
## Определение

**Silent push** — это тип push-уведомлений, которые **не отображаются пользователю** на экране, но позволяют приложению **выполнять фоновую работу**, например синхронизировать данные или обновлять локальный кэш.

> Silent push полезен для оффлайн-приложений, обновления контента и подготовки UI без вмешательства пользователя.

---

## Основные характеристики

|Характеристика|Описание|
|---|---|
|Видимость|Пользователь **не видит уведомление**|
|Взаимодействие|Приложение получает событие и может выполнить код в фоне|
|Payload|`content-available: 1` (APNs), `data message` (FCM)|
|Ограничения|Ограниченное время выполнения (до ~30 секунд)|
|Цель|Фоновая синхронизация, обновление данных, prefetch контента|

---

## Payload для silent push

### APNs (iOS)

```json
{
  "aps": {
    "content-available": 1
  },
  "customData": {
    "updateType": "syncOrders"
  }
}
```

### FCM

```json
{
  "to": "DEVICE_FCM_TOKEN",
  "content_available": true,
  "priority": "high",
  "data": {
    "updateType": "syncOrders"
  }
}
```

**Ключевые моменты:**

- `content-available: 1` → указывает системе, что уведомление silent.
    
- `priority: high` → рекомендуется для своевременной доставки.
    
- `data` / `customData` → передача информации для фоновой обработки.

## Работа в приложении ([[iOS]])

### 1. Обработка silent push

```swift
func application(_ application: UIApplication,
                 didReceiveRemoteNotification userInfo: [AnyHashable: Any],
                 fetchCompletionHandler completionHandler: @escaping (UIBackgroundFetchResult) -> Void) {
    if let updateType = userInfo["updateType"] as? String {
        switch updateType {
        case "syncOrders":
            syncOrders { success in
                completionHandler(success ? .newData : .failed)
            }
        default:
            completionHandler(.noData)
        }
    } else {
        completionHandler(.noData)
    }
}
```

### 2. Ограничения iOS

- Приложение должно быть **в фоне или приостановлено**.
    
- Время выполнения ограничено (~30 секунд).
    
- Система может отклонить silent push при высокой нагрузке или низкой батарее.
    
- Не гарантирована **немедленная доставка**.

## Применение silent push

1. **Синхронизация данных**
    
    - Обновление заказов, сообщений, новостей, кэша без вмешательства пользователя.
        
2. **Prefetch контента**
    
    - Подготовка данных для UI перед открытием приложения.
        
3. **Обновление статусов**
    
    - Обновление статусов заказов, задач, активных сессий.
        
4. **Eventual consistency**
    
    - Позволяет поддерживать согласованность данных между клиентом и сервером в оффлайн-режиме.
        

---

## Best Practices

1. **Использовать только для фоновых задач** — не для уведомления пользователя.
    
2. **Минимизировать время выполнения** — быстро обрабатывать и сохранять данные.
    
3. **Использовать `priority: high`** для своевременной доставки.
    
4. **Сохранять данные локально** ([[Core Data]], [[Realm]], [[SQLite]]) перед отправкой на сервер.
    
5. **Комбинировать с очередями изменений** для повторной синхронизации при неудаче.

## Отличия от UI push

|Характеристика|UI push|Silent push|
|---|---|---|
|Видимость|Видно пользователю|Не видно|
|Взаимодействие|Пользователь нажимает|Выполняется автоматически|
|Payload|`alert`, `badge`, `sound`|`content-available`, `data`|
|Цель|Информирование|Фоновая работа, синхронизация|

---

## Итог

- **Silent push** — основной механизм фонового обновления данных на iOS.
    
- Используется для синхронизации, prefetch и поддержки оффлайн-приложений.
    
- Важные элементы: **`content-available`, priority, обработка в `didReceiveRemoteNotification`**.
    
- Позволяет **поддерживать актуальность данных без вмешательства пользователя**.
