#system_design
## Определение

**[[HTTPS]] (HyperText Transfer Protocol Secure)** — это **защищённая версия [[HTTP]]**, использующая шифрование для защиты данных, передаваемых между клиентом ([[iOS]]-приложением) и сервером.

**TLS (Transport Layer Security)** — протокол, обеспечивающий конфиденциальность, целостность и аутентичность данных. **TLS 1.3** — последняя версия протокола с улучшенной безопасностью и производительностью.

---

## 1. Принцип работы HTTPS

1. **Инициализация соединения** → клиент обращается к серверу по HTTPS.
    
2. **TLS handshake** → сервер отправляет сертификат, клиент проверяет подлинность.
    
3. **Установка шифрованного канала** → обмен ключами для симметричного шифрования данных.
    
4. **Передача данных** → весь трафик зашифрован, что предотвращает MITM и подслушивание.
    

---

## 2. Преимущества TLS 1.3

|Особенность|Описание|
|---|---|
|Ускоренный handshake|Меньше раундов обмена ключами → быстрее устанавливается соединение|
|Современные шифры|Только безопасные алгоритмы (AES-GCM, ChaCha20-Poly1305)|
|Прямое forward secrecy|Даже при компрометации ключей прошлые сессии остаются защищёнными|
|Упрощённая структура|Меньше уязвимостей в реализации протокола|
|Защита от downgrade-атак|Приложения не могут быть принуждены использовать устаревшие версии TLS|

## 3. Интеграция HTTPS в iOS

### [[URLSession]] (основной способ сетевого взаимодействия)

```swift
let url = URL(string: "https://api.server.com/data")!
var request = URLRequest(url: url)
request.httpMethod = "GET"

let session = URLSession(configuration: .default)
session.dataTask(with: request) { data, response, error in
    guard let data = data, error == nil else { return }
    // обработка полученных данных
}.resume()
```

**Особенности iOS:**

- `URLSession` автоматически использует TLS для HTTPS.
    
- Поддерживается TLS 1.2 и 1.3 (в зависимости от версии iOS и сервера).
    
- Поддержка **certificate pinning** для дополнительной безопасности.
    

---

## 4. Best Practices для iOS

1. **Использовать только HTTPS** → блокировать HTTP-трафик (App Transport Security).
    
2. **Обновление TLS и сертификатов** → поддерживать TLS 1.3 на сервере.
    
3. **Certificate Pinning** → защита от MITM-атак (см. предыдущий конспект).
    
4. **Forward secrecy** → обеспечивается автоматически в TLS 1.3.
    
5. **Отслеживание устаревших шифров** → сервер должен поддерживать только современные алгоритмы.
    
6. **Проверка hostname** → убедиться, что сертификат принадлежит нужному серверу.
    

---

## 5. TLS 1.3 vs TLS 1.2

|Параметр|TLS 1.2|TLS 1.3|
|---|---|---|
|Handshake|2-3 раунда|1 раунд (ускорено)|
|Алгоритмы|Широкий набор, включая устаревшие|Только современные, безопасные|
|Forward secrecy|Не всегда|По умолчанию|
|Уязвимости|Более подвержен downgrade и атакам BEAST/POODLE|Минимизированы|
|Производительность|Меньше оптимизирован|Быстрее установка соединения, меньше вычислений|

---

## 6. Итог

- **HTTPS + TLS 1.3** → базовый уровень защиты сетевого взаимодействия в iOS.
    
- **TLS 1.3** обеспечивает: скорость, forward secrecy, современные алгоритмы, защиту от атак.
    
- **iOS-разработка:** использовать URLSession, HTTPS, certificate pinning, проверку hostname и актуальные сертификаты.
    
- В сочетании с OAuth, JWT и Secure Enclave TLS 1.3 создаёт **надежный и безопасный стек для мобильного приложения**.
