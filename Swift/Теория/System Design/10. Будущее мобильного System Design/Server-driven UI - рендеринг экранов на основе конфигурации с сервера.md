#system_design
## Определение

**Server-driven UI (SDUI)** — это подход, при котором **структура и содержимое экранов мобильного приложения формируется на сервере**, а клиент (iOS-приложение) получает **конфигурацию и рендерит интерфейс динамически**.

Цель:

- Сократить количество обновлений приложения через App Store.
    
- Позволить команде сервера быстро изменять интерфейс без новых релизов.
    
- Обеспечить единообразный UI на разных платформах ([[iOS]], Android).
    

---

## 1. Принцип работы

1. **Сервер** хранит конфигурации экранов в формате JSON (или YAML).
    
2. **Клиент** запрашивает конфигурацию при открытии экрана.
    
3. **Рендеринг UI** происходит динамически на устройстве на основе полученных данных.
    

```json
{
  "screen": "home",
  "components": [
    {"type": "banner", "imageUrl": "https://..."},
    {"type": "button", "title": "Buy Now", "action": "openProduct?id=123"},
    {"type": "text", "content": "Welcome!"}
  ]
}
```

4. iOS-приложение интерпретирует [[JSON]] и строит **[[Swift/Теория/Swift/UIKit/UIView]] или [[SwiftUI]] View**.
    

---

## 2. Примеры компонентов SDUI

| Компонент         | Описание                                          |
| ----------------- | ------------------------------------------------- |
| Banner / Image    | Изображение или карусель изображений              |
| Text / Label      | Статический или динамический текст                |
| Button            | Действия: переходы, [[deep link]], вызовы [[API]] |
| List / Table      | Список элементов с динамическим контентом         |
| Form / Input      | Поля ввода данных (форма)                         |
| Container / Stack | Layout контейнеры для вложенных элементов         |

---

## 3. Преимущества

| Преимущество                         | Описание                                                                    |
| ------------------------------------ | --------------------------------------------------------------------------- |
| **Быстрое изменение UI**             | Сервер может менять контент и расположение компонентов без App Store релиза |
| **Единая логика для платформ**       | iOS и Android могут рендерить один и тот же [[JSON]]                        |
| **A/B тестирование и feature flags** | Легко показывать разные экраны разным группам пользователей                 |
| **Снижение числа релизов**           | Новые баннеры, кнопки или секции не требуют сборки приложения               |

---

## 4. Ограничения и риски

1. **Скорость и offline**
    
    - Требуется кэширование конфигураций, иначе при отсутствии сети экран не загрузится.
        
2. **Сложность разработки**
    
    - Клиент должен поддерживать множество типов компонентов.
        
    - При увеличении количества компонентов система может стать тяжеловесной.
        
3. **Версионирование**
    
    - Сервер и клиент должны быть согласованы по поддерживаемым компонентам.
        
    - Ошибки конфигурации могут ломать UI.
        
4. **Безопасность**
    
    - Нельзя доверять серверу полностью: проверка действий и контента обязательна.

## 5. Пример реализации на iOS

```swift
struct Component: Decodable {
    let type: String
    let title: String?
    let imageUrl: String?
    let action: String?
}

struct Screen: Decodable {
    let components: [Component]
}

func render(screenData: Data) {
    let decoder = JSONDecoder()
    guard let screen = try? decoder.decode(Screen.self, from: screenData) else { return }

    for component in screen.components {
        switch component.type {
        case "button":
            let button = UIButton()
            button.setTitle(component.title, for: .normal)
            // Добавить действие из component.action
        case "text":
            let label = UILabel()
            label.text = component.title
        case "banner":
            let imageView = UIImageView()
            // Загрузка imageUrl
        default:
            break
        }
    }
}
```

> ⚠️ В production рекомендуется использовать **динамические контейнеры, стеки и layout engine**, чтобы рендеринг был гибким и масштабируемым.

## 6. Best Practices

1. **Кэширование конфигураций**
    
    - Позволяет работать offline и ускоряет рендеринг.
        
2. **Валидация данных**
    
    - Проверять JSON на стороне клиента перед рендерингом.
        
3. **Модульная архитектура**
    
    - Каждый компонент рендерится отдельно и может переиспользоваться.
        
4. **Версионирование**
    
    - Поддерживать backward compatibility: старые клиенты должны рендерить старые компоненты корректно.
        
5. **Интеграция с A/B тестированием**
    
    - Легко менять порядок компонентов или показывать новые баннеры без релиза.
        

---

## 7. Итог

- **Server-driven UI** позволяет быстро и динамично изменять интерфейсы iOS-приложений без новых релизов.
    
- Подходит для приложений с часто меняющимся контентом (e-commerce, новостные сервисы, маркетинг).
    
- Основные аспекты: **конфигурация сервером**, **динамический рендеринг на устройстве**, **кэширование и безопасность**.
    
- Комбинация **SDUI + feature flags + caching** позволяет создавать гибкие и масштабируемые приложения будущего.
