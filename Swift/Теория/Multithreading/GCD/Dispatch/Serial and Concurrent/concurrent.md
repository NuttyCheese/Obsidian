### 1. Что такое конкурентная очередь в [[GCD]] (реальность 2026 года)

**Конкурентная очередь** (concurrent queue) — это очередь, в которой задачи **могут выполняться параллельно** (одновременно в нескольких потоках).

Основные характеристики:

- **Параллелизм** — да (несколько задач одновременно)  
- **Порядок выполнения** — **не гарантируется** (может быть любой)  
- **Потоки** — задачи берутся из **глобального пула потоков** (4–64+ потоков в зависимости от устройства)  
- **Барьерные задачи** (`.barrier`) — позволяют безопасно выполнять запись, пока другие задачи читают  
- **[[QoS]]** — наследуется от вызывающего контекста или задаётся явно

**Важнейший факт 2026 года**:
> Apple считает создание собственных конкурентных очередей с помощью  
> `DispatchQueue(label:attributes: .concurrent)` **почти устаревшим** (legacy) подходом.  
> В большинстве случаев лучше использовать:
> - `DispatchQueue.global(qos:)`  
> - [[TaskGroup]] / [[async let]] ([[Swift Concurrency]])  
> - [[actor]]  
> - [[OperationQueue]] (если нужен сложный контроль зависимостей)

### 2. Как создаются конкурентные очереди (и почему это почти не нужно)

```swift
// Старый/legacy способ (почти не рекомендуется в 2026)
let concurrentQueue = DispatchQueue(
    label: "com.example.concurrentQueue",
    attributes: .concurrent
)

// Более современный и рекомендуемый способ — глобальная очередь
let globalConcurrent = DispatchQueue.global(qos: .utility)
```

**Почему `concurrent(label:)` почти не используют в 2026**:

- Нет реального преимущества перед `DispatchQueue.global(qos:)`  
- Глобальные очереди уже оптимизированы Apple (баланс потоков, энергопотребление, QoS)  
- Собственные конкурентные очереди **не дают изоляции** — задачи всё равно берутся из общего пула  
- Для защиты mutable состояния лучше использовать **actor**  
- Для параллельных асинхронных задач — **TaskGroup** / **async let**

### 3. Когда конкурентные очереди всё ещё оправданы в 2026

Очень редкие, но реальные случаи:

1. Нужен **собственный барьер** для конкретной коллекции/ресурса  
2. Требуется **чётко отделить очередь** от глобальных пулов (логирование, аналитика, специфическая изоляция)  
3. Совместимость с очень старым кодом (iOS 12–13 поддержка)  
4. Специфические профили производительности (крайне редко)

Пример (единственный оправданный сценарий в 2026):

```swift
let concurrentProcessingQueue = DispatchQueue(
    label: "com.myapp.image-processing",
    attributes: .concurrent
)

var processedImages: [String: UIImage] = [:]

func processImage(_ image: UIImage, key: String) {
    concurrentProcessingQueue.async(flags: .barrier) {
        // Тяжёлая обработка (фильтры, ресайз, ML)
        let processed = applyHeavyFilter(image)
        
        concurrentProcessingQueue.async {
            processedImages[key] = processed  // безопасно
        }
    }
}
```

Но даже здесь в 2026 чаще пишут через `actor`:

```swift
actor ImageProcessor {
    private var processed: [String: UIImage] = [:]
    
    func process(_ image: UIImage, key: String) async {
        let processedImage = await applyHeavyFilterAsync(image)
        processed[key] = processedImage
    }
}
```

### 4. Типичные ошибки при использовании конкурентных очередей

| Ошибка                                | Последствия                                                | Как избежать в 2026                     |
| ------------------------------------- | ---------------------------------------------------------- | --------------------------------------- |
| Забыть `.barrier` при записи          | [[Data Race]] / повреждение коллекции                      | Использовать actor вместо concurrent    |
| Выполнять долгую работу без барьера   | Гонка данных                                               | actor или serial queue                  |
| Использовать `sync` внутри concurrent | [[Deadlock]] (если sync на той же очереди)                 | Только async + barrier                  |
| Перегружать очередь тысячами задач    | [[Thread Explosion]] / [[Priority Starvation\|starvation]] | Ограничивать concurrency (TaskGroup)    |
| Использовать для UI-обновлений        | [[Main Thread Violation]]                                  | Только @MainActor / Dispatch.main.async |

### 5. Конкурентные очереди vs современные альтернативы (2026 сравнение)

| Механизм                  | Параллелизм | Простота | Потокобезопасность | Рекомендация 2026 | Когда всё ещё использовать |
|---------------------------|-------------|----------|---------------------|-------------------|-----------------------------|
| DispatchQueue.concurrent  | Пул потоков | ★★★☆☆    | Требует barrier     | Почти никогда     | Legacy, специфический барьер |
| actor                     | 1 поток     | ★★★★★    | Встроенная          | Основной выбор    | Изменяемое состояние        |
| TaskGroup                 | Контролируемый | ★★★★★ | Встроенная          | Параллельные задачи | Асинхронные загрузки        |
| async let                 | Параллельный | ★★★★★    | Встроенная          | 2–5 задач           | Простые параллельные операции |
| OperationQueue            | Контролируемый | ★★★★☆   | Встроенная          | Legacy + сложные зависимости | Старые проекты              |

### 6. Лучшие практики 2026 года

- **Не создавайте** новые конкурентные очереди с `concurrent(label:)` — это почти всегда ошибка  
- Если нужен **барьер** — используйте `actor` или `DispatchQueue.global(flags: .barrier)`  
- Для параллельных задач — **TaskGroup** с `addTask` и ограничением concurrency  
- Для UI — **@MainActor** или `MainActor.run`  
- Для тестов — используйте `TaskGroup` + `XCTestExpectation`  
- В Swift 6 — strict concurrency checking часто подсвечивает небезопасное использование очередей  
- Добавляйте **os_signpost** для мониторинга длительности задач в Instruments

**Короткий девиз 2026**:
> «Собственные конкурентные очереди в 2026 году — это почти антипаттерн.  
> Параллелизм = TaskGroup + actor + @MainActor + async let.  
> Если ты всё ещё пишешь DispatchQueue.concurrent(label:) — спроси себя: «А точно ли это нужно?»»
