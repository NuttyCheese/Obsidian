### 1. Что такое DispatchSemaphore и зачем он нужен

**DispatchSemaphore** — это **классический семафор** из [[GCD]], который позволяет:

- контролировать доступ к **общему ресурсу** (файл, база данных, кэш, пул соединений и т.д.)  
- ограничивать **количество одновременно работающих задач** (например, не больше 5 сетевых запросов одновременно)  
- синхронизировать потоки, когда одна задача должна **дождаться** завершения другой (или нескольких)

**Ключевые характеристики**:

- Имеет **внутренний счётчик** (value)  
- `wait()` → уменьшает счётчик на 1 (если =0 → блокирует поток)  
- `signal()` → увеличивает счётчик на 1 (разблокирует один ждущий поток, если есть)  
- Работает **только с потоками GCD** (не с `Thread`, не с `Task` напрямую)  
- **Не отменяем** (в отличие от `DispatchWorkItem` или `Task.cancel()`)  
- **Не потокобезопасен сам по себе** — нужно правильно использовать

**Важнейший факт 2026 года**:
> `DispatchSemaphore` — это **legacy-инструмент**.  
> Apple **очень сильно рекомендует** заменять его на:
> - `actor`  
> - `TaskGroup` с ограничением concurrency  
> - `DispatchSemaphore` остаётся актуальным только для:
>   - поддержки очень старого кода  
>   - интеграции с C-библиотеками  
>   - специфических низкоуровневых сценариев (например, пул потоков C-[[API]])

### 2. Основные методы DispatchSemaphore

| Метод / Свойство          | Что делает                                                                 | Блокирует поток? | Самый частый use-case в 2026 |
|---------------------------|-----------------------------------------------------------------------------|-------------------|-------------------------------|
| `init(value:)`            | Создаёт семафор с начальным значением счётчика                              | —                 | Ограничение concurrency (value = 4–8) |
| `wait()`                  | Уменьшает счётчик на 1. Если ≤0 → блокирует поток                           | Да                | Блокировка доступа к ресурсу  |
| `wait(timeout:)`          | То же + таймаут (`.now() + 5`, `.distantFuture`)                           | Да (с таймаутом)  | Защита от зависания в тестах  |
| `signal()`                | Увеличивает счётчик на 1. Если есть ждущие — разблокирует одного            | Нет               | Освобождение ресурса          |
| `value` (только чтение)   | Текущее значение счётчика (не рекомендуется полагаться в production)        | Нет               | Отладка / логирование         |

### 3. Самые популярные шаблоны использования в 2026 году

#### Шаблон 1 — Ограничение количества параллельных сетевых запросов (самый частый)

```swift
let maxConcurrentRequests = 5
let semaphore = DispatchSemaphore(value: maxConcurrentRequests)

func fetchData(for url: URL, completion: @escaping (Data?) -> Void) {
    semaphore.wait()  // ждём, пока не освободится слот
    
    URLSession.shared.dataTask(with: url) { data, _, _ in
        completion(data)
        semaphore.signal()  // освобождаем слот
    }.resume()
}

// Запуск 20 запросов — одновременно будет не больше 5
for url in 20_urls {
    fetchData(for: url) { data in
        // обработка
    }
}
```

#### Шаблон 2 — Современный эквивалент ([[Swift Concurrency]] 2026)

```swift
actor RequestLimiter {
    private let maxConcurrent = 5
    private var active = 0
    
    func acquire() async {
        while active >= maxConcurrent {
            try? await Task.sleep(nanoseconds: 50_000_000) // backoff
        }
        active += 1
    }
    
    func release() {
        active -= 1
    }
}

let limiter = RequestLimiter()

func fetchData(for url: URL) async throws -> Data {
    await limiter.acquire()
    defer { Task { await limiter.release() } }
    
    let (data, _) = try await URLSession.shared.data(from: url)
    return data
}
```

#### Шаблон 3 — Ожидание завершения нескольких задач (альтернатива DispatchGroup)

```swift
let semaphore = DispatchSemaphore(value: 0)

for i in 0..<5 {
    DispatchQueue.global().async {
        // тяжёлая работа
        print("Task \(i) finished")
        semaphore.signal()
    }
}

// Ждём все 5 задач
for _ in 0..<5 {
    semaphore.wait()
}

print("Все задачи завершены")
```

**Лучше в 2026** — через `TaskGroup`:

```swift
await withTaskGroup(of: Void.self) { group in
    for i in 0..<5 {
        group.addTask {
            // тяжёлая работа
            print("Task \(i) finished")
        }
    }
    await group.waitForAll()
}
```

### 4. Типичные ошибки и ловушки с DispatchSemaphore в 2026

| Ошибка                                  | Последствия                          | Как избежать                                  |
| --------------------------------------- | ------------------------------------ | --------------------------------------------- |
| Забыть `signal()` после `wait()`        | [[Deadlock]] (все ждут вечно)        | Всегда парно: wait → signal (лучше [[defer]]) |
| `wait()` на главном потоке              | UI freeze / ANR                      | Никогда не использовать `wait()` на [[main]]  |
| Долгое удержание семафора               | [[Priority Starvation]] / задержки   | Минимизировать критическую секцию             |
| Использовать семафор вместо actor       | Сложный код, ошибки                  | actor почти всегда лучше                      |
| `wait(timeout:)` без обработки таймаута | Зависание или некорректное поведение | Всегда проверять `.timedOut`                  |
| Передача mutable данных без защиты      | [[Data Race]]                        | Всё [[mutable]] состояние — через actor       |

### 5. DispatchSemaphore vs современные альтернативы (2026 сравнение)

| Механизм                    | Ограничение concurrency | Отмена задач | Потокобезопасность | Рекомендация 2026    | Когда всё ещё использовать |
| --------------------------- | ----------------------- | ------------ | ------------------ | -------------------- | -------------------------- |
| DispatchSemaphore           | Да                      | Нет          | Ручная             | Legacy               | Поддержка старого кода     |
| [[actor]]                   | Да (1 поток)            | Да           | Встроенная         | Основной выбор       | Изменяемое состояние       |
| [TaskGroup] + limited tasks | Да (addTask лимит)      | Да           | Встроенная         | Параллельные задачи  | Асинхронные загрузки       |
| [[async let]] (2–5 задач)   | Да                      | Да           | Встроенная         | Простые сценарии     | Независимые задачи         |
| [[OperationQueue]]          | Да (maxConcurrent)      | Да           | Встроенная         | Legacy + зависимости | Старые проекты             |

**Вывод 2026**:
- **Новый код** → **actor** (простота + безопасность) или **TaskGroup** (параллелизм с лимитом)  
- **DispatchSemaphore** — только для поддержки legacy-кода или очень специфических низкоуровневых сценариев

### 6. Лучшие практики 2026 года

- **Переходите на Swift Concurrency** — DispatchSemaphore уже считается legacy  
- **Не используйте** `wait()` в production — только в тестах с таймаутом  
- **defer { signal() }** — спасает от забытого `signal()`  
- **Для UI** — только `@MainActor` / `MainActor.run`  
- **Для тестов** — используйте `XCTestExpectation` или `TaskGroup`  
- **Мониторинг** — добавляйте `os_signpost` для отслеживания длительности ожидания в Instruments  
- **Swift 6 strict concurrency** — часто подсвечивает небезопасное использование семафоров с mutable данными

**Короткий девиз 2026**:
> «DispatchSemaphore — это когда нужно ограничить количество параллельных задач или синхронизировать потоки.  
> В 2026 году это уже legacy-инструмент.  
> Новый стандарт — actor + TaskGroup + Sendable + strict concurrency.  
> Если ты всё ещё пишешь semaphore.wait() в новом коде — спроси себя: «А точно ли это нужно?»»
