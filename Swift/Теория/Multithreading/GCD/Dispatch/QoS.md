### 1. Что такое QoS и зачем он нужен

**QoS** (Quality of Service) — это **приоритет выполнения задачи** (или потока), который сообщает системе [[iOS]]/macOS:

- насколько **критична** эта задача для пользователя  
- сколько **ресурсов CPU, GPU, энергии** ей выделять  
- может ли она **вытеснять** другие задачи  
- насколько быстро её нужно выполнить

QoS влияет на:

- скорость получения процессорного времени  
- энергопотребление (особенно важно для iPhone/iPad)  
- отзывчивость UI (анимации, скролл, реакция на касание)  
- вероятность выполнения низкоприоритетных задач (если CPU перегружен)

**Ключевой факт 2026**:
> QoS — это **не жёсткий приоритет CPU**, а **подсказка** планировщику задач.  
> Система может игнорировать или корректировать QoS в зависимости от состояния устройства (низкий заряд, перегрев, режим энергосбережения и т.д.).

### 2. Все уровни QoS в 2026 году (с реальными рекомендациями Apple)

| QoS                     | Когда использовать (рекомендация Apple 2026)                          | Примеры задач в iOS-приложении 2026                          | Относительная скорость | Влияние на батарею | Приоритет в планировщике | Риск starvation низких задач |
|-------------------------|------------------------------------------------------------------------|---------------------------------------------------------------|--------------------------|---------------------|---------------------------|-------------------------------|
| `.userInteractive`      | Анимации, реакция на касание, скролл, рендеринг кадров                | Обновление UI во время жеста, анимации, Metal rendering       | Самый высокий            | Высокое             | Максимальный              | Очень высокий                 |
| `.userInitiated`        | Задачи, инициированные пользователем, но не критичные для отзывчивости | Загрузка данных для текущего экрана, отправка формы           | Очень высокий            | Высокое             | Очень высокий             | Высокий                       |
| `.default`              | Задачи без явного приоритета (по умолчанию)                           | Большинство фоновых операций без явного требования            | Средний                  | Среднее             | Средний                   | Средний                       |
| `.utility`              | Долгие, некритичные задачи                                            | Обработка фото, аналитика, индексация, сжатие видео           | Средний-низкий           | Низкое              | Средний-низкий            | Низкий                        |
| `.background`           | Очень низкий приоритет, задачи, которые могут ждать                   | Резервное копирование, очистка кэша, отправка статистики     | Самый низкий             | Минимальное         | Низкий                    | Очень низкий                  |

**Важное правило 2026**:
> Никогда не используйте `.userInteractive` или `.userInitiated` для задач дольше 1–2 секунд — это приводит к **Priority Starvation** фоновых задач и быстрому разряду батареи.

### 3. Как правильно выбирать QoS в 2026 году

| Тип задачи                              | Рекомендуемый QoS 2026                  | Почему именно этот |
|-----------------------------------------|------------------------------------------|--------------------|
| Анимации, скролл, реакция на касание    | `.userInteractive`                       | Максимальная отзывчивость UI |
| Загрузка данных для текущего экрана     | `.userInitiated`                         | Пользователь ждёт результат |
| Обработка фото, аналитика, индексация   | `.utility`                               | Долгая задача, но не критичная |
| Резервное копирование, очистка кэша    | `.background`                            | Может ждать, минимальное влияние на батарею |
| Отправка логов / статистики             | `.background`                            | Самый низкий приоритет |
| Большинство фоновых операций без явной важности | `.default`                         | Сбалансированный вариант |

### 4. Примеры правильного использования QoS в 2026 году

#### Пример 1 — Загрузка данных для экрана (userInitiated)

```swift
func loadFeed() {
    DispatchQueue.global(qos: .userInitiated).async {
        let posts = fetchPostsFromNetwork()
        
        DispatchQueue.main.async {
            self.posts = posts
            self.tableView.reloadData()
        }
    }
}
```

#### Пример 2 — Фоновая аналитика (utility)

```swift
func trackScreenView(screen: String) {
    DispatchQueue.global(qos: .utility).async {
        let event = AnalyticsEvent(screenView: screen, timestamp: Date())
        sendAnalytics(event)
    }
}
```

#### Пример 3 — Современный эквивалент в Swift Concurrency

```swift
Task(priority: .userInitiated) {
    let posts = await fetchPosts()
    await MainActor.run {
        self.posts = posts
        tableView.reloadData()
    }
}

Task(priority: .background) {
    await trackAnalyticsEvent("screen_view")
}
```

### 5. Типичные ошибки и ловушки с QoS в 2026

| Ошибка                                             | Последствия                                               | Как избежать                                       |
| -------------------------------------------------- | --------------------------------------------------------- | -------------------------------------------------- |
| Все задачи с `.userInitiated` / `.userInteractive` | [[Priority Starvation]] фоновых задач                     | Использовать правильные QoS                        |
| Забыть указать QoS                                 | По умолчанию `.default` — может быть слишком низко/высоко | Всегда явно задавать                               |
| `sync` на глобальной очереди из главного потока    | Deadlock / сильное замедление                             | Никогда не использовать `sync` на global из main   |
| Обновление UI без `.main.async`                    | [[Main Thread Violation]]                                 | Всегда `@MainActor` или `DispatchQueue.main.async` |
| Перегруженная очередь `.userInitiated`             | Фоновые задачи почти не запускаются                       | Ограничивать concurrency ([[TaskGroup]])           |

### 6. QoS в DispatchQueue vs Swift Concurrency (честное сравнение 2026)

| Характеристика         | DispatchQueue.global(qos:)                                     | [[Swift Concurrency]] (Task(priority:))  | Что выбрать в 2026       |
| ---------------------- | -------------------------------------------------------------- | ---------------------------------------- | ------------------------ |
| Потокобезопасность     | Нужно вручную ([[NSLock\|lock]], [[DispatchBarrier\|barrier]]) | Встроенная ([[actor]], [[Sendable]])     | Swift Concurrency        |
| Приоритеты             | QoS                                                            | Task priority                            | Оба хороши, Task проще   |
| Отмена задач           | [[DispatchWorkItem]] (сложно)                                  | Нативная ([[Task]].cancel())             | Swift Concurrency        |
| UI-обновления          | [[DispatchQueue]].[[main]].[[async]]                           | [[@MainActor]] / [[await]] MainActor.run | @MainActor лучше         |
| Читаемость             | Старый стиль                                                   | Современный, меньше кода                 | Swift Concurrency        |
| Совместимость с legacy | Отличная                                                       | Требует адаптации                        | [[GCD]] для старого кода |

### 7. Лучшие практики 2026 года

- **Переходите на Swift Concurrency** — DispatchQueue.global уже считается legacy  
- **QoS / Task priority** — всегда указывайте явно  
- **Не используйте** `sync` в production — почти всегда это ошибка  
- **Для UI** — только `@MainActor` / `MainActor.run` / `DispatchQueue.main.async`  
- **Для тестов** — используйте `XCTestExpectation` + `queue.async` или `Task`  
- **Мониторинг** — добавляйте `os_signpost` для отслеживания длительности задач в Instruments  
- **Swift 6 strict concurrency** — часто подсвечивает небезопасное использование глобальных очередей

**Короткий девиз 2026**:
> «QoS — это когда ты говоришь системе: «Эта задача важна (или не очень)».  
> В 2026 году это уже legacy-механизм в GCD.  
> Новый стандарт — Task(priority:) + actor + @MainActor.  
> Используй GCD только для поддержки старого кода.»
