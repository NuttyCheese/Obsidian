### 1. Что такое Dispatch Barrier и зачем он нужен

**Dispatch Barrier** — это специальный флаг (`.barrier`), который можно передать в метод `async(flags:execute:)` конкурентной очереди.

Когда барьерная задача добавлена в очередь, она создаёт **«барьер»**:

- **Все обычные задачи**, добавленные **до барьера**, выполняются **до него** (параллельно, как обычно).  
- **Барьерная задача** ждёт, пока **все предыдущие задачи завершатся**.  
- **Барьерная задача** выполняется **одна** (эксклюзивно, без параллелизма).  
- **Все задачи**, добавленные **после барьера**, ждут, пока барьерная задача **завершится**.

**Главный смысл барьера** в 2026 году:

- **Безопасная запись** в общую коллекцию / кэш / ресурс при параллельном чтении  
- Гарантированное выполнение критической секции **один раз** без race condition  
- Альтернатива `NSLock` / `os_unfair_lock` в конкурентных сценариях

**Важнейший факт 2026**:
> Барьер **работает только в конкурентных очередях** (`.concurrent` или `global`).  
> В **сериальной очереди** барьер **не имеет смысла** — там и так всё последовательно.

### 2. Когда барьер действительно нужен (реальные сценарии 2026)

| Сценарий                                             | Почему нужен барьер                                 | Альтернатива в 2026 (рекомендуемая) |
| ---------------------------------------------------- | --------------------------------------------------- | ----------------------------------- |
| Параллельное чтение + редкая запись в словарь/массив | Чтение может идти параллельно, запись — эксклюзивно | **actor** (основной выбор)          |
| Кэширование изображений / данных                     | Много потоков читают, один пишет обновлённые данные | actor или `@MainActor`              |
| Логирование / аналитика в фоне                       | Много потоков пишут логи, нужно сохранять порядок   | actor или серийная очередь          |
| Обновление общего состояния в legacy-коде            | Нет возможности переписать на actor                 | барьер (временное решение)          |
| Миграция старого [[GCD]]-кода на Swift Concurrency   | Нужно сохранить поведение барьера                   | actor + `withCheckedContinuation`   |

### 3. Классический и правильный пример использования барьера

```swift
// Конкурентная очередь (самый частый способ в 2026)
let cacheQueue = DispatchQueue.global(qos: .utility)

// Общий кэш (доступ из разных потоков)
var imageCache: [URL: UIImage] = [:]

// Чтение (параллельно)
func getCachedImage(for url: URL) -> UIImage? {
    cacheQueue.sync {  // безопасное чтение
        imageCache[url]
    }
}

// Запись (эксклюзивно через барьер)
func cacheImage(_ image: UIImage, for url: URL) {
    cacheQueue.async(flags: .barrier) {
        imageCache[url] = image
        print("Изображение закэшировано для \(url)")
    }
}
```

**Поведение**:
- Много потоков могут одновременно вызывать `getCachedImage` — они выполняются параллельно  
- Только **один** поток может выполнять `cacheImage` в любой момент времени  
- Все чтения, начатые до барьера, завершатся **до** записи  
- Все чтения, начатые после барьера, начнутся **после** записи

### 4. Самые популярные шаблоны 2026 года

#### Шаблон 1 — Барьер для безопасной записи в словарь

```swift
let queue = DispatchQueue.global(qos: .utility)
var sharedData: [String: String] = [:]

func updateValue(_ value: String, for key: String) {
    queue.async(flags: .barrier) {
        sharedData[key] = value
    }
}

func readValue(for key: String) -> String? {
    queue.sync { sharedData[key] }
}
```

#### Шаблон 2 — Барьер + уведомление о завершении

```swift
let queue = DispatchQueue.global(qos: .utility)
let group = DispatchGroup()

for url in urls {
    group.enter()
    queue.async {
        let image = downloadImage(url)
        queue.async(flags: .barrier) {
            cacheImage(image, url: url)
            group.leave()
        }
    }
}

group.notify(queue: .main) {
    // Все изображения загружены и закэшированы
    collectionView.reloadData()
}
```

### 5. Типичные ошибки и ловушки с барьером в 2026

| Ошибка                                       | Последствия                             | Как избежать                             |
| -------------------------------------------- | --------------------------------------- | ---------------------------------------- |
| Использовать барьер в **сериальной** очереди | Барьер игнорируется, нет эффекта        | Барьер только в `.concurrent` / `global` |
| Забыть `.barrier` при записи                 | [[Race condition]] / повреждение данных | Всегда явно указывать `flags: .barrier`  |
| Долгая работа внутри барьера                 | Priority Starvation / UI freeze         | Минимизировать критическую секцию        |
| Много барьеров подряд                        | Всё равно последовательное выполнение   | Использовать actor вместо барьеров       |
| Использовать `sync` внутри барьера           | [[Deadlock]]                            | Только `async` + barrier                 |

### 6. Dispatch Barrier vs современные альтернативы (2026 сравнение)

| Механизм                  | Защита от race condition | Сложность | Производительность | Рекомендация 2026 | Когда всё ещё использовать |
|---------------------------|---------------------------|-----------|---------------------|-------------------|-----------------------------|
| Dispatch Barrier          | Да (для записи)           | ★★★★☆     | ★★★★☆               | Legacy            | Поддержка старого кода      |
| actor                     | Да (полная)               | ★★★★★     | ★★★★☆               | Основной выбор    | Изменяемое состояние        |
| TaskGroup + Sendable      | Да (через Sendable)       | ★★★★★     | ★★★★★               | Параллельные задачи | Асинхронные загрузки        |
| os_unfair_lock            | Да                        | ★★★★☆     | ★★★★★               | Высокопроизводительные места | Редко, только примитивы     |
| @MainActor                | Да (для UI)               | ★★★★☆     | ★★★★☆               | UI и ViewModel    | SwiftUI / UIKit             |

**Вывод 2026**:
- **Новый код** → **actor** (самый простой и безопасный способ)  
- **Поддержка старого кода** → барьер в глобальной очереди  
- `DispatchQueue.concurrent(label:)` — почти никогда не создавайте

### 7. Лучшие практики 2026 года

- **Переходите на Swift Concurrency** — барьер уже считается legacy  
- **Барьер** — используйте только для **записи** в конкурентной очереди  
- **Минимизируйте критическую секцию** — захватывайте барьер только на время изменения данных  
- **Для UI** — только `@MainActor` / `MainActor.run` / `DispatchQueue.main.async`  
- **Для тестов** — используйте `TaskGroup` + `XCTestExpectation`  
- **Мониторинг** — добавляйте `os_signpost` для отслеживания длительности барьеров в Instruments  
- **Swift 6 strict concurrency** — часто подсвечивает небезопасное использование барьеров с mutable данными

**Короткий девиз 2026**:
> «Dispatch Barrier — это когда нужно безопасно писать в коллекцию из многих потоков.  
> В 2026 году это уже legacy-инструмент.  
> Новый стандарт — [[actor]] + [[TaskGroup]] + [[Sendable]] + strict concurrency.  
> Если ты всё ещё пишешь async(flags: .barrier) в новом коде — спроси себя: «А точно ли это нужно?»»
