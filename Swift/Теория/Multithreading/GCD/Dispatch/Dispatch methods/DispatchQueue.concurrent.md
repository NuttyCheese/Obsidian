### 1. Что такое DispatchQueue.concurrent (2026 реальность)

`DispatchQueue.concurrent(label:)` — это **статический метод** типа [[DispatchQueue]], который создаёт **конкурентную** ([[concurrent]]) очередь.

**Ключевые характеристики** (актуально в 2026):

- очередь **конкурентная** (parallel) — может выполнять **несколько задач одновременно**  
- задачи выполняются в **глобальном пуле потоков** (не привязаны к одному потоку)  
- порядок выполнения задач **не гарантируется** (может быть любым)  
- внутри очереди можно использовать **barrier**-задачи для безопасной записи  
- QoS (приоритет) наследуется от вызывающего контекста или задаётся явно

**Важное изменение в Swift 5.5+ и особенно в 2026**:
> Apple **настоятельно рекомендует** использовать `DispatchQueue.global(qos:)` вместо создания собственных конкурентных очередей с помощью `concurrent(label:)`.  
> Собственные конкурентные очереди почти никогда не нужны в современном коде.

### 2. Сравнение типов очередей в 2026 году

| Тип очереди                      | Создание                           | Параллелизм | Порядок задач   | Лучший use-case в 2026 году                  | Рекомендуется Apple |
| -------------------------------- | ---------------------------------- | ----------- | --------------- | -------------------------------------------- | ------------------- |
| Сериальная ([[serial]])          | `DispatchQueue(label:)`            | 1 поток     | Строго [[FIFO]] | Защита mutable состояния, последовательность | Да                  |
| Конкурентная (concurrent, label) | `DispatchQueue.concurrent(label:)` | Пул потоков | Не гарантирован | Редко — только специфические случаи          | Почти нет           |
| Глобальная конкурентная          | `DispatchQueue.global(qos:)`       | Пул потоков | Не гарантирован | 99% всех фоновых задач                       | Да (основной выбор) |
| Главная (main)                   | `DispatchQueue.main`               | 1 поток     | Строго FIFO     | Всё, что касается UI                         | Да                  |

**Вывод 2026 года**:
> В 95–99% случаев используйте `DispatchQueue.global(qos:)`  
> `DispatchQueue.concurrent(label:)` — это уже **почти legacy**-конструкция.

### 3. Когда всё-таки используют concurrent(label:) в 2026

Очень редкие, но реальные случаи:

- Нужно **иметь собственный барьер** для конкретной коллекции/ресурса  
- Требуется **чётко отделить очередь** от глобальных пулов (логирование, аналитика, специфическая изоляция)  
- Совместимость с очень старым кодом ([[iOS]] 12–13 поддержка)  
- Специфические профили производительности (очень редко)

Пример (единственный оправданный сценарий):

```swift
let concurrentImagesQueue = DispatchQueue(
    label: "com.myapp.image-processing",
    attributes: .concurrent
)

var processedImages: [String: UIImage] = [:]

func processImage(_ image: UIImage, key: String) {
    concurrentImagesQueue.async(flags: .barrier) {
        // Тяжёлая обработка изображения (фильтры, ресайз)
        let processed = heavyFilter(image)
        
        concurrentImagesQueue.async {
            processedImages[key] = processed  // безопасно
        }
    }
}
```

Но даже здесь в 2026 чаще пишут через `actor`:

```swift
actor ImageProcessor {
    private var processed: [String: UIImage] = [:]
    
    func process(_ image: UIImage, key: String) async {
        let processedImage = await heavyFilterAsync(image)
        processed[key] = processedImage
    }
}
```

### 4. Типичные ошибки при использовании concurrent(label:)

| Ошибка                              | Последствия                                                | Как избежать в 2026                     |
| ----------------------------------- | ---------------------------------------------------------- | --------------------------------------- |
| Забыть `.barrier` при записи        | [[Data Race]] / повреждение коллекции                      | Использовать actor вместо concurrent    |
| Выполнять долгую работу без барьера | Гонка данных                                               | actor или serial queue                  |
| Использовать sync внутри concurrent | [[Deadlock]] (если sync на той же очереди)                 | Только async + barrier                  |
| Перегружать очередь тысячами задач  | [[Thread Explosion]] / [[Priority Starvation\|starvation]] | Ограничивать concurrency (TaskGroup)    |
| Использовать для UI-обновлений      | [[Main Thread Violation]]                                  | Только @MainActor / Dispatch.main.async |

### 5. DispatchQueue.concurrent vs современные альтернативы (2026)

| Механизм                 | Параллелизм    | Простота | Потокобезопасность | Рекомендация 2026            | Когда всё ещё использовать    |
| ------------------------ | -------------- | -------- | ------------------ | ---------------------------- | ----------------------------- |
| DispatchQueue.concurrent | Пул потоков    | ★★★☆☆    | Требует barrier    | Почти никогда                | Legacy, специфический барьер  |
| [[actor]]                | 1 поток        | ★★★★★    | Встроенная         | Основной выбор               | Изменяемое состояние          |
| [[TaskGroup]]            | Контролируемый | ★★★★★    | Встроенная         | Параллельные задачи          | Асинхронные загрузки          |
| async let                | Параллельный   | ★★★★★    | Встроенная         | 2–5 задач                    | Простые параллельные операции |
| [[OperationQueue]]       | Контролируемый | ★★★★☆    | Встроенная         | Legacy + сложные зависимости | Старые проекты                |

**Вывод**:  
В 2026 году `DispatchQueue.concurrent(label:)` — это **почти антипаттерн**.  
Apple и сообщество рекомендуют:

- **actor** — для защиты состояния  
- **TaskGroup / async let** — для параллельных асинхронных задач  
- **DispatchQueue.global(qos:)** — если всё ещё нужен [[GCD]]  
- `@MainActor` — для UI

### 6. Лучшие практики 2026 года

- **Не создавайте** новые конкурентные очереди с `concurrent(label:)` — это почти всегда ошибка  
- Если нужен **барьер** — используйте `actor` или `DispatchQueue.global(flags: .barrier)`  
- Для параллельных задач — **TaskGroup** с `addTask` и ограничением concurrency  
- Для UI — **@MainActor** или `MainActor.run`  
- Для тестов — используйте `TaskGroup` + `XCTestExpectation`  
- В Swift 6 — strict concurrency checking часто подсвечивает небезопасное использование очередей  
- Добавляйте **os_signpost** для мониторинга длительности задач в Instruments

**Короткий девиз 2026**:
> «DispatchQueue.concurrent — это вчерашний день.  
> В 2026 году конкурентность = TaskGroup + actor + @MainActor + async let.  
> Если ты всё ещё пишешь `concurrent(label:)`, спроси себя: «А точно ли это нужно?»»
