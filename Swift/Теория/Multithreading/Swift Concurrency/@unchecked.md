### 1. Что такое @unchecked [[Sendable]]

**`@unchecked Sendable`** — это **способ принудительно пометить тип как соответствующий протоколу `Sendable`**, даже если компилятор **не может** самостоятельно это доказать.

Коротко:  
«Я, программист, беру на себя ответственность и гарантирую, что этот тип **безопасен для передачи между потоками / задачами / актёрами**.  
Не проверяй его, просто поверь мне».

Без `@unchecked` компилятор в строгом режиме ([[Swift]] 6 complete concurrency checking) **отказывается** считать тип `Sendable`, если:

- это класс ([[reference type]])  
- у него есть изменяемые свойства (var)  
- нет внутренней синхронизации (lock, [[actor]], serial queue и т.д.)  
- свойства не являются сами по себе `Sendable`

С `@unchecked Sendable` компилятор **перестаёт ругаться** и разрешает передавать тип между задачами.

**Но**:
- это **отключает автоматическую проверку**  
- если ты соврал → **гонки данных, краши, undefined behavior** в runtime  
- ответственность **полностью на тебе**

### 2. Когда @unchecked Sendable действительно оправдан (2026)

| Ситуация                                        | Почему именно @unchecked Sendable            | Альтернатива (лучше)                 |
| ----------------------------------------------- | -------------------------------------------- | ------------------------------------ |
| Обёртка над legacy [[Objective-C]] / C классом  | Класс не поддаётся анализу [[Sendable]]      | Переписать на actor / [[value type]] |
| Класс с внутренней синхронизацией (lock, queue) | Компилятор не видит, что внутри безопасно    | actor (рекомендуется)                |
| final-класс без изменяемых свойств              | Компилятор всё равно требует Sendable        | Просто `final class: Sendable`       |
| Кэш / менеджер, доступный только через actor    | Фактически безопасен, но компилятор не знает | actor + инкапсуляция                 |
| Миграционный период (временно)                  | Нужно быстро убрать ошибки компиляции        | Планировать полную миграцию          |

**Золотое правило 2026**:
> `@unchecked Sendable` — это **последний рубеж**.  
> Используй его только тогда, когда:
> - ты **точно знаешь**, как обеспечивается безопасность  
> - и **нет возможности** переписать на actor / value type / Sendable

### 3. Самые популярные и правильные шаблоны 2026 года

#### Шаблон 1 — Класс с внутренней синхронизацией (самый оправданный случай)

```swift
final class ThreadSafeCache<Key: Hashable, Value>: @unchecked Sendable {
    private let queue = DispatchQueue(label: "cache.serial", qos: .utility)
    private var storage: [Key: Value] = [:]
    
    func get(_ key: Key) -> Value? {
        queue.sync { storage[key] }
    }
    
    func set(_ value: Value, for key: Key) {
        queue.async(flags: .barrier) {
            self.storage[key] = value
        }
    }
}
```

Здесь `@unchecked Sendable` оправдан, потому что **вся мутация** идёт через сериальную очередь.

#### Шаблон 2 — Обёртка над старым [[API]]

```swift
final class LegacyAnalyticsTracker: @unchecked Sendable {
    private let tracker: OldObjCAnalytics // из Objective-C
    
    func trackEvent(_ name: String) {
        tracker.track(name) // внутренне синхронизировано в Obj-C
    }
}
```

#### Шаблон 3 — [[final]]-класс без мутации (почти Sendable)

```swift
final class ConstantConfig: @unchecked Sendable {
    let apiKey: String
    let endpoint: URL
    
    init(apiKey: String, endpoint: URL) {
        self.apiKey = apiKey
        self.endpoint = endpoint
    }
}
```

Здесь можно было бы просто `: Sendable`, но если компилятор по какой-то причине не выводит — `@unchecked` спасает.

### 4. Типичные ошибки и ловушки 2026 года

| Ошибка                                      | Последствия                              | Как избежать |
|---------------------------------------------|------------------------------------------|--------------|
| Поставить @unchecked Sendable на класс с открытыми var | Гонки данных в runtime                   | Всё mutable — через lock / actor / serial queue |
| Использовать @unchecked без внутренней синхронизации | Краши, повреждение данных                | Всегда обеспечивать безопасность вручную |
| Думать, что @unchecked делает класс потокобезопасным | Ложное чувство безопасности              | Это **только отключение проверки** — безопасность на тебе |
| Применять к новому коду без причины         | Код становится менее проверяемым         | Писать новый код concurrency-safe |
| Забыть final-класс                           | Подклассы могут сломать безопасность     | Всегда `final` при @unchecked Sendable |

### 5. @unchecked Sendable vs другие механизмы (2026 сравнение)

| Механизм                  | Проверка компилятором | Ответственность за безопасность | Сложность | Рекомендация 2026 | Когда использовать |
|---------------------------|-------------------------|----------------------------------|-----------|-------------------|---------------------|
| `@unchecked Sendable`     | Отключена               | Полностью на программисте        | Высокая   | Только как исключение | Legacy-обёртки, временно |
| `Sendable` (автоматически) | Полная                  | Компилятор + программист         | Низкая    | Основной выбор    | Структуры, enum, final-классы без var |
| `actor`                   | Полная                  | Компилятор полностью             | Низкая    | Самый безопасный  | Любое изменяемое состояние |
| `@MainActor`              | Полная                  | Компилятор полностью             | Низкая    | Для UI            | ViewModel, UI-логика |
| `nonisolated(unsafe)`     | Отключена               | Полностью на программисте        | Высокая   | Редко             | Низкоуровневые случаи |

**Вывод 2026**:
- `@unchecked Sendable` — это **последний рубеж**, когда компилятор не может помочь  
- В идеальном проекте 2026 года его использование должно быть **минимальным** (идеально — нулевым в новом коде)  
- Лучшая защита — **actor** + **value types** + **Sendable**-структуры

### 6. Лучшие практики 2026 года

- **Используйте @unchecked Sendable** только когда:
  - тип **действительно** потокобезопасен (lock, serial queue, actor-инкапсуляция)  
  - это **legacy-обёртка** над старым кодом  
  - нет возможности переписать на actor / value type
- **Всегда** делай класс `final` при @unchecked Sendable  
- **Документируй** — пиши комментарий «@unchecked Sendable — внутренне синхронизировано через X»  
- **Тестируй** — проверяй Thread Sanitizer и Instruments на гонки данных  
- **Swift 6 strict concurrency** — включай полную проверку — она покажет, где @unchecked можно убрать  
- **План миграции** — постепенно заменяй @unchecked на обычный `Sendable` или `actor`

**Короткий девиз 2026**:
> «@unchecked Sendable — это когда ты говоришь компилятору: «я сам отвечаю за безопасность, не проверяй».  
> В 2026 году это крайняя мера.  
> Лучше actor, value types и настоящий Sendable.»
