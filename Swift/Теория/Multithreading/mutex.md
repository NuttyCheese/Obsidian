### 1. Что такое мьютекс и зачем он нужен в 2026 году

**Мьютекс** (mutual exclusion — взаимоисключение) — это примитив синхронизации, который обеспечивает, что **только один поток** одновременно может выполнять критическую секцию кода.

Классическое определение остаётся актуальным и в 2026 году:
- поток захватывает мьютекс → входит в критическую секцию → работает → освобождает мьютекс  
- любой другой поток, пытающийся захватить мьютекс, **блокируется** (ждёт) до освобождения

**Зачем мьютекс всё ещё нужен в 2026** (даже при наличии actor):

| Сценарий                                      | Почему мьютекс всё ещё используют | Альтернатива (рекомендуемая) |
|-----------------------------------------------|------------------------------------|-------------------------------|
| Очень горячие критические секции (< 100 нс)   | Минимальные накладные расходы      | `os_unfair_lock` / `Atomic`   |
| Интеграция с низкоуровневыми C/C++ библиотеками | Прямое соответствие pthread_mutex  | `UnsafeMutablePointer` + lock |
| Legacy-код / поддержка iOS 12–14              | Совместимость                      | —                             |
| Максимальная производительность в lock-free структурах | CAS + мьютекс как fallback         | `swift-atomics` + `Atomic`    |
| Простые сценарии с редкими конфликтами        | Код проще и понятнее               | `actor` (в 90% случаев)       |

**Вывод 2026**:
> Мьютексы **не устарели**, но их роль сильно сузилась.  
> В новом коде их используют **только** там, где `actor` даёт слишком большие накладные расходы или нужна интеграция с C-API.

### 3. Самые актуальные и рекомендуемые паттерны 2026 года

#### Паттерн 1 — `os_unfair_lock` — современный стандарт для горячих секций

```swift
import os.lock

final class FastCounter {
    private var value = 0
    private var lock = os_unfair_lock()
    
    func increment() {
        os_unfair_lock_lock(&lock)
        value += 1
        os_unfair_lock_unlock(&lock)
    }
    
    func get() -> Int {
        os_unfair_lock_lock(&lock)
        defer { os_unfair_lock_unlock(&lock) }
        return value
    }
}

// 10 000 потоков увеличивают счётчик
let counter = FastCounter()

for _ in 0..<10_000 {
    Task.detached {
        for _ in 0..<1_000 {
            counter.increment()
        }
    }
}

// Итог всегда будет 10_000_000
print(counter.get())
```

**Почему `os_unfair_lock` побеждает в 2026**:

- быстрее `NSLock` в 3–10 раз  
- минимальные накладные расходы  
- оптимизирован под современные ARM-процессоры (Apple Silicon)  
- официально рекомендован Apple вместо `NSLock`

#### Паттерн 2 — actor как замена мьютексу в 90% случаев

```swift
actor Counter {
    private var value = 0
    
    func increment() {
        value += 1
    }
    
    func get() -> Int {
        value
    }
}

let counter = Counter()

Task {
    for _ in 0..<1_000_000 {
        await counter.increment()
    }
}

print(await counter.get())  // всегда 1_000_000
```

**Когда actor лучше мьютекса**:

- сложная логика внутри критической секции  
- много чтений и редкие записи  
- нужно безопасно работать с коллекциями / объектами  
- читаемость и поддержка кода важнее 5–10% производительности

#### Паттерн 3 — `Atomic` как замена мьютексу для примитивов

```swift
import Atomics

let counter = Atomic<Int>(0)

Task.detached {
    for _ in 0..<1_000_000 {
        counter.wrappingIncrement(by: 1, ordering: .relaxed)
    }
}

print(counter.load(ordering: .relaxed))  // всегда 1_000_000
```

**Когда `Atomic` лучше мьютекса**:

- только одна переменная (Int, Bool, UInt64, UnsafeRawPointer)  
- нужна максимальная производительность  
- операции простые (increment, decrement, exchange, CAS)

- 5–20 операций внутри критической секции → `os_unfair_lock`  
- сложная логика / коллекции → `actor`

### 5. Лучшие практики 2026 года

- **actor** — основной выбор для **90%** случаев с изменяемым состоянием  
- **`os_unfair_lock`** — для **горячих примитивных** критических секций  
- **`swift-atomics`** — для **счётчиков**, **флагов**, **lock-free** структур  
- **`NSLock`** — только в **legacy-коде** (очень медленно)  
- **Никогда** не используйте `NSRecursiveLock` — очень дорого  
- **Для UI** — только `@MainActor`  
- **Swift 6 strict concurrency** — включайте полную проверку — она ловит небезопасное использование мьютексов  
- **Тестирование** — Thread Sanitizer + Instruments + stress-тесты на 8–16 ядер  
- **Документируйте** — пишите комментарий «os_unfair_lock — критическая секция 50 нс»
