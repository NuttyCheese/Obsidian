#system_design
## Определение

**Rate limiting** — механизм ограничения числа запросов к серверу за определённый период времени.  
**Кэширование на сервере** — хранение результатов частых запросов для ускорения обработки и уменьшения нагрузки на backend.

Для [[iOS]]-приложений это критично, чтобы:

- Предотвратить перегрузку серверов.
    
- Защитить API от злоупотреблений (например, ботами).
    
- Ускорить ответы для часто запрашиваемых данных.
    

---

## 1. Rate limiting

### Основные цели

1. Защита от перегрузки сервера.
    
2. Предотвращение злоупотреблений [[API]].
    
3. Обеспечение равномерного распределения ресурсов для всех клиентов.
    

### Методы реализации

|Метод|Описание|
|---|---|
|**Fixed Window**|Ограничение числа запросов за фиксированный интервал (например, 100 запросов в минуту).|
|**Sliding Window**|Более точное ограничение, учитывает запросы в скользящем окне времени.|
|**Token Bucket**|Каждый клиент получает «токены», тратящиеся на запросы; токены пополняются со временем.|
|**Leaky Bucket**|Запросы обрабатываются с фиксированной скоростью; лишние запросы «вытекают» за пределы ограничения.|

### Пример в iOS-клиенте

- Отслеживать число запросов к API.
    
- При достижении лимита приостанавливать новые запросы и показывать пользователю сообщение.
    

```swift
if requestCount >= maxRequestsPerMinute {
    print("Превышен лимит запросов, подождите минуту")
} else {
    sendRequest()
}
```

## 2. Кэширование на сервере

### Основные цели

1. Ускорение отклика для часто запрашиваемых данных.
    
2. Снижение нагрузки на базу данных и микросервисы.
    
3. Повышение отказоустойчивости.
    

### Методы кэширования

| Метод                | Описание                                                                |
| -------------------- | ----------------------------------------------------------------------- |
| **In-memory cache**  | Быстрое кэширование в RAM (Redis, Memcached).                           |
| **[[HTTP]] caching** | Использование заголовков `ETag`, `Cache-Control`, `Expires`.            |
| **CDN caching**      | Кэширование на edge-серверах для географически распределённого доступа. |
| **Database caching** | Кэширование сложных запросов в базе данных.                             |

### Пример использования Redis для кэширования ответов

```python
# Python пример сервера
import redis
cache = redis.Redis(host='localhost', port=6379)

def get_user_profile(user_id):
    cached = cache.get(f"user:{user_id}")
    if cached:
        return cached
    profile = db.query_user(user_id)
    cache.set(f"user:{user_id}", profile, ex=300)  # кэш 5 минут
    return profile
```

## 3. Комбинированное использование

- **Rate limiting** защищает сервер от перегрузки и злоупотреблений.
    
- **Кэширование** снижает нагрузку и ускоряет ответы.
    
- Вместе они обеспечивают **стабильную и быструю работу API** для iOS-клиентов.
    

```text
iOS App → API Gateway → [Rate Limiter → Cache → Microservices] → API Gateway → iOS App
```

---

## 4. Best Practices

1. **Использовать распределённый кэш** для масштабируемых систем (Redis Cluster, Memcached).
    
2. **Комбинировать rate limiting и кэширование**:
    
    - Ограничивать частые запросы.
        
    - Отдавать кэшированные ответы, если данные не критичны.
        
3. **Настраивать TTL (Time To Live)**:
    
    - Кэшировать данные на разумный срок (например, 5–10 минут для часто обновляемых данных).
        
4. **Мониторинг и метрики**:
    
    - Отслеживать hit/miss ratio, rejected requests.
        
5. **Уведомления клиенту**:
    
    - Возвращать HTTP статус `429 Too Many Requests` при превышении лимита.
        

---

## 5. Итог

- **Rate limiting** → защита API от перегрузки и злоупотреблений.
    
- **Кэширование на сервере** → ускорение ответов и снижение нагрузки.
    
- В iOS-приложениях совместное использование этих механизмов обеспечивает **стабильное, быстрое и безопасное взаимодействие с сервером**.
    
- Популярные инструменты: **Redis, Memcached, Nginx, Cloudflare**, встроенные механизмы API Gateway.
