#system_design
## Определение

**Нагрузочное тестирование (Load Testing)** — это процесс проверки работы системы под **высокой нагрузкой**, чтобы оценить:

- Производительность серверного [[API]].
    
- Устойчивость мобильного приложения при большом числе одновременных запросов.
    
- Ограничения rate limiting, кэширования и масштабируемости сервера.
    

Для [[iOS]]-приложений важно понимать, как API ведёт себя под нагрузкой и как клиент обрабатывает ошибки и задержки.

---

## 1. Цели нагрузочного тестирования

1. **Определение максимальной нагрузки**, которую выдерживает сервер.
    
2. **Выявление узких мест** (bottlenecks) в API.
    
3. **Проверка поведения приложения** при высоком трафике: таймауты, повторные запросы, отображение ошибок.
    
4. **Подготовка к реальному росту пользователей** после релиза.
    

---

## 2. Методы имитации запросов

|Метод|Описание|Инструменты|
|---|---|---|
|**Статический поток**|Определённое число запросов в единицу времени|Postman, cURL, JMeter|
|**Распределённая нагрузка**|Имитация множества пользователей из разных источников|Locust, Gatling|
|**Spike-тестирование**|Внезапный всплеск нагрузки|JMeter, k6|
|**Soak-тестирование (долговременная нагрузка)**|Проверка стабильности сервера при постоянной нагрузке|k6, BlazeMeter|

---

## 3. Пример использования JMeter для API

1. Создаём **Thread Group** — имитация числа пользователей.
    
2. Добавляем **[[HTTP]] Request** с [[URL]] API.
    
3. Настраиваем **Loop Count / Ramp-Up**:
    
    - **Ramp-Up** — время, за которое все пользователи начнут делать запросы.
        
    - **Loop Count** — число повторов каждого запроса.
        
4. Добавляем **Listeners** для мониторинга:
    
    - Graph Results, Summary Report, Response Times.
        

```text
Thread Group:
- Number of Threads (Users): 100
- Ramp-Up Period: 10 seconds
- Loop Count: 50
```

## 4. Имитация нагрузки с iOS-клиента

- Можно использовать **[[URLSession]]** и **[[DispatchQueue]]** для параллельных запросов:
    

```swift
let url = URL(string: "https://api.example.com/data")!

for i in 0..<100 {
    DispatchQueue.global().async {
        let task = URLSession.shared.dataTask(with: url) { data, response, error in
            if let error = error {
                print("Ошибка запроса \(i): \(error)")
            } else {
                print("Успешный запрос \(i)")
            }
        }
        task.resume()
    }
}
```

> ⚠️ На практике рекомендуется тестировать нагрузку с отдельного сервера, а не с реального iOS-устройства, чтобы не создавать локальных ограничений и сетевых искажений.

---

## 5. Best Practices

1. **Использовать отдельную среду тестирования**
    
    - Не нагружать production.
        
2. **Мониторинг сервера**
    
    - CPU, RAM, latency, ошибки, rate limit.
        
3. **Сценарии тестирования**
    
    - Пиковая нагрузка, длительная нагрузка, «спайк»-тест.
        
4. **Обработка ошибок на клиенте**
    
    - Таймауты, повторные запросы, fallback.
        
5. **Интеграция с [[CI]]/[[CD]]**
    
    - Автоматическое нагрузочное тестирование после сборки API.
        

---

## 6. Итог

- **Нагрузочное тестирование** помогает выявлять узкие места серверного API до выхода приложения в продакшн.
    
- Для iOS важно тестировать **обработку ошибок, таймаутов и повторных запросов**.
    
- Популярные инструменты: **JMeter, Locust, Gatling, k6**, а для простых сценариев можно использовать **URLSession**.
    
- Совмещение нагрузочного тестирования с мониторингом сервера обеспечивает **устойчивость и стабильность приложения при росте пользователей**.
