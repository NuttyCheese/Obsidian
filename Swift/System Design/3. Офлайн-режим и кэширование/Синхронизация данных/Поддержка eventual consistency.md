#system_design
## Определение

**Eventual Consistency (событийная/отложенная консистентность)** — это стратегия синхронизации данных, при которой **данные на клиенте и сервере могут временно отличаться**, но через некоторое время гарантированно станут согласованными.

> В отличие от строгой консистентности, eventual consistency допускает временные расхождения, чтобы повысить **отказоустойчивость и масштабируемость**.

---

## Почему это важно для мобильных приложений

1. **Оффлайн-режим**
    
    - Пользователь может изменять данные без сети, а синхронизация произойдёт позже.
        
2. **Производительность**
    
    - Нет необходимости блокировать интерфейс или сеть до полной синхронизации.
        
3. **Масштабируемость**
    
    - Серверы могут обрабатывать множество клиентов, не блокируя операции.
        
4. **Отказоустойчивость**
    
    - Даже при сбое сети или сервера данные будут синхронизированы, когда соединение восстановится.
        

---

## Основные принципы

1. **Локальные изменения**
    
    - Все изменения сначала сохраняются локально (например, в [[Swift/Теория/Сторонние библиотеки и расширения/Realm]], [[Core Data]], [[SQLite]]).
        
2. **Отложенная синхронизация**
    
    - Изменения отправляются на сервер, когда есть сеть.
        
3. **Конфликтное разрешение**
    
    - Если два клиента изменили один и тот же ресурс, сервер или клиент решает конфликт.
        
    - Стратегии:
        
        - **Last Write Wins (LWW)** — сохраняется последнее изменение.
            
        - **Merge** — объединение изменений, если это возможно (например, списки).
            
        - **Custom rules** — правила на основе бизнес-логики.
            
4. **Повторные попытки (Retry)**
    
    - Если синхронизация не удалась (например, из-за сети), данные остаются в очереди и пытаются снова.
        

---

## Пример workflow для iOS

1. Пользователь создаёт новую запись (например, заказ).
    
2. Данные сохраняются **локально в Realm или Core Data**.
    
3. Отправляется запрос на сервер (через REST или gRPC).
    
4. Если нет сети → операция откладывается.
    
5. Когда сеть доступна → данные отправляются на сервер.
    
6. Сервер проверяет конфликт и применяет стратегию разрешения.
    
7. Клиент получает подтверждение и обновляет локальную копию.
    

---

## Архитектурные подходы

1. **Queue-based Sync**
    
    - Все локальные изменения помещаются в очередь и выполняются по порядку.
        
2. **Versioning / Timestamps**
    
    - Каждое изменение имеет метку времени или версию для разрешения конфликтов.
        
3. **Background Sync**
    
    - Использовать `BackgroundTasks` или [[URLSession]] с background configuration для синхронизации без блокировки UI.
        
4. **Conflict Resolution Strategy**
    
    - Определить правила для каждого типа данных заранее.

## Пример на Swift (с использованием Realm)

```swift
class Order: Object {
    @objc dynamic var id: String = UUID().uuidString
    @objc dynamic var status: String = "pending"
    @objc dynamic var updatedAt: Date = Date()
}

func saveOrderLocally(order: Order) {
    let realm = try! Realm()
    try! realm.write {
        realm.add(order)
    }
}

// Очередь для синхронизации
var syncQueue: [Order] = []

func enqueueOrderForSync(order: Order) {
    syncQueue.append(order)
}

func syncOrders() {
    for order in syncQueue {
        APIService.shared.sendOrder(order) { success in
            if success {
                syncQueue.removeAll { $0.id == order.id }
            }
        }
    }
}
```

## Best Practices

1. Всегда сохранять **локальные копии изменений**.
    
2. Использовать **timestamps и versioning** для разрешения конфликтов.
    
3. Минимизировать вероятность потери данных → **retry и очередь синхронизации**.
    
4. Информировать пользователя о статусе синхронизации (например, "данные синхронизированы", "ожидает сеть").
    
5. Для сложных данных применять **merge strategies**, а не просто LWW.
    

---

## Применение в мобильных приложениях

- **Чат-приложения** → сообщения отправляются оффлайн, синхронизируются при восстановлении сети.
    
- **Приложения доставки** → обновление статусов заказов, даже если сеть нестабильна.
    
- **Приложения заметок / списков** → пользователи могут редактировать локально, изменения синхронизируются позже.
    

---

## Итог

- **Eventual consistency** → подход для оффлайн-приложений с высокой масштабируемостью и отказоустойчивостью.
    
- Позволяет работать **без сети**, синхронизируя данные позже.
    
- Ключевые элементы: **локальное хранение, очередь синхронизации, конфликтное разрешение, повторные попытки**.
    
- Подходит для большинства современных мобильных приложений с **нестрогой консистентностью**.
