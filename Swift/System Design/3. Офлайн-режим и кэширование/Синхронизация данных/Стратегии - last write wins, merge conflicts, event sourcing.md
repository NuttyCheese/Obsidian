# Стратегии синхронизации данных

## Определение

Синхронизация данных между клиентом и сервером в мобильных приложениях может сталкиваться с конфликтами, особенно в оффлайн-режиме. **Стратегии синхронизации** определяют, как разрешать эти конфликты и поддерживать консистентность данных.

> Основные стратегии: **Last Write Wins (LWW)**, **Merge Conflicts**, **Event Sourcing**.

---

## 1. Last Write Wins (LWW)

### Описание

- В случае конфликта **выбирается последнее изменение** по метке времени или версии.
    
- Простая стратегия, но может потерять промежуточные данные.
    

### Применение

- Полезно, когда важен только **финальный результат**, а не история изменений.
    
- Часто используется в **чате, списках задач, статусах заказов**.
    

### Пример

```swift
struct Note {
    var id: String
    var content: String
    var updatedAt: Date
}

func resolveConflict(local: Note, remote: Note) -> Note {
    return local.updatedAt > remote.updatedAt ? local : remote
}
```

**Плюсы:**

- Простота реализации
    
- Минимальные вычислительные ресурсы
    

**Минусы:**

- Потеря данных при одновременных изменениях
    
- Не хранит историю изменений

## 2. Merge Conflicts (слияние изменений)

### Описание

- Различные изменения объединяются, вместо того чтобы перезаписать данные.
    
- Требует **правил слияния** для каждого типа данных.
    

### Типы слияния

|Тип данных|Стратегия слияния|
|---|---|
|Списки|объединение элементов, удаление дубликатов|
|Строки / текст|line-based merge, diff/patch|
|Структуры|комбинирование отдельных полей по приоритету|

### Пример на Swift (список задач)

```swift
var localTasks = ["task1", "task2"]
var remoteTasks = ["task2", "task3"]

let mergedTasks = Array(Set(localTasks + remoteTasks)) // ["task1", "task2", "task3"]
```

**Плюсы:**

- Минимизация потери данных
    
- Более точное разрешение конфликтов
    

**Минусы:**

- Сложнее реализовать
    
- Требует правил для разных типов данных

## 3. Event Sourcing

### Описание

- Все изменения сохраняются как **события (events)**, а состояние формируется путем их применения.
    
- История изменений всегда доступна, что позволяет **воспроизводить состояние в любой момент**.
    

### Принцип работы

1. Пользователь создаёт событие (например, `OrderCreated`, `OrderUpdated`).
    
2. Событие сохраняется локально и отправляется на сервер.
    
3. Сервер применяет событие к основной модели данных.
    
4. В случае конфликта можно повторно применить события и разрешить несоответствия.
    

### Пример ([[Swift]])

```swift
struct Event {
    let type: String
    let timestamp: Date
    let payload: [String: Any]
}

var eventStore: [Event] = []

func applyEvent(event: Event) {
    eventStore.append(event)
    // обновляем локальную модель на основе события
}
```

**Плюсы:**

- Полная история изменений
    
- Позволяет откатывать и восстанавливать состояние
    
- Подходит для сложных бизнес-логик и оффлайн-сценариев
    

**Минусы:**

- Более сложная архитектура
    
- Требуется хранение всех событий
    
- Увеличение объёма данных

## Сравнение стратегий

|Стратегия|Потеря данных|Сложность|Применение|
|---|---|---|---|
|Last Write Wins|Высокая|Низкая|Простые данные, финальный результат важнее истории|
|Merge Conflicts|Низкая|Средняя|Списки, коллекции, структуры данных с приоритетом полей|
|Event Sourcing|Нет|Высокая|Приложения с оффлайн режимом, история изменений, аудит|

---

## Рекомендации для iOS

1. **LWW** → быстро и просто для большинства CRUD-приложений.
    
2. **Merge** → когда данные составные, например, списки задач или избранные элементы.
    
3. **Event Sourcing** → для сложных оффлайн-приложений с необходимостью аудита и отката.
    
4. Всегда хранить **локальную очередь изменений** для повторной синхронизации.
    
5. Использовать **timestamps / versioning** для разрешения конфликтов.
    

---

## Итог

- Синхронизация данных — ключ к оффлайн-приложениям.
    
- Выбор стратегии зависит от: **сложности данных, требований к истории изменений и уровня допустимой потери данных**.
    
- LWW → просто, Merge → умеренно, Event Sourcing → максимально гибко.
