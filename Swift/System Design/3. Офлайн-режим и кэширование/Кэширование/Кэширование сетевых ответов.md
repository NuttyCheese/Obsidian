#system_design
## Определение

**Кэширование сетевых ответов** — это сохранение ответов API-запросов локально на устройстве, чтобы при повторных обращениях **не выполнять лишние сетевые запросы**, ускорять загрузку данных и снижать расход трафика.

> Главная цель — повысить **производительность приложения**, уменьшить задержки и обеспечить **работу в оффлайн-режиме**.

---

## Почему важно

1. **Снижение сетевой нагрузки**
    
    - Повторные запросы к серверу не выполняются, экономится трафик.
        
2. **Быстрая загрузка данных**
    
    - Ответы отображаются мгновенно из кэша, улучшая UX.
        
3. **Поддержка оффлайн-режима**
    
    - Пользователь видит данные, даже если нет соединения.
        
4. **Снижение нагрузки на сервер**
    
    - Сервер обрабатывает меньше запросов → лучше масштабируемость.
        

---

## Основные подходы к кэшированию

|Подход|Описание|Пример|
|---|---|---|
|**Memory Cache**|Ответы хранятся в оперативной памяти|NSCache для JSON-объектов|
|**Disk Cache**|Ответы сохраняются на диске|Файлы в Caches или Documents|
|**Hybrid Cache**|Комбинация памяти и диска|Alamofire, URLCache|
|**HTTP Caching**|Использование заголовков HTTP (`ETag`, `Cache-Control`)|Сервер указывает, можно ли кэшировать|

## Кэширование в iOS

### 1. **URLCache (стандартный подход)**

- Встроенный класс для кэширования HTTP-ответов.
    
- Поддерживает **memory + disk cache**.
    

Пример:

```swift
let cache = URLCache(memoryCapacity: 20 * 1024 * 1024, // 20 MB
                     diskCapacity: 100 * 1024 * 1024, // 100 MB
                     diskPath: "urlCache")

let configuration = URLSessionConfiguration.default
configuration.urlCache = cache
configuration.requestCachePolicy = .returnCacheDataElseLoad

let session = URLSession(configuration: configuration)

if let url = URL(string: "https://api.example.com/users") {
    let task = session.dataTask(with: url) { data, response, error in
        if let data = data {
            print("Данные получены: \(data)")
        }
    }
    task.resume()
}
```

**Ключевые моменты:**

- `returnCacheDataElseLoad` → сначала кэш, если нет — запрос к сети.
    
- `returnCacheDataDontLoad` → только кэш, без сети.
    

---

### 2. **Кэширование через сторонние библиотеки**

- **Alamofire** с `RequestCache` или `ResponseCacher`.
    
- Позволяет кэшировать **только нужные ответы**.
    
- Поддержка expiration time и size limits.
    

Пример:

```swift
let cache = ResponseCacher(behavior: .cache) // кэшируем все ответы
AF.request("https://api.example.com/users")
    .cacheResponse(using: cache)
    .responseJSON { response in
        print(response.result)
    }
```

---

### 3. **Ручное кэширование в [[Core Data]] / [[Swift/Теория/Сторонние библиотеки и расширения/Realm]]**

- Можно сохранять **декодированные модели** в локальную базу.
    
- Позволяет легко фильтровать и сортировать данные без сети.
    
- Полезно, если нужно **кэшировать не весь ответ**, а только часть (например, список заказов).

## Стратегии кэширования

|Стратегия|Описание|
|---|---|
|**Time-based expiration**|Кэш действителен определённое время (например, 5 минут)|
|**ETag / Last-Modified**|Сервер указывает, изменился ли ресурс → обновлять только при изменении|
|**Manual invalidation**|Явное удаление кэша при изменении данных пользователем|
|**Hybrid**|Сначала memory cache → потом disk cache → потом сеть|

---

## Пример использования

Приложение доставки еды:

1. Пользователь открывает список ресторанов.
    
2. Сначала проверяется **memory cache** → если данные есть, отображаем мгновенно.
    
3. Если нет — проверяется **disk cache** (раньше загруженные данные).
    
4. Если данных нет или они устарели → запрос к серверу.
    
5. Ответ сохраняется в **memory + disk cache** для следующего открытия.
    

---

## Best Practices

1. Ограничивать размер кэша (память и диск).
    
2. Сохранять только важные и часто используемые данные.
    
3. Использовать expiration time или ETag для обновления устаревшего кэша.
    
4. Асинхронная загрузка и обновление UI при доступности новых данных.
    
5. Сохранять кэш отдельно для разных версий API, чтобы избежать конфликтов.

## Итог

- Кэширование сетевых ответов → ключ к **ускорению мобильного приложения** и поддержке оффлайн-режима.
    
- Используем **URLCache**, сторонние библиотеки или локальные базы данных.
    
- Стратегии: **time-based, ETag, manual invalidation, hybrid**.
    
- Позволяет **экономить трафик, ускорять UI и снижать нагрузку на сервер**.
