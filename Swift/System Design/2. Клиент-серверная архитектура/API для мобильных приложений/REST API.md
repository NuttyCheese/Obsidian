# REST API

## Определение

**REST (Representational State Transfer)** — это архитектурный стиль для создания [[API]], который использует стандартные HTTP-методы для взаимодействия клиента и сервера.

> В REST клиент и сервер обмениваются данными через [[HTTP]]-запросы к различным ресурсам (endpoints).

---

## Основные компоненты REST

1. **Ресурсы (Resources)**
    
    - Всё в REST представлено как ресурс: пользователи, заказы, товары.
        
    - Каждый ресурс имеет уникальный URL, например:
        
        ```
        GET https://api.example.com/users/123
        ```
        
2. **HTTP-методы**  
    | Метод | Назначение | Пример |  
    |-------|------------|--------|  
    | [[GET-HTTP|GET]] | Получение ресурса | `GET /users/123` |  
    | [[POST-HTTP|POST]] | Создание нового ресурса | `POST /orders` |  
    | [[PUT-HTTP|PUT]] | Полное обновление ресурса | `PUT /users/123` |  
    | [[PATCH-HTTP|PATCH]] | Частичное обновление ресурса | `PATCH /users/123` |  
    | [[DELETE-HTTP|DELETE]] | Удаление ресурса | `DELETE /orders/456` |
    
3. **Статусы HTTP**  
    | Код | Значение | Пример |  
    |-----|---------|--------|  
    | 200 | OK | Запрос успешен |  
    | 201 | Created | Ресурс создан |  
    | 204 | No Content | Ресурс удалён, тело пусто |  
    | 400 | Bad Request | Ошибка запроса |  
    | 401 | Unauthorized | Требуется авторизация |  
    | 404 | Not Found | Ресурс не найден |  
    | 500 | Internal Server Error | Ошибка сервера |
    

---

## Преимущества REST для мобильных приложений

- Простота и распространённость → легко интегрировать.
    
- Стандартизированные HTTP-методы и коды.
    
- Хорошо кэшируется через HTTP-кэширование.
    
- Подходит для CRUD-операций (Create, Read, Update, Delete).
    

---

## Недостатки REST

- **Overfetching / Underfetching** — клиент может получать лишние данные или не все нужные.
    
- Отдельные endpoints → больше сетевых запросов.
    
- Ограниченная поддержка реального времени (нужны WebSockets или SSE).
    

---

## REST vs GraphQL

| Критерий       | REST                        | GraphQL                          |
| -------------- | --------------------------- | -------------------------------- |
| [[URL]]        | Много endpoints             | Один endpoint `/graphql`         |
| Данные         | Сервер определяет структуру | Клиент выбирает поля             |
| Сложность      | Простая                     | Требует генерации схем и клиента |
| Реальное время | Нет встроено                | Подписки через [[WebSocket]]     |

## Использование REST в [[iOS]]

### Пример через [[URLSession]]

```swift
import Foundation

struct User: Decodable {
    let id: String
    let name: String
    let email: String
}

class UserService {
    func fetchUser(id: String) {
        guard let url = URL(string: "https://api.example.com/users/\(id)") else { return }
        
        let task = URLSession.shared.dataTask(with: url) { data, response, error in
            if let data = data {
                do {
                    let user = try JSONDecoder().decode(User.self, from: data)
                    print("User: \(user.name)")
                } catch {
                    print("Ошибка декодирования: \(error)")
                }
            }
        }
        task.resume()
    }
}
```

## Кэширование и оптимизация

- Использовать **[[URLCache]]** для GET-запросов.
    
- Применять **ETag** и заголовки `If-None-Match`.
    
- Пагинация → для списков с большим количеством данных.
    
- Сжатие данных (gzip).

## Пример применения

Приложение доставки еды:

- REST endpoints:
    
    - `GET /restaurants` → список ресторанов
        
    - `POST /orders` → создание заказа
        
    - `GET /orders/123` → получение статуса заказа
        
- Простая интеграция и быстрое прототипирование.
    
- Недостаток: для получения полной информации о заказе с деталями пользователя и рестораном нужно несколько запросов.
    

---

## Итог

- **REST API** — стандартный способ взаимодействия мобильного приложения с сервером через HTTP.
    
- Преимущества: простота, стандартизированность, кэширование, поддержка CRUD.
    
- Недостатки: возможен overfetching, несколько запросов для сложных данных, ограниченная поддержка real-time.
    
- В iOS реализуется через `URLSession`, `Alamofire` или другие сетевые библиотеки.
