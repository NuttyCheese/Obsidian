#system_design
## Определение

**Логирование и алертинг** — это процессы **фиксации ошибок и оповещения команды разработки** при критических сбоях в приложении.

Цель: **быстро обнаруживать и устранять проблемы**, минимизировать влияние на пользователя и улучшать стабильность приложения.

---

## 1. Логирование ошибок

### Определение

**Логирование** — запись информации о событиях и ошибках приложения в **структурированном виде**, чтобы их можно было проанализировать.

**Виды логов:**

- **Debug logs** → для разработки, подробная информация, трассировка.
    
- **Error logs** → ошибки, исключения, критические сбои.
    
- **Crash logs** → полные сведения о падениях приложения.
    

### Инструменты для iOS

1. **Crashlytics ([[Firebase]])** → автоматическое логирование сбоев и крашей.
    
2. **Sentry** → мониторинг ошибок, трассировка стека, интеграция с [[GitHub]]/Slack.
    
3. **Custom logging** → собственная реализация логов через файловую систему или сервер.
    

### Пример логирования ошибки (Swift)

```swift
do {
    try riskyOperation()
} catch {
    // Логируем локально
    print("Ошибка: \(error.localizedDescription)")
    
    // Отправляем на сервер аналитики
    Crashlytics.crashlytics().record(error: error)
    
    // Дополнительно: отправка события в Sentry
    SentrySDK.capture(error: error)
}
```

## 2. Алертинг (уведомление команды)

### Определение

**Алертинг** — автоматическое **уведомление разработчиков или DevOps команды** о критических сбоях приложения в реальном времени.

**Цель:** быстро реагировать на падения и ошибки, минимизировать влияние на пользователей.

### Примеры инструментов

|Инструмент|Возможности|
|---|---|
|**Crashlytics**|Push-уведомления, email, Slack при критических крашах|
|**Sentry**|Настройка правил алертов по типу ошибки, количеству событий, пользователям|
|**PagerDuty / Opsgenie**|Интеграция для автоматического алерта и on-call поддержки|

## 3. Критические ошибки (Critical Errors)

**Критические ошибки** — это ситуации, при которых приложение **не может продолжать работу корректно**:

- Падение приложения (`crash`)
    
- Ошибка загрузки важного ресурса (например, конфигурация или база данных)
    
- Нарушение целостности данных
    

**При критических ошибках важно:**

1. Логировать **все детали ошибки** (stack trace, состояние приложения).
    
2. Отправлять **уведомление команде** для оперативного реагирования.
    
3. Сохранять информацию для последующего анализа и исправления.

## 4. Best Practices по логированию и алертингу

1. **Использовать структурированные логи** → [[JSON]], ключ-значение для удобного анализа.
    
2. **Логировать контекст** → экран, пользовательский ID, версия приложения, устройство.
    
3. **Разделять уровни логов** → debug, info, warning, error, critical.
    
4. **Отправлять логи на сервер или в облако** → чтобы были доступны после краша устройства.
    
5. **Настраивать алерты по критичности** → критические ошибки — немедленное уведомление, менее важные — агрегированное уведомление.
    
6. **Соблюдать конфиденциальность** → не логировать персональные данные пользователей.

## 5. Применение в мобильных приложениях

- **Crashlytics** → автоматическое логирование и уведомления о крашах, интеграция с Firebase.
    
- **Sentry** → отслеживание ошибок и исключений в продакшене, подробная трассировка, алерты.
    
- **Custom logging** → при специфических требованиях проекта, можно логировать в локальные файлы или отправлять на свой сервер.
    

### Пример workflow

1. Пользователь вызывает функцию, происходит исключение.
    
2. Ошибка логируется локально и отправляется в Crashlytics/Sentry.
    
3. Если ошибка критическая → срабатывает алерт (Slack, email, push).
    
4. Разработчики анализируют стек вызовов и исправляют проблему.

## Итог

- **Логирование** → фиксация ошибок и состояния приложения.
    
- **Алертинг** → мгновенное уведомление команды о критических сбоях.
    
- Ключевые инструменты: **Crashlytics, Sentry, Custom logging**.
    
- Best Practices: структурированные логи, контекст ошибок, уровни критичности, защита данных.
    
- Цель: **быстрое выявление и исправление ошибок для стабильности приложения и повышения качества UX**.
