## Название

**Push: тап по уведомлению с полем `url` не открывает чат, если строка URL не парсится**

---

## Описание

В `AppDelegate.userNotificationCenter(_:didReceive:withCompletionHandler:)` при `UNNotificationDefaultActionIdentifier` если в `userInfo` есть ключ `"url"`, приложение пытается вызвать `openUrlWhenReady` только при успешном `URL(string: dataUrl)`. Если парсинг не удался, **не выполняется ни открытие URL, ни запасной сценарий** с `peerIdFromNotification` / `openChatWhenReady` — он целиком в ветке `else` и срабатывает только когда ключа `"url"` нет.

В результате при некорректной или неожиданной строке в `"url"` пользователь тапает по уведомлению, приложение может активироваться, но **целевой экран (чат/сообщение) не открывается**.

Фрагмент текущей логики:

```swift
if let dataUrl = response.notification.request.content.userInfo["url"] as? String {
    if let url = URL(string: dataUrl) {
        self.openUrlWhenReady(accountId: accountId, url: url, external: true)
    }
    // при неудачном парсе — ничего
} else {
    // openChatWhenReady(...)
}
```

## Шаги для воспроизведения / сценарий

1. Отправить push (или симулировать payload), в котором одновременно есть:
   - `userInfo["url"]` — строка, для которой `URL(string:)` возвращает `nil` (например, невалидное экранирование, пустая строка после ошибки на сервере, обрезанный URL).
   - стандартные поля для `peerIdFromNotification` (как в обычных чат-уведомлениях).
2. Тапнуть по уведомлению (default action).
3. **Ожидаемое поведение:** открывается чат (или обрабатывается ссылка, если URL валиден).
4. **Фактическое:** при невалидном `url` ничего из этого не происходит.

Альтернатива без реального сервера: временно подставить в отладочной сборке в обработчике `userInfo` с фиктивным `"url"` и проверить отсутствие fallback.

---

## Влияние

- **UX:** высокое для затронутых пользователей — «тапнул по уведомлению, а чат не открылся».
- **Стабильность:** крашей нет, но нарушается ожидаемая навигация.
- **Производительность:** не затрагивается.

---

## Предложение решения / улучшения

**Идея:** если `dataUrl` присутствует, но `URL(string: dataUrl) == nil`, выполнить ту же цепочку, что сейчас в `else`: извлечь peer/thread/message/story из уведомления и вызвать `openChatWhenReady` (и при необходимости сохранить приоритет: сначала валидный URL, иначе чат).

Псевдокод:

```swift
if response.actionIdentifier == UNNotificationDefaultActionIdentifier {
    var openedUrl = false
    if let dataUrl = userInfo["url"] as? String,
       let url = URL(string: dataUrl) {
        self.openUrlWhenReady(accountId: accountId, url: url, external: true)
        openedUrl = true
    }
    if !openedUrl, let (peerId, threadId) = peerIdFromNotification(response.notification) {
        // messageId, storyId — как в текущем else
        self.openChatWhenReady(...)
    }
    completionHandler()
}
```

Уточнение продукта: если при **валидном** `url` всё же нужно всегда открывать и чат — это отдельное требование; предложенный вариант сохраняет текущее «URL или только чат из else», устраняя только провал при битом `url`.

## Тесты / проверки

- Уведомление **без** `"url"` — по-прежнему открывается чат (регрессия).
- Уведомление с **валидным** `"url"` — открывается обработка URL (как сейчас).
- Уведомление с **невалидным** `"url"` и валидным peer в payload — открывается чат (новое ожидаемое поведение).
- Категории `"c"` / `"t"` — корректный `messageId` при fallback на чат.
- Reply-action и другие `actionIdentifier` — без изменений поведения.
